# Clarity and Feedback（清晰度与反馈）

> 状态：V1 草案  
> 适用范围：通用的软件、游戏与交互产品设计  
> 核心问题：玩家如何知道当前发生了什么、为什么发生、接下来可能发生什么，以及自己的操作是否有效  
> 目的：建立一套覆盖状态可见性、因果关系、输入反馈、结果反馈、预测性和多通道表达的设计框架，帮助玩家形成稳定认知并减少无意义困惑

---

## 1. 定义

清晰度，是玩家能够准确理解当前状态、可用行动、规则关系、风险和结果的程度。

反馈，是系统针对玩家输入、状态变化、成功、失败和环境变化所提供的信息。

两者之间的关系是：

```text
反馈提供信息
→ 信息支持理解
→ 理解形成预测
→ 预测支持决策
→ 决策产生新的反馈
```

清晰度不是信息越多越好。

真正有效的清晰度要求：

- 重要信息被看见；
- 信息在正确时机出现；
- 信息表达方式与重要程度匹配；
- 玩家能够理解因果关系；
- 玩家能够根据反馈调整行为；
- 非关键信息不会淹没核心信息。

反馈也不只是动画、音效或数字。

完整反馈通常包括：

```text
输入是否被接收
状态是否发生变化
结果是什么
结果为什么发生
结果是否重要
玩家接下来可以做什么
```

---

## 2. 为什么清晰度与反馈重要

玩家不能直接看到系统内部逻辑。

他们只能通过界面、动画、声音、文本、数值和结果推测系统如何运行。

如果反馈不足或表达混乱，会导致：

- 玩家不知道输入是否成功；
- 玩家无法判断当前状态；
- 玩家无法预测风险；
- 玩家无法理解失败原因；
- 玩家将随机结果误解为规则；
- 玩家不知道应该调整什么；
- 玩家对系统产生不信任；
- 玩家依赖试错而不是理解；
- 玩家将界面困难误认为玩法困难。

一个系统即使规则合理，如果玩家无法看见和理解，也会在体验上显得任意。

---

## 3. 清晰度的对象

清晰度并不是单一维度。

设计需要明确究竟希望玩家看清什么。

### 3.1 状态清晰度

玩家能否知道：

- 自己当前处于什么状态；
- 角色、资源、任务和系统处于什么状态；
- 哪些状态正在生效；
- 哪些状态即将结束；
- 哪些状态发生了变化。

### 3.2 行动清晰度

玩家能否知道：

- 当前可以做什么；
- 哪些操作不可用；
- 为什么不可用；
- 操作需要什么条件；
- 操作将产生什么结果。

### 3.3 规则清晰度

玩家能否理解：

- 系统如何运作；
- 哪些规则稳定存在；
- 哪些情况属于例外；
- 不同系统之间如何关联。

### 3.4 因果清晰度

玩家能否理解：

- 什么导致了当前结果；
- 哪个行动产生了主要影响；
- 哪些因素只是同时发生；
- 哪些变化来自外部系统。

### 3.5 风险清晰度

玩家能否知道：

- 哪些行为具有风险；
- 风险可能带来什么损失；
- 风险大致有多高；
- 是否存在规避和恢复方式。

### 3.6 目标清晰度

玩家能否理解：

- 当前主要目标；
- 次要目标；
- 完成条件；
- 失败条件；
- 进度；
- 下一步。

### 3.7 结果清晰度

玩家能否知道：

- 操作是否成功；
- 获得了什么；
- 失去了什么；
- 哪些状态发生改变；
- 是否需要继续处理。

### 3.8 优先级清晰度

玩家能否判断：

- 哪些信息最重要；
- 哪个目标更紧急；
- 哪个风险需要立即处理；
- 哪些内容可以稍后查看。

---

## 4. 清晰度不等于完全公开

设计可以保留未知、隐藏信息、随机性和探索。

清晰度要求的不是让玩家知道全部答案，而是让玩家拥有足够信息进行合理判断。

例如：

```text
可以隐藏敌人的完整数值，
但应让玩家知道敌人当前是否危险。

可以隐藏事件的具体结果，
但应让玩家理解大致风险方向。

可以保留剧情谜团，
但不应让玩家无法判断当前任务目标。

可以使用随机掉落，
但应表达可能范围、稀有度或风险。
```

### 4.1 隐藏答案与隐藏线索

隐藏答案可以产生探索感。

隐藏必要线索则容易产生任意感。

### 4.2 不确定性与不可理解性

不确定性表示玩家知道存在多个可能结果。

不可理解性表示玩家连结果由什么决定都不知道。

前者可以产生策略，后者通常只产生困惑。

### 4.3 信息精度分层

信息可以按精度逐层开放：

```text
方向性信息
→ 范围信息
→ 近似信息
→ 精确数值
→ 完整公式
```

不同游戏和不同玩家阶段可以选择不同层级。

## 5. 状态可见性

系统状态不应只存在于内部数据中。

玩家需要能够看到与当前决策有关的状态。

### 5.1 状态表达内容

完整状态表达通常包括：

```text
状态是什么
状态作用于谁
状态何时开始
状态持续多久
状态强度如何
状态来自哪里
状态如何结束
```

并非所有状态都需要展示全部信息，但高影响状态需要足够完整。

### 5.2 持续状态

例如：

- 中毒；
- 护盾；
- 加速；
- 冷却；
- 增益；
- 减益；
- 锁定；
- 工作进行中。

持续状态应避免只在触发瞬间反馈一次。

如果状态仍影响玩家决策，就应保持可见。

### 5.3 状态变化

状态变化需要让玩家知道：

- 发生了变化；
- 变化方向；
- 变化幅度；
- 变化原因；
- 变化是否永久。

例如，单纯显示数字变化，可能不足以说明：

```text
这是临时加成还是永久成长？
这是资源消耗还是数值重置？
这是当前装备变化还是角色基础属性变化？
```

### 5.4 状态层级

可以将状态分为：

```text
关键状态
当前决策状态
短期相关状态
长期管理状态
背景状态
```

关键状态应持续可见。

背景状态可以按需查看。

### 5.5 状态可见性评审问题

```text
玩家此刻必须知道哪些状态？
哪些状态正在影响当前决策？
哪些状态只在内部存在但没有表达？
状态是否能被快速识别？
状态结束时是否有明确反馈？
```

## 6. 输入反馈

玩家输入后，系统应及时表达输入是否被接收。

### 6.1 输入反馈的基本问题

```text
我刚才的输入被识别了吗？
系统正在处理吗？
操作成功了吗？
如果没有成功，为什么？
```

### 6.2 即时反馈

常见即时反馈包括：

- 按钮按下状态；
- 角色动作启动；
- 声音；
- 振动；
- 高亮；
- 光标变化；
- 加载状态；
- 选中状态；
- 数值变化。

### 6.3 延迟操作

当结果无法立即完成时，应提供：

```text
输入已接收
→ 正在处理
→ 当前进度或等待状态
→ 最终结果
```

否则玩家可能重复输入或误以为系统失效。

### 6.4 输入被拒绝

系统拒绝操作时，不应只表现为“没有反应”。

应根据重要程度说明：

- 当前不可用；
- 不可用原因；
- 所需条件；
- 可替代操作。

### 6.5 反馈延迟

反馈延迟会削弱输入与结果之间的因果联系。

对于直接操作，反馈应尽可能接近输入发生时刻。

对于刻意延迟的动作，应通过准备动画、进度或状态提示保持联系。

### 6.6 输入反馈评审问题

```text
玩家操作后多久得到反馈？
反馈是否明确对应这次输入？
连续输入时能否区分每次结果？
输入失败时是否说明原因？
系统处理期间玩家是否知道当前状态？
```

## 7. 成功反馈

成功反馈需要表达：

- 成功发生；
- 成功内容；
- 成功价值；
- 对后续的影响。

### 7.1 成功强度匹配

反馈强度应与结果重要程度匹配。

```text
普通操作
→ 轻量反馈

阶段完成
→ 中等反馈

重大突破
→ 高强度反馈
```

如果每个小操作都使用高强度表现，会造成：

- 注意力疲劳；
- 重要事件失去区分度；
- 节奏被打断；
- 反馈显得廉价。

### 7.2 成功的可归因性

玩家需要知道成功来自：

- 技能；
- 决策；
- 准备；
- 成长；
- 随机；
- 队友；
- 系统帮助。

如果成功来源不清晰，玩家难以建立能力感和学习。

### 7.3 奖励反馈

奖励应表达：

- 获得什么；
- 为什么获得；
- 数量和品质；
- 放到哪里；
- 可以用来做什么；
- 是否解锁新内容。

### 7.4 成功反馈评审问题

```text
玩家是否知道自己成功了？
玩家是否知道成功来自什么？
结果的重要程度是否与反馈强度匹配？
奖励是否能被理解和追踪？
成功反馈是否打断了核心节奏？
```

---

## 8. 失败反馈

失败反馈的目的不只是宣布失败，而是帮助玩家理解并决定下一步。

### 8.1 失败反馈的层级

```text
发生了什么
为什么发生
主要影响因素
可以如何调整
如何重新尝试
```

### 8.2 不应过度解释

失败反馈也不应直接替玩家完成全部分析。

可以根据体验目标选择：

- 只提示方向；
- 展示关键因素；
- 提供完整记录；
- 提供可选详细分析。

### 8.3 失败与情绪

失败反馈需要考虑：

- 挫败强度；
- 玩家投入；
- 损失程度；
- 是否可恢复；
- 是否立即重试；
- 是否需要情绪缓冲。

高投入失败如果只显示简单“失败”，容易显得冷漠和任意。

### 8.4 失败归因

失败归因应避免误导。

例如：

```text
玩家失败主要由于等级不足，
系统却只提示“技巧不够”。

玩家失败来自隐藏规则，
系统却暗示玩家操作失误。
```

错误归因会破坏信任。

### 8.5 失败反馈评审问题

```text
玩家是否知道主要失败原因？
失败原因是否真实而非推测？
反馈是否提供可行动信息？
玩家是否知道如何重新尝试？
失败表现是否与损失程度匹配？
```

## 9. 因果关系

玩家需要建立“行动—变化—结果”的关系。

### 9.1 因果链

```text
玩家行动
→ 系统响应
→ 状态变化
→ 结果
→ 后续影响
```

每个关键环节都应有足够反馈。

### 9.2 因果中断

因果关系容易被以下情况破坏：

- 反馈延迟过长；
- 多个变化同时发生；
- 动画遮挡结果；
- 数值变化没有来源；
- 隐藏状态参与计算；
- 相同表现对应不同规则；
- 系统自动执行未说明行为。

### 9.3 多来源结果

当结果来自多个因素时，应突出主要因素。

例如：

```text
伤害 = 基础攻击 + 技能倍率 + 增益 + 暴击 + 敌方减伤。
```

玩家未必需要看到完整公式，但可以看到：

- 暴击；
- 被减伤；
- 属性克制；
- 关键增益生效。

### 9.4 时间上的关联

反馈越接近原因，越容易被理解。

如果因果必须跨较长时间发生，可以使用：

- 来源标记；
- 事件日志；
- 关联线；
- 回顾；
- 变化历史；
- 状态来源说明。

### 9.5 因果评审问题

```text
玩家能否将结果与具体行为联系起来？
同时发生的变化是否容易混淆？
关键结果是否有来源说明？
自动行为是否会被误解为玩家输入？
```

## 10. 预测性

良好反馈不仅解释过去，也帮助预测未来。

### 10.1 可预测内容

根据项目需要，玩家可能需要预测：

- 操作结果；
- 资源消耗；
- 伤害范围；
- 风险等级；
- 状态持续时间；
- 路径；
- 奖励；
- 冷却；
- 敌人行为；
- 长期成长影响。

### 10.2 结果预览

高成本操作应优先提供预览。

例如：

- 强化后属性；
- 装备替换差异；
- 技能范围；
- 建造占用；
- 删除影响；
- 资源结算；
- 任务风险。

### 10.3 预警

危险事件应根据反应需求提前表达。

预警可以包含：

- 即将发生什么；
- 发生位置；
- 发生时间；
- 危险程度；
- 可用应对方式。

### 10.4 预测不应替代决策

过度精确的预测可能消除风险和探索。

可以根据体验目标保留：

- 结果范围；
- 方向性提示；
- 风险等级；
- 部分信息；
- 延迟揭示。

### 10.5 预测性评审问题

```text
玩家在行动前需要知道什么？
高成本操作是否提供足够预览？
危险是否有合理预警？
未知是否仍然允许合理判断？
```

## 11. 信息层级

所有信息同时出现会降低整体清晰度。

### 11.1 信息优先级

可以按以下层级组织：

```text
P0：立即影响生存、安全或不可逆结果
P1：当前决策必须信息
P2：短期规划信息
P3：长期管理信息
P4：补充解释
P5：装饰和氛围
```

### 11.2 渐进披露

信息应在需要时出现。

常见方式：

- 默认显示概要；
- 悬停或选中显示详情；
- 高级面板显示完整数据；
- 首次出现提供解释；
- 重复使用减少提示；
- 错误发生时补充相关信息。

### 11.3 视觉层级

视觉层级可以通过以下方式建立：

- 尺寸；
- 位置；
- 对比；
- 动画；
- 留白；
- 图标；
- 字重；
- 持续时间；
- 深度；
- 声音。

高强度表达应保留给真正重要的信息。

### 11.4 信息密度

信息密度应匹配：

- 玩家经验；
- 当前任务；
- 会话长度；
- 设备尺寸；
- 输入方式；
- 场景强度。

### 11.5 信息层级评审问题

```text
当前最重要信息是什么？
界面是否清楚表达了优先级？
哪些信息可以延后或隐藏？
是否存在多个同强度焦点？
高阶玩家是否能访问更详细信息？
```

## 12. 多通道反馈

反馈可以通过视觉、声音、动画、触觉和文本共同表达。

### 12.1 视觉反馈

适合表达：

- 位置；
- 状态；
- 数量；
- 范围；
- 方向；
- 优先级；
- 持续变化。

风险：

- 视觉拥挤；
- 色彩依赖；
- 动画遮挡；
- 小尺寸不可读。

### 12.2 声音反馈

适合表达：

- 输入确认；
- 危险；
- 成功；
- 稀有事件；
- 方向；
- 节奏。

风险：

- 玩家可能关闭声音；
- 环境不允许播放；
- 多声音重叠；
- 单靠声音无法支持听觉受限玩家。

### 12.3 动画反馈

适合表达：

- 状态变化；
- 因果；
- 运动方向；
- 结果强度；
- 情绪。

风险：

- 动画过长；
- 延迟操作；
- 过度晃动；
- 影响舒适性；
- 重复后造成疲劳。

### 12.4 触觉反馈

适合表达：

- 输入确认；
- 碰撞；
- 危险；
- 节奏；
- 强度差异。

风险：

- 设备不支持；
- 过度使用；
- 触觉含义不一致。

### 12.5 文本反馈

适合表达：

- 复杂原因；
- 条件；
- 规则；
- 风险；
- 详细结果。

风险：

- 阅读负担；
- 时机不合适；
- 文本过长；
- 本地化后布局变化。

### 12.6 冗余编码

关键反馈不应只依赖单一通道。

例如危险状态可以同时使用：

- 颜色；
- 图标；
- 动画；
- 声音；
- 文本。

冗余不是重复浪费，而是提高可识别性和可访问性。

### 12.7 通道一致性

不同通道不应表达冲突信息。

例如：

```text
视觉表现为成功，
声音却像失败。

按钮显示不可用，
但仍可触发。

红色有时代表危险，
有时代表稀有奖励。
```

---

## 13. 反馈强度与节奏

反馈需要形成层级，而不是持续最大化。

### 13.1 强度维度

反馈强度可以由以下因素构成：

- 视觉面积；
- 对比；
- 动画幅度；
- 声音大小；
- 持续时间；
- 屏幕震动；
- 停顿；
- 镜头变化；
- 粒子数量。

### 13.2 强度匹配

```text
影响越大
→ 反馈通常越强

频率越高
→ 单次反馈通常越轻

风险越高
→ 预警通常越明确

结果越稀有
→ 反馈可以越独特
```

### 13.3 反馈疲劳

频繁高强度反馈会导致：

- 玩家习惯化；
- 重要事件失去突出；
- 视觉和听觉疲劳；
- 节奏被反复打断；
- 操作效率下降。

### 13.4 可跳过与加速

重复演出应根据价值考虑：

- 跳过；
- 加速；
- 减弱；
- 关闭；
- 仅首次完整播放。

---

## 14. 反馈的一致性

反馈一致性帮助玩家形成稳定心智模型。

### 14.1 语义一致性

相同颜色、图标、声音和动画应尽量表达相同含义。

### 14.2 强度一致性

同等重要事件应使用近似强度。

### 14.3 时序一致性

相似操作的反馈顺序应保持稳定。

### 14.4 平台一致性

跨平台可以调整表现，但核心语义应一致。

例如：

- 触屏使用触觉；
- 手柄使用震动；
- 键鼠使用视觉与声音。

### 14.5 例外管理

如果必须违反已有反馈语言，应：

- 明确原因；
- 限制范围；
- 提供教学；
- 避免与关键状态混淆。

---

## 15. 清晰度与学习

反馈是玩家学习系统的主要方式。

### 15.1 学习循环

```text
观察
→ 假设
→ 行动
→ 反馈
→ 修正理解
→ 再次行动
```

如果反馈无法支持修正，玩家只能机械记忆答案。

### 15.2 教学与实际反馈一致

教学中表达的规则必须与实际系统一致。

避免：

- 教程使用特殊规则；
- 教程隐藏实际风险；
- 教程按钮行为与正式界面不同；
- 文字说明与实际反馈冲突。

### 15.3 逐步提高信息精度

新手阶段可以提供：

- 强方向提示；
- 明确目标；
- 简化状态；
- 重点反馈。

熟练阶段可以开放：

- 详细数据；
- 事件记录；
- 高级预览；
- 自定义界面；
- 减少辅助提示。

---

## 16. 清晰度与复杂度

复杂系统不一定不清晰。

简单系统也可能表达混乱。

### 16.1 规则复杂与表达复杂

需要区分：

```text
系统本身包含多少规则
和
玩家需要同时处理多少信息
```

复杂系统可以通过：

- 分层；
- 逐步引入；
- 规则复用；
- 状态可视化；
- 结果预览；
- 按需详情；

保持可理解性。

### 16.2 清晰度债务

当系统不断增加但反馈结构没有同步扩展，会形成清晰度债务。

表现包括：

- 图标数量过多；
- 状态无法区分；
- 日志缺失；
- 教学持续补丁；
- 文本越来越长；
- 玩家依赖外部攻略。

清晰度债务需要像技术债务一样被记录和处理。

## 18. 设计评审问题

### 18.1 状态

```text
玩家当前最需要知道哪些状态？
这些状态是否持续可见？
状态变化是否有来源和结束反馈？
```

### 18.2 输入

```text
输入是否立即得到响应？
处理中的状态是否明确？
输入失败时是否解释原因？
```

### 18.3 结果

```text
玩家是否知道成功或失败？
是否知道具体获得和失去什么？
结果是否与反馈强度匹配？
```

### 18.4 因果

```text
玩家能否理解结果由什么造成？
多个同时变化是否会混淆？
关键来源是否可以追踪？
```

### 18.5 预测

```text
行动前是否有足够信息判断？
高风险行为是否提供预览？
危险是否提供合理预警？
```

### 18.6 信息层级

```text
当前最重要信息是否最突出？
哪些信息可以延后？
是否存在视觉、声音或文本过载？
```

### 18.7 多通道

```text
关键反馈是否只依赖颜色或声音？
不同通道是否表达一致？
玩家关闭某种通道后是否仍能理解？
```

### 18.8 学习

```text
反馈是否帮助玩家修正心智模型？
玩家能否根据结果调整下一步？
是否必须依赖外部攻略理解核心规则？
```

## 19. 检查清单

### 19.1 状态可见性

- [ ] 当前关键状态可见；
- [ ] 持续状态具有持续表达；
- [ ] 状态来源在必要时可追踪；
- [ ] 状态持续时间或结束条件明确；
- [ ] 高影响状态与普通状态容易区分；
- [ ] 状态结束具有反馈。

### 19.2 输入反馈

- [ ] 输入被接收时有即时反馈；
- [ ] 延迟操作具有处理中状态；
- [ ] 不可用操作说明原因；
- [ ] 连续操作反馈不会混淆；
- [ ] 反馈延迟不会破坏因果关系。

### 19.3 成功与失败

- [ ] 成功内容和价值明确；
- [ ] 反馈强度与结果重要程度匹配；
- [ ] 失败原因可理解；
- [ ] 失败提供可行动信息；
- [ ] 玩家知道如何恢复或重试；
- [ ] 反馈没有错误归因。

### 19.4 因果与预测

- [ ] 行动、变化和结果之间关系清楚；
- [ ] 多来源结果突出主要原因；
- [ ] 高成本操作提供结果预览；
- [ ] 危险事件提供合理预警；
- [ ] 不确定性不会变成不可理解性。

### 19.5 信息层级

- [ ] 关键信息具有最高优先级；
- [ ] 非关键信息不会持续竞争注意力；
- [ ] 信息按需和渐进披露；
- [ ] 高阶信息可以进一步访问；
- [ ] 信息密度匹配设备与情境。

### 19.6 多通道与可访问性

- [ ] 关键反馈不只依赖单一颜色；
- [ ] 关键反馈不只依赖声音；
- [ ] 视觉、声音、动画和文本语义一致；
- [ ] 重复高强度反馈可以减弱或跳过；
- [ ] 动画和震动可根据需要调整；
- [ ] 本地化后仍保持可读性。

### 19.7 验证

- [ ] 玩家能识别关键状态；
- [ ] 玩家能复述主要因果关系；
- [ ] 玩家能预测高成本操作结果；
- [ ] 玩家失败后能指出调整方向；
- [ ] 关键反馈在真实游玩强度下仍可被注意；
- [ ] 已测试关闭声音、色觉差异或小屏幕情境。

## 20. 常见失败模式

### 20.1 信息越多越清晰

表现：

将所有数值、状态和说明同时显示。

问题：

关键信息失去优先级。

修正：

使用信息层级和渐进披露。

### 20.2 只在触发瞬间显示状态

表现：

状态开始时出现提示，之后完全不可见。

问题：

玩家无法在持续期间做判断。

### 20.3 没有反应就是不可用

表现：

按钮不可用时点击无反馈。

问题：

玩家无法判断是条件不足、系统故障还是输入失败。

### 20.4 只展示结果，不展示原因

表现：

生命值减少、任务失败、资源消失，但没有来源。

问题：

玩家无法学习和调整。

### 20.5 使用更多特效掩盖表达问题

表现：

反馈不清晰时持续增加粒子、震动和音效。

问题：

强度增加，但语义仍然不明确。

### 20.6 所有事件都高强度反馈

表现：

普通掉落、升级、任务完成都使用大演出。

问题：

反馈疲劳，重要事件失去层级。

### 20.7 只依赖颜色

表现：

红绿是唯一状态区别。

问题：

色觉差异、屏幕质量和环境会降低可识别性。

修正：

加入形状、图标、文本或动画差异。

### 20.8 反馈与结果不一致

表现：

角色播放命中动画，但实际没有造成效果。

问题：

玩家建立错误心智模型。

### 20.9 预警时间不匹配反应需求

表现：

危险提示出现时已经无法应对。

问题：

预警只是通知，不是决策支持。

### 20.10 失败建议过于笼统

表现：

```text
提升实力后再试。
```

问题：

玩家不知道具体问题。

修正：

提供方向性信息，例如：

- 防御不足；
- 缺少范围应对；
- 未处理某状态；
- 资源配置不匹配。

### 20.11 反馈延迟导致重复操作

表现：

点击后没有即时响应，玩家反复点击。

问题：

可能产生重复提交或误操作。

### 20.12 只在教程中清晰

表现：

正式流程中的反馈比教程弱很多。

问题：

玩家无法将学习迁移到真实系统。

### 20.13 动画优先于操作

表现：

频繁演出阻止玩家继续操作。

问题：

反馈开始妨碍流程本身。

### 20.14 用外部攻略承担核心解释

表现：

玩家必须查询社区才能理解基础规则。

问题：

核心系统内部反馈不足。

外部攻略可以支持深度研究，但不应替代基础可理解性。

## 21. 与其他原则的冲突

### 21.1 清晰度 vs 探索感

完全公开会削弱发现。

处理方式：

- 隐藏答案，不隐藏必要线索；
- 提供方向，不一定提供精确数值；
- 允许玩家通过观察逐步提高信息精度。

### 21.2 清晰度 vs 视觉简洁

更多状态可能导致界面拥挤。

处理方式：

- 建立信息优先级；
- 使用按需详情；
- 只常驻当前决策相关内容；
- 利用空间、时间和通道分层。

### 21.3 清晰度 vs 节奏

详细说明可能打断操作。

处理方式：

- 即时反馈保持轻量；
- 复杂解释延后到结算或详情；
- 允许玩家主动展开；
- 首次完整、重复简化。

### 21.4 反馈强度 vs 舒适性

强烈表现可能提高冲击力，但造成疲劳或不适。

处理方式：

- 建立强度层级；
- 限制闪烁和震动；
- 提供设置；
- 使用语义而非纯强度表达重要性。

### 21.5 结果可解释 vs 系统神秘感

完整公式可能破坏氛围。

处理方式：

- 提供主要原因和方向；
- 保留精确公式作为高级信息；
- 用世界观一致的方式表达机制。

### 21.6 预测性 vs 风险感

精确预览可能消除紧张。

处理方式：

- 展示范围、趋势或风险等级；
- 保留部分未知；
- 确保玩家仍有管理风险的方法。

### 21.7 一致性 vs 特殊事件

特殊内容可能需要独特反馈。

处理方式：

- 保留基础语义；
- 在表现上创新；
- 不让特殊表现与已有关键状态混淆。

## 22. 示例

### 22.1 示例一：技能不可用

#### 问题

玩家点击技能后没有任何反应。

#### 可能原因

- 冷却中；
- 资源不足；
- 目标无效；
- 被沉默；
- 动画锁定；
- 系统延迟。

#### 改进

- 按钮显示不可用状态；
- 冷却显示剩余时间；
- 资源不足高亮相关资源；
- 点击时给予轻量拒绝反馈；
- 必要时显示简短原因。

#### 保护价值

- 输入确认；
- 状态可见；
- 因果理解；
- 减少重复输入。

---

### 22.2 示例二：战斗失败回顾

#### 问题

玩家知道队伍失败，但不知道为什么。

#### 方案

结算显示：

- 主要伤害来源；
- 最危险敌人；
- 关键未处理状态；
- 队伍倒下顺序；
- 可选详细日志；
- 重新配置入口。

#### 不直接提供

- 唯一正确构筑；
- 完整自动优化；
- 强制建议。

#### 目标

提供学习方向，同时保留玩家分析空间。

---

### 22.3 示例三：高风险强化

#### 问题

玩家消耗稀有材料前无法理解结果。

#### 方案

强化前预览：

```text
当前属性
→ 强化后属性

消耗材料
成功概率
失败影响
是否可恢复
```

确认只在高风险材料或不可逆结果时出现。

---

### 22.4 示例四：隐藏敌人信息

#### 目标

保留探索和发现感。

#### 方案

首次遇到敌人时显示：

- 攻击类型；
- 危险等级；
- 明显行为预警；
- 已观察到的弱点。

重复战斗后逐步记录：

- 更精确属性；
- 掉落范围；
- 行为规律；
- 状态抗性。

#### 结果

信息从方向性逐步提高到精确性，而不是一开始完全隐藏或完全公开。

---

### 22.5 示例五：资源变化

#### 问题

任务结束后多个资源同时变化，玩家不知道来源。

#### 方案

结算按来源分组：

```text
任务基础奖励
额外目标奖励
消耗
惩罚
角色加成
```

总数变化保留，但允许展开明细。

## 23. 验证方法

### 23.1 状态识别测试

询问玩家：

```text
你现在处于什么状态？
哪些效果正在生效？
哪个风险最需要处理？
```

### 23.2 因果复述测试

```text
刚才为什么成功？
刚才为什么失败？
哪个因素影响最大？
```

### 23.3 预测测试

在操作前询问：

```text
你认为点击后会发生什么？
会消耗什么？
有什么风险？
```

### 23.4 注意力测试

观察：

- 玩家首先注意什么；
- 是否错过关键提示；
- 是否被非关键信息吸引；
- 高强度场景下是否仍能识别重要反馈。

### 23.5 错误恢复测试

故意制造：

- 条件不足；
- 无效目标；
- 高风险操作；
- 网络延迟；
- 多状态叠加。

观察玩家是否知道发生了什么和如何继续。

### 23.6 通道关闭测试

测试以下情况：

- 无声音；
- 无震动；
- 色觉差异；
- 动画减弱；
- 小尺寸屏幕；
- 高环境噪声。

### 23.7 定量信号

可以参考：

- 重复点击率；
- 无效操作率；
- 误操作率；
- 失败后重试率；
- 帮助入口使用率；
- 日志查看率；
- 高风险操作取消率；
- 关键提示错过率。

### 23.8 失败标准

示例：

```text
多数玩家无法识别关键状态。
玩家无法解释主要失败原因。
高风险操作前无法预测主要损失。
输入后频繁出现重复点击。
关键反馈只在低强度环境下有效。
```

## 25. V1 完成标准

清晰度与反馈体系可以被视为 V1 完成，当：

- 关键状态具有稳定可见性；
- 输入能够获得及时反馈；
- 延迟操作具有处理中状态；
- 不可用操作能够说明原因；
- 成功和失败反馈具有明确语义；
- 重要结果能够追踪主要原因；
- 高风险操作具有结果预览；
- 危险事件具有合理预警；
- 信息具有明确优先级；
- 高级信息可以按需访问；
- 关键反馈使用多通道或冗余编码；
- 反馈强度与事件重要程度匹配；
- 重复高强度反馈可以调整；
- 教学反馈与正式系统一致；
- 玩家能够通过反馈修正心智模型；
- 关键反馈通过真实场景和可访问性测试；
- 清晰度债务能够被记录和复审。

## 26. 总结

清晰度的目标不是让玩家看到所有信息，而是让玩家在正确时间看到足够的信息。

反馈的目标也不只是让操作显得有反应，而是帮助玩家建立稳定的因果理解。

一个成熟的清晰度与反馈体系应做到：

```text
状态可以被看见；
输入可以被确认；
变化可以被识别；
结果可以被理解；
原因可以被追踪；
风险可以被判断；
未来可以被大致预测；
重要性可以被区分；
反馈可以被多种玩家感知；
玩家可以据此调整下一步。
```

清晰度最终服务的不是“信息展示”，而是玩家的理解、控制、学习和信任。

当玩家能够回答：

> 我知道现在发生了什么，也知道自己为什么会得到这个结果，以及接下来可以怎样行动。

反馈才真正完成了它的设计职责。
