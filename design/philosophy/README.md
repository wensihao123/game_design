# Design Philosophy（设计理念）

> 状态：草案  
> 适用范围：通用的软件、游戏与交互产品设计  
> 目的：定义设计工作的底层价值、判断原则与长期约束，为系统设计和实例分析提供统一基础。

---

## 1. 本目录的作用

`design/philosophy/` 用于记录项目中相对稳定、能够跨功能和跨系统复用的设计理念。

这里不直接回答：

- 某个具体系统如何实现；
- 某个界面应该如何布局；
- 某个数值应该设为多少；
- 某个案例为什么成功或失败。

这里主要回答：

- 我们认为好的体验是什么；
- 设计决策优先保护什么；
- 当目标冲突时如何取舍；
- 什么样的复杂度是合理的；
- 什么样的反馈是必要的；
- 什么样的挑战是公平的；
- 什么样的留存和激励是健康的；
- 如何判断一个设计是否值得继续。

设计理念应当比具体功能稳定，但仍允许随着项目经验而更新。

## 2. Philosophy、Principle、Heuristic 与 Rule

这些概念应区分使用。

### 2.1 Philosophy：理念

理念表达长期价值取向。

例如：

```text
尊重玩家时间。
让结果可以被理解。
让深度来自组合，而不是来自信息混乱。
```

理念通常不会直接告诉设计师具体怎么做，但会影响大量后续决策。

---

### 2.2 Principle：原则

原则将理念转化为较明确的设计要求。

例如：

```text
任何重要失败都应提供可理解的原因。
重复操作应尽量支持批量或自动化处理。
新机制应优先建立在已有认知模型之上。
```

原则可以用于评审方案。

---

### 2.3 Heuristic：启发式规则

启发式规则是经验判断，不一定在所有场景成立。

例如：

```text
重要操作最好在三步以内完成。
首次教学应一次只引入一个核心概念。
危险操作通常需要确认或撤销。
```

启发式规则允许被具体情境推翻。

---

### 2.4 Rule：规则

规则是项目明确要求遵守的约束。

例如：

```text
所有不可逆删除操作必须提供二次确认。
所有核心操作必须支持键盘和手柄。
所有用户可见文本必须使用本地化 Key。
```

规则的例外应被正式记录。

## 3. 设计理念与其他目录的关系

```text
Philosophy
    ↓
Systems
    ↓
Case Studies
```

### Philosophy

定义：

- 价值；
- 原则；
- 判断方式；
- 取舍逻辑；
- 设计边界。

### Systems

定义：

- 具体机制；
- 规则；
- 状态；
- 数据；
- 交互；
- 平衡；
- 集成关系。

### Case Studies

分析：

- 一个具体设计如何工作；
- 为什么有效或无效；
- 使用了哪些理念；
- 违反了哪些原则；
- 可以借鉴什么；
- 哪些内容不能直接复制。

---

## 4. 设计理念的使用方式

设计理念不应只作为宣言存在。

它应被用于：

- Feature 设计；
- System Design；
- UI / UX Review；
- Content Review；
- Balance Review；
- Playtest；
- Bug Triage；
- Scope Decision；
- Roadmap；
- Postmortem；
- Case Study。

一个设计方案应能够回答：

```text
它支持了哪些设计原则？
它牺牲了哪些设计原则？
这种取舍是否值得？
```

## 8. 原则优先级

不同原则之间可能冲突。

例如：

```text
清晰度 vs 探索感
可预测性 vs 惊喜
深度 vs 易学
自由度 vs 引导
惩罚感 vs 可恢复性
一致性 vs 新鲜感
```

出现冲突时，不应假设所有原则都能同时最大化。

应根据：

- Core Experience；
- 目标用户；
- 使用场景；
- 风险；
- 项目阶段；
- 平台；
- 价值优先级；

进行取舍。

---

## 9. 默认优先顺序

在没有项目特定说明时，可以使用以下默认顺序：

```text
正确与安全
→ 可理解
→ 可控制
→ 可恢复
→ 核心体验
→ 深度
→ 效率
→ 新鲜感
→ 装饰性表现
```

这不是绝对规则。

但当设计牺牲前面的价值来换取后面的价值时，应提供充分理由。

## 10. 原则冲突记录

重要冲突应记录：

```text
冲突原则
当前情境
选择
理由
接受的后果
复审条件
```

示例：

```text
冲突：
清晰反馈 vs 神秘探索。

选择：
隐藏具体数值，但保留明确方向性反馈。

理由：
当前体验需要保持发现感，但不能让玩家无法判断行为是否有效。

后果：
精确优化难度上升。

复审条件：
如果测试中玩家无法建立基本因果关系，则增加更明确反馈。
```

## 11. 设计原则的证据等级

设计原则可能来自不同证据。

建议区分：

```text
Project Value
User Research
Playtest Evidence
Production Experience
Industry Heuristic
Personal Preference
```

评审时应避免将个人偏好包装成普遍原则。

---

## 12. 设计例外

原则允许例外，但例外应满足：

- 有明确目标；
- 有具体场景；
- 有风险评估；
- 有验证方式；
- 有 Owner；
- 有复审条件；
- 不自动扩展为全局规则。

例外记录可使用 Decision Record。

## 13. 如何评审一个设计

可以按以下顺序检查：

### 13.1 Goal

```text
这个设计解决什么问题？
```

### 13.2 Player Experience

```text
玩家会看到什么、理解什么、决定什么、感受什么？
```

### 13.3 Rule

```text
系统实际允许和限制什么？
```

### 13.4 Feedback

```text
玩家如何知道结果和原因？
```

### 13.5 Cost

```text
它增加了多少学习、操作、时间和维护成本？
```

### 13.6 Trade-off

```text
它牺牲了什么？
```

### 13.7 Validation

```text
如何证明它有效？
```

## 14. 设计理念不应做什么

设计理念不应：

- 成为空洞口号；
- 只描述“好玩、清晰、有深度”；
- 用来阻止所有新尝试；
- 代替用户测试；
- 代替系统设计；
- 代替具体验收标准；
- 将个人偏好绝对化；
- 假设所有项目都应采用同样答案；
- 忽略商业、技术和制作约束。

---

## 15. 设计理念的维护

设计理念应在以下情况下复审：

- 项目方向变化；
- 目标用户变化；
- 核心玩法变化；
- 大量测试结果与原则冲突；
- 新平台引入；
- 设计债务持续出现；
- 多个系统产生不一致；
- 原则无法指导实际决策；
- 原则经常被例外绕过。

## 18. 最终原则

设计理念的本质不是：

> 为设计工作添加一组听起来正确的口号。

它的本质是：

> 建立一套可以被实际使用的判断框架，让不同功能、系统和内容在面对冲突、风险和不确定性时，仍然能够做出一致而可解释的取舍。

一个有效的设计理念体系应做到：

- 价值明确；
- 原则可执行；
- 冲突可处理；
- 例外可记录；
- 证据可追踪；
- 能够进入评审；
- 能够指导系统设计；
- 能够被案例验证；
- 能够随着经验更新。
