# Core Experience and Fantasy（核心体验与玩家幻想）

> 状态：V1 草案  
> 适用范围：通用的软件、游戏与交互产品设计  
> 核心问题：项目究竟希望玩家持续体验什么、扮演什么、感受什么，以及所有系统如何共同强化这一承诺  
> 目的：建立一套用于定义、筛选、评审和验证核心体验的统一框架，防止功能堆积、体验漂移与系统割裂

---

## 1. 定义

核心体验，是玩家在反复游玩过程中持续获得、并最能代表项目身份的体验。

玩家幻想，是玩家希望“成为谁”“做什么”“以什么方式影响世界”的主观想象。

情绪目标，是设计希望玩家在某个阶段产生的主要感受。

核心循环，是玩家为了获得核心体验而反复进行的一组行为与反馈。

体验支柱，是为了稳定实现核心体验而长期坚持的少数设计方向。

它们之间的关系可以表示为：

```text
Player Fantasy
    ↓
Core Experience
    ↓
Emotional Goal
    ↓
Experience Pillars
    ↓
Core Loop
    ↓
Systems / Content / UI / Feedback
```

例如：

```text
玩家幻想：
成为一名带领小队成长的冒险团长。

核心体验：
观察队伍表现、调整阵容与资源，让一支弱小队伍逐渐形成独特战斗风格。

情绪目标：
安心、期待、掌控、成长满足。

体验支柱：
清晰成长、轻量决策、队伍表达、低压失败。

核心循环：
准备 → 出征 → 观察 → 获得结果 → 调整队伍 → 再次出征。
```

核心体验不是一句宣传语，也不是功能列表。

它必须能够影响：

- 哪些系统应该存在；
- 哪些功能应该削减；
- 哪些内容值得投入；
- 哪些反馈必须强化；
- 哪些复杂度可以接受；
- 哪些商业或技术妥协会破坏项目身份。

## 2. 为什么核心体验重要

一个项目可以拥有很多功能，但玩家最终记住的通常不是功能数量，而是反复出现的体验模式。

例如，玩家可能记住的是：

- 每次战斗前规划队伍的期待；
- 在危险环境中艰难生存的紧张；
- 通过观察线索逐渐理解世界的满足；
- 收集、布置和照顾空间的安定感；
- 操作角色形成流畅动作的爽快感；
- 与朋友共同解决问题的协作感。

如果核心体验不明确，项目容易出现以下问题：

```text
每个系统单独看都合理，但彼此服务不同目标。
功能不断增加，却没有明显强化项目特色。
新内容只增加数量，没有带来新的体验变化。
系统复杂度持续上升，玩家却无法说明游戏核心是什么。
团队对“什么最重要”缺乏一致判断。
项目开始模仿市场上流行功能，逐渐失去自身方向。
```

核心体验的价值，在于建立一个长期稳定的筛选器：

```text
这个设计是否强化玩家来到这里最想获得的体验？
```

## 3. 核心体验不是题材、类型或机制

这些概念经常被混淆。

### 3.1 题材

题材回答：

```text
故事和世界表面上关于什么？
```

例如：

- 中世纪幻想；
- 太空探索；
- 现代都市；
- 农场生活；
- 末日生存。

题材可以支持体验，但不能替代体验。

同样是中世纪幻想，可以提供：

- 高压生存；
- 英雄冒险；
- 政治策略；
- 轻松经营；
- 战术战斗。

### 3.2 类型

类型回答：

```text
游戏大致属于什么结构传统？
```

例如：

- RPG；
- Roguelike；
- 模拟经营；
- 动作游戏；
- 卡牌构筑；
- 解谜游戏。

类型提供玩家预期，但无法完整定义项目身份。

### 3.3 机制

机制回答：

```text
玩家和系统具体如何互动？
```

例如：

- 卡牌抽取；
- 格挡；
- 合成；
- 资源生产；
- 队伍编成；
- 随机地图。

机制是实现体验的工具，而不是体验本身。

### 3.4 核心体验

核心体验回答：

```text
玩家通过这些机制反复获得什么样的认知、决策与感受？
```

例如：

```text
不是“卡牌战斗”，而是：
在有限信息下组合手牌，逐步构建一套能够自我运转的策略。

不是“农场经营”，而是：
通过缓慢照顾和布置，让一个属于自己的空间逐渐充满生命。

不是“自动战斗”，而是：
在战斗前做关键准备，然后观察自己的队伍如何执行计划。
```

## 4. 玩家幻想

玩家幻想不是世界观设定本身，而是玩家在体验中希望实现的身份、能力与作用。

### 4.1 身份幻想

玩家希望成为什么。

例如：

- 强大的战士；
- 精明的管理者；
- 可靠的队长；
- 神秘世界的探索者；
- 创造独特空间的设计者；
- 照顾角色成长的陪伴者。

### 4.2 能力幻想

玩家希望拥有什么能力。

例如：

- 击败强敌；
- 预判复杂局势；
- 发现隐藏规律；
- 建立高效系统；
- 保护同伴；
- 创造和改变世界。

### 4.3 作用幻想

玩家希望自己对世界产生什么影响。

例如：

- 恢复一座城镇；
- 帮助角色成长；
- 改变阵营关系；
- 建立繁荣组织；
- 解决长期危机；
- 创造独特战术体系。

### 4.4 表达幻想

玩家希望如何表达自己。

例如：

- 选择角色组合；
- 设计外观；
- 建立个人空间；
- 选择道德立场；
- 形成独特打法；
- 发展偏爱的角色。

### 4.5 关系幻想

玩家希望与角色或其他玩家建立怎样的关系。

例如：

- 领导；
- 陪伴；
- 保护；
- 合作；
- 竞争；
- 教导；
- 被认可。

### 4.6 玩家幻想评审问题

```text
玩家希望在这个项目中成为什么？
玩家最想拥有哪种能力？
玩家希望自己的决定改变什么？
哪些系统让这种身份和能力变得可信？
哪些系统只在文本中声称幻想存在，却没有让玩家实际行动？
```

## 5. 核心体验

核心体验应描述玩家反复进行的认知、选择、行动和感受。

一个较完整的核心体验陈述可以包含：

```text
玩家身份
+ 主要行为
+ 决策方式
+ 反馈形式
+ 情绪结果
```

模板：

```text
玩家作为【身份】，
通过【主要行为】，
不断进行【核心决策】，
并从【反馈和变化】中，
获得【主要情绪与满足】。
```

示例：

```text
玩家作为一名冒险小队的管理者，
通过组织角色、分配资源和选择任务，
不断判断当前队伍应该承担何种风险，
并从战斗表现和长期成长中，
获得掌控、期待与培养成果逐渐兑现的满足。
```

### 5.1 好的核心体验陈述

应当：

- 描述玩家而不是系统；
- 描述重复体验而不是一次性事件；
- 能够解释多个核心系统；
- 能够产生明确取舍；
- 与目标玩家匹配；
- 可以通过实际游玩验证；
- 足够具体，不能适用于所有游戏；
- 足够稳定，不会随单个功能变化。

### 5.2 无效核心体验陈述

过于宽泛：

```text
让玩家获得快乐。
提供有趣而有深度的体验。
打造沉浸式世界。
```

过于功能化：

```text
玩家抽卡、升级、装备角色并进行战斗。
```

过于宣传化：

```text
开启一段前所未有的史诗冒险。
```

过于结果化：

```text
让玩家持续游玩三个月。
```

## 6. 核心体验的层级

项目通常不只有一个体验，但需要明确层级。

### 6.1 Primary Experience

项目最不可替代的主要体验。

如果移除它，项目身份将发生根本变化。

### 6.2 Supporting Experience

支持主要体验的次级体验。

例如：

- 收集；
- 装饰；
- 社交；
- 剧情；
- 探索；
- 资源管理。

这些体验应当强化主要体验，而不是争夺全部注意力。

### 6.3 Contextual Experience

只在特定阶段、模式或内容中出现。

例如：

- Boss 战紧张感；
- 城镇休息感；
- 教学阶段安全感；
- 高难模式压迫感。

### 6.4 Decorative Experience

主要用于表现、氛围和丰富性。

例如：

- 短动画；
- 环境互动；
- 非核心小游戏；
- 收藏展示。

装饰性体验可以提升品质，但不应消耗过多核心资源。

### 6.5 体验层级判断

```text
如果这个体验被移除，项目是否仍然成立？
它是否服务主要体验？
它是否与主要体验争夺时间、注意力和制作成本？
它应该在所有阶段出现，还是只在特定情境出现？
```

## 7. 情绪目标

情绪目标用于描述玩家在体验中的主要感受。

### 7.1 情绪不是单一常量

一个项目可以有多种情绪，但应明确：

- 基础情绪；
- 高峰情绪；
- 恢复情绪；
- 长期情绪。

例如：

```text
基础情绪：轻松、可控。
高峰情绪：紧张、兴奋。
恢复情绪：安心、整理。
长期情绪：陪伴、成长满足。
```

### 7.2 情绪来源

情绪不是通过文案声明产生，而是由实际体验结构形成。

例如：

```text
安心：
来自低惩罚、稳定节奏、柔和反馈和可恢复失败。

紧张：
来自明确风险、有限资源和不可立即逆转的决定。

掌控：
来自可理解规则、有效选择和稳定反馈。

期待：
来自未来目标、未揭示内容和逐步兑现的成长。

成就：
来自真实挑战、能力提升和结果认可。
```

### 7.3 情绪目标与系统关系

每个重要情绪目标都应有系统支持。

```markdown
| 情绪目标 | 主要来源 | 可能破坏因素 |
|---|---|---|
| 掌控 | 清晰规则、有效选择、及时反馈 | 隐藏机制、任意随机 |
| 期待 | 可见目标、逐步解锁、未来可能性 | 奖励重复、目标不透明 |
| 安心 | 低损失、可恢复、稳定节奏 | 强制时限、永久损失 |
| 成长满足 | 能力变化、内容变化、反馈强化 | 纯数字膨胀、成长无感 |
```

### 7.4 情绪目标评审问题

```text
玩家此刻应该主要感受到什么？
这种情绪来自哪些具体规则和反馈？
是否存在系统持续制造相反情绪？
情绪是否与目标用户和会话场景匹配？
高强度情绪后是否存在恢复空间？
```

## 8. 体验支柱

体验支柱是支持核心体验的少数长期方向。

通常建议保留三到五项。

过多支柱会失去优先级。

### 8.1 体验支柱的结构

每项支柱应包含：

```text
支柱名称
→ 保护的体验
→ 设计含义
→ 不意味着什么
→ 评审问题
```

示例：

```markdown
## 支柱：轻量但有意义的决策

保护的体验：
玩家可以在短时间内完成判断，但选择会明显影响队伍表现。

设计含义：
- 选项数量保持有限；
- 关键差异清晰；
- 决策集中在准备阶段；
- 结果提供可理解反馈。

不意味着：
- 所有系统都必须简单；
- 玩家不需要长期学习；
- 选择可以没有风险。

评审问题：
- 玩家是否理解主要差异？
- 选择是否改变结果？
- 决策成本是否符合会话长度？
```

### 8.2 支柱的类型

体验支柱可能关注：

- 决策方式；
- 情绪基调；
- 成长方式；
- 玩家表达；
- 信息透明；
- 挑战风格；
- 节奏；
- 社交关系；
- 内容组织。

### 8.3 支柱之间的关系

支柱不应只是并列口号。

需要检查：

- 是否共同支持同一个核心体验；
- 是否存在长期冲突；
- 冲突时谁优先；
- 是否覆盖核心体验的关键维度；
- 是否有重复支柱可以合并。

## 9. Core Loop（核心循环）

核心循环是玩家反复进行、并持续获得核心体验的行为结构。

### 9.1 核心循环不是流程图本身

流程图可以描述玩家做了什么，但核心循环还需要说明：

- 玩家为什么开始；
- 在哪里做出决定；
- 得到什么反馈；
- 为什么愿意再次进入循环；
- 循环每次如何产生变化。

### 9.2 核心循环基本结构

```text
Motivation
→ Preparation
→ Action
→ Feedback
→ Reward / Consequence
→ Adjustment
→ New Motivation
```

例如：

```text
看到新的挑战
→ 调整队伍
→ 进入战斗
→ 观察表现
→ 获得资源和结果
→ 强化角色或改变组合
→ 挑战更高目标
```

### 9.3 循环中的体验职责

#### Motivation

让玩家知道为什么值得开始。

#### Preparation

提供预测、计划和表达空间。

#### Action

让玩家执行核心行为。

#### Feedback

帮助玩家理解过程和结果。

#### Reward / Consequence

让行动产生实际变化。

#### Adjustment

让玩家根据结果学习和改变。

#### New Motivation

产生下一轮目标。

### 9.4 循环必须产生变化

如果每次循环只增加数值，而没有改变：

- 决策；
-能力；
- 目标；
- 风险；
- 内容；
- 表达；

循环容易变成重复劳动。

### 9.5 核心循环与元循环

可以区分：

```text
Moment-to-Moment Loop
Session Loop
Meta Loop
Long-Term Loop
```

#### Moment-to-Moment Loop

秒级到分钟级。

例如：

```text
观察攻击 → 选择应对 → 执行动作 → 获得反馈。
```

#### Session Loop

一次游玩中的主要结构。

例如：

```text
选择任务 → 准备 → 战斗 → 结算 → 调整。
```

#### Meta Loop

多次会话之间的成长结构。

例如：

```text
积累资源 → 解锁能力 → 形成构筑 → 挑战更高内容。
```

#### Long-Term Loop

支撑长期目标和项目生命周期。

例如：

```text
建立队伍身份 → 完成长期区域 → 追求高阶挑战 → 扩展新玩法。
```

不同循环应相互支持，而不是各自追求不同方向。

## 10. 核心体验与系统

每个主要系统都应明确自己如何支持核心体验。

### 10.1 系统支持矩阵

```markdown
| 系统 | 支持的体验支柱 | 核心贡献 | 主要成本 | 偏离风险 |
|---|---|---|---|---|
| 队伍编成 | 队伍表达、轻量决策 | 形成策略差异 | 学习成本 | 角色差异不明显 |
| 战斗 | 观察反馈、计划兑现 | 展示准备结果 | 注意力成本 | 结果不可解释 |
| 成长 | 长期培养 | 提供持续变化 | 重复成本 | 纯数值膨胀 |
| 任务 | 目标与风险选择 | 推动循环 | 内容成本 | 任务只做入口 |
```

### 10.2 系统贡献类型

系统可以通过以下方式支持核心体验：

- 直接承载核心行为；
- 提供决策条件；
- 展示结果；
- 提供成长变化；
- 提供情绪节奏；
- 支持玩家表达；
- 建立长期目标；
- 降低无关成本。

### 10.3 系统必须有存在理由

每个系统应能回答：

```text
如果没有这个系统，核心体验会损失什么？
它提供的体验是否已由其他系统提供？
它增加的复杂度是否值得？
它是否把玩家注意力从主要体验移开？
```

## 11. 核心体验与内容

内容不是独立价值，内容需要承载体验变化。

### 11.1 内容变化的层级

#### 表现变化

外观、文本、场景和音效不同，但玩法结构基本相同。

#### 参数变化

数值、速度、规模、概率不同。

#### 规则变化

玩家需要理解和应对新的规则关系。

#### 决策变化

玩家需要做出不同判断。

#### 情绪变化

内容改变节奏、风险和感受。

#### 表达变化

内容允许玩家形成新的风格和身份。

高价值内容通常不只停留在表现或参数变化。

### 11.2 内容评审问题

```text
这项内容提供了什么新的体验？
它改变玩家的决策，还是只改变数值？
它是否强化玩家幻想？
它是否支持某个体验支柱？
它是否引入与收益不匹配的新规则？
它是否值得对应的制作和维护成本？
```

## 12. 体验承诺

项目会通过宣传、教学、系统和长期表现形成对玩家的承诺。

例如：

```text
你的队伍组合会影响战斗结果。
失败可以帮助你理解问题。
角色成长会带来新的玩法可能。
离线不会让你失去核心进度。
付费不会绕过核心策略价值。
```

### 12.1 显性承诺

通过以下方式直接表达：

- 商店页面；
- 宣传材料；
- 教学；
- 系统说明；
- 任务目标；
- 文本。

### 12.2 隐性承诺

通过反复体验建立：

- 某类攻击始终有预警；
- 某类失败始终可恢复；
- 某类资源始终具有稳定用途；
- 某类选择始终产生明显差异。

### 12.3 承诺一致性

当项目长期强调某种体验，却通过系统奖励相反行为时，会破坏承诺。

例如：

```text
声称鼓励自由构筑，但只有少数角色组合有效。
声称尊重时间，但核心成长依赖大量重复操作。
声称轻松低压，但使用强制限时和错过损失。
声称策略重要，但结果主要由隐藏随机性决定。
```

---

## 13. 体验偏离

体验偏离是指设计逐渐不再服务核心体验，甚至开始与其竞争。

### 13.1 偏离来源

常见来源包括：

- 模仿竞品功能；
- 局部指标优化；
- 新成员缺乏核心体验理解；
- 内容更新压力；
- 商业需求；
- 技术能力展示；
- 既有系统扩张；
- 用户反馈的表面响应；
- 临时活动成为长期系统。

### 13.2 偏离信号

```text
团队难以解释新功能如何支持体验支柱。
功能主要理由是“其他游戏都有”。
玩家在核心循环外花费越来越多时间。
新系统需要大量例外才能与既有规则共存。
界面和教学成本快速增加。
玩家对项目的描述与设计目标明显不同。
核心行为被资源领取、管理或等待替代。
```

### 13.3 偏离不一定是错误

项目可以主动改变核心体验，但必须被视为方向调整，而不是局部功能添加。

方向调整应重新评估：

- 目标用户；
- 体验支柱；
- 核心循环；
- 系统优先级；
- 内容结构；
- 品牌承诺；
- 已有玩家预期。

---

## 14. 功能筛选框架

新增功能可以按以下问题评估。

### 14.1 Core Fit

```text
它直接支持哪个核心体验或体验支柱？
```

### 14.2 Player Value

```text
它解决了什么玩家问题？
它提供了什么新的体验价值？
```

### 14.3 Differentiation

```text
它是否强化项目独特性？
还是只是补齐常见功能清单？
```

### 14.4 Complexity Cost

```text
它增加多少规则、状态、界面、内容和维护成本？
```

### 14.5 Attention Cost

```text
它会占用多少玩家时间和注意力？
是否挤压核心循环？
```

### 14.6 System Interaction

```text
它是否与现有系统形成清晰关系？
是否需要大量例外？
```

### 14.7 Validation

```text
最关键假设是什么？
如何低成本验证？
```

### 14.8 功能分类

可以将功能分为：

```text
Core：直接承载核心体验。
Support：支持核心体验。
Quality：改善可用性和完成度。
Expansion：扩展新的体验空间。
Decorative：主要提供表现和丰富性。
Distracting：与核心体验竞争。
```

`Distracting` 功能不一定必须删除，但需要非常明确的理由。

## 15. 体验预算

项目资源有限，因此体验也需要预算。

### 15.1 复杂度预算

每个新系统都会消耗：

- 学习容量；
- 记忆容量；
- 界面空间；
- 教学时间；
- 平衡成本；
- 测试成本；
- 维护成本。

复杂度应优先投入：

- 核心决策；
- 长期深度；
- 玩家表达；
- 核心体验变化。

### 15.2 注意力预算

玩家在一次会话中能够处理的目标有限。

应避免同时要求玩家：

- 管理多个进度；
- 领取多个奖励；
- 处理多个红点；
- 学习多个系统；
- 完成多个限时目标。

### 15.3 制作预算

高成本内容应优先服务：

- 核心循环；
- 情绪高峰；
- 玩家幻想；
- 长期记忆点。

### 15.4 情绪预算

持续高强度会造成疲劳。

持续低强度会造成平淡。

需要在：

- 紧张与释放；
- 决策与观察；
- 风险与安全；
- 新鲜与熟悉；

之间安排节奏。

## 17. 设计评审问题

### 17.1 玩家幻想

```text
玩家希望成为什么？
系统是否真的允许玩家扮演这一身份？
玩家是否拥有与幻想匹配的能力和影响力？
```

### 17.2 核心体验

```text
玩家最常重复的行为是什么？
这些行为是否产生项目承诺的体验？
核心体验是否能被一句具体的话表达？
```

### 17.3 情绪

```text
玩家当前主要感受是什么？
这种感受来自哪些规则和反馈？
是否存在系统持续制造相反情绪？
```

### 17.4 支柱

```text
当前设计支持哪个支柱？
它是否同时破坏另一个更重要支柱？
支柱是否足够明确，可以排除方案？
```

### 17.5 核心循环

```text
玩家为什么进入循环？
循环中的主要决策在哪里？
玩家如何知道自己的选择产生了作用？
循环每次如何变化？
```

### 17.6 系统

```text
如果移除该系统，核心体验会损失什么？
它是否与其他系统重复？
它增加的复杂度是否值得？
```

### 17.7 内容

```text
新内容带来了新的决策、情绪或表达吗？
还是只增加数量和参数？
```

### 17.8 偏离

```text
该功能是否让玩家远离核心循环？
它存在的主要理由是否只是市场惯例？
它是否正在改变项目身份？
```

### 17.9 验证

```text
玩家如何描述这个项目？
他们最享受的部分是否与目标一致？
哪些行为证据能够说明核心体验真的成立？
```

## 18. 检查清单

### 18.1 核心体验定义

- [ ] 核心体验以玩家行为和感受描述；
- [ ] 核心体验不是题材、类型或功能列表；
- [ ] 核心体验足够具体；
- [ ] 核心体验可以影响取舍；
- [ ] 核心体验能够被实际验证；
- [ ] 核心体验与目标玩家匹配。

### 18.2 玩家幻想

- [ ] 已定义玩家身份；
- [ ] 已定义核心能力；
- [ ] 已定义玩家对世界的影响；
- [ ] 已定义表达空间；
- [ ] 系统实际支持幻想，而不只是文本声明；
- [ ] 失败方式不会持续破坏玩家身份。

### 18.3 情绪目标

- [ ] 已定义基础情绪；
- [ ] 已定义高峰与恢复；
- [ ] 每种主要情绪具有系统来源；
- [ ] 系统不会持续产生相反情绪；
- [ ] 情绪强度符合会话和目标用户。

### 18.4 体验支柱

- [ ] 支柱数量保持有限；
- [ ] 每项支柱保护明确体验；
- [ ] 每项支柱能够影响评审；
- [ ] 支柱之间不存在未处理冲突；
- [ ] 支柱不是重复口号；
- [ ] 支柱能够映射到具体系统。

### 18.5 核心循环

- [ ] 循环具有明确动机；
- [ ] 循环包含真实决策；
- [ ] 行动结果具有清晰反馈；
- [ ] 奖励或后果支持下一轮调整；
- [ ] 循环每次能够产生变化；
- [ ] Moment、Session、Meta 和长期循环相互支持。

### 18.6 系统与内容

- [ ] 每个主要系统具有体验贡献；
- [ ] 系统复杂度与核心价值匹配；
- [ ] 新内容不只提供参数变化；
- [ ] 核心行为没有被管理与领取流程淹没；
- [ ] 装饰性功能没有占用过多资源；
- [ ] 新功能没有依赖大量例外。

### 18.7 承诺与偏离

- [ ] 显性宣传与实际体验一致；
- [ ] 隐性体验承诺得到保护；
- [ ] 新功能能够说明 Core Fit；
- [ ] 已识别功能的注意力与复杂度成本；
- [ ] 重大方向变化被正式记录；
- [ ] 项目定期进行体验偏离审查。

## 19. 常见失败模式

### 19.1 将核心体验写成宣传语

表现：

```text
史诗级冒险。
前所未有的沉浸。
轻松又有深度。
```

问题：

无法指导设计，也无法验证。

修正：

描述玩家反复做什么、决定什么、感受什么。

### 19.2 用题材替代体验

表现：

```text
核心体验是经营一家魔法酒馆。
```

问题：

只说明情境，没有说明玩家如何经营以及获得何种满足。

修正：

进一步描述：

```text
通过观察客人需求、安排有限资源和改善空间，
让一间混乱的小酒馆逐渐形成属于玩家自己的经营节奏。
```

### 19.3 用功能列表替代体验

表现：

```text
核心体验包括战斗、升级、装备、抽卡和任务。
```

问题：

只说明系统存在，没有说明它们共同服务什么。

### 19.4 玩家幻想只存在于剧情

表现：

文本称玩家是伟大指挥官，但实际玩法中没有重要决策。

问题：

身份与行为不一致。

修正：

让系统赋予玩家与身份匹配的能力、责任和反馈。

### 19.5 支柱数量过多

表现：

项目拥有十多个同等重要的体验支柱。

问题：

无法排序，也无法进行范围取舍。

修正：

保留三到五项真正定义项目身份的支柱。

### 19.6 支柱没有排除能力

表现：

```text
好玩、清晰、有深度、有品质。
```

问题：

任何方案都可以声称符合。

修正：

明确设计含义、不意味着什么和冲突处理。

### 19.7 核心循环没有决策

表现：

玩家只是重复点击、等待和领取。

问题：

循环依赖奖励驱动，而不是体验驱动。

修正：

增加判断、表达、技能或可感知变化，或减少无价值重复。

### 19.8 奖励掩盖循环问题

表现：

通过更高奖励要求玩家继续完成乏味内容。

问题：

短期行为可能提升，核心体验没有改善。

### 19.9 所有系统争夺核心地位

表现：

战斗、经营、收集、社交和剧情都要求大量时间。

问题：

玩家注意力分散，项目身份模糊。

修正：

明确主体验、支持体验和情境体验。

### 19.10 内容只做数值替换

表现：

新敌人只有更高生命和伤害，新装备只有更高属性。

问题：

长期内容不改变决策，成长变成数字膨胀。

### 19.11 为完整性添加功能

表现：

```text
同类游戏都有公会，所以我们也需要。
RPG 都应该有锻造系统。
```

问题：

功能来源于类型清单，而不是核心体验。

### 19.12 局部指标导致体验漂移

表现：

红点、每日任务和限时奖励持续增加，因为短期活跃数据提升。

问题：

核心体验逐渐被管理负担和压力替代。

### 19.13 把方向改变当作普通迭代

表现：

项目从轻松经营逐渐变成高压竞争，却没有重新定义目标用户和体验承诺。

问题：

新旧系统持续冲突，玩家预期被破坏。

## 20. 与其他原则的冲突

### 20.1 核心体验 vs 玩家便利

某些复杂操作可能服务核心体验。

处理方式：

- 保留表达技能和决策的成本；
- 移除与核心体验无关的摩擦；
- 不把界面不便当作体验深度。

### 20.2 核心体验 vs 内容多样性

过度多样可能稀释项目身份。

处理方式：

- 允许情境变化；
- 保持底层体验支柱稳定；
- 新内容应改变表达，而不是改变项目根本方向。

### 20.3 玩家幻想 vs 公平性

强大幻想可能与挑战、公平或多人平衡冲突。

处理方式：

- 让玩家感到有能力，而不必保证永远胜利；
- 通过成长、反馈和表达强化幻想；
- 不依赖破坏规则一致性的特殊优待。

### 20.4 情绪目标 vs 系统效率

仪式、演出和等待可能支持情绪，但降低效率。

处理方式：

- 首次体验保留完整表达；
- 重复后允许跳过或加速；
- 将高成本表现集中在关键节点。

### 20.5 核心体验 vs 商业目标

商业机制可能挤压核心循环。

处理方式：

- 商业价值应建立在核心体验之上；
- 不通过破坏体验再出售修复；
- 不让付费成为绕过核心玩法的唯一方式；
- 定期检查商业系统占用的时间和注意力。

### 20.6 核心体验 vs 技术展示

新技术不一定产生新体验。

处理方式：

- 先定义玩家价值；
- 技术应服务体验，而不是反过来寻找使用场景；
- 高成本技术功能必须证明其体验贡献。

### 20.7 核心体验 vs 用户反馈

玩家可能要求与核心方向冲突的功能。

处理方式：

- 理解反馈背后的问题；
- 评估是否存在不改变核心方向的解决方案；
- 不因为单一反馈立即扩展项目边界。

## 21. 示例

### 21.1 示例一：队伍管理型冒险游戏

#### 玩家幻想

成为一名培养并指挥小型冒险队伍的团长。

#### 核心体验

通过观察任务风险、角色能力和战斗结果，持续调整队伍组合，让一支普通队伍逐渐形成独特而可靠的战斗方式。

#### 情绪目标

```text
基础：轻松、可控。
高峰：期待、紧张。
恢复：整理、安心。
长期：陪伴、成长满足。
```

#### 体验支柱

```text
清晰的队伍成长
轻量但有意义的准备决策
战斗结果可解释
失败低压但具有学习价值
角色组合支持表达
```

#### 核心循环

```text
选择任务
→ 评估风险
→ 调整队伍
→ 出征
→ 观察战斗
→ 获得反馈与资源
→ 培养和调整
→ 挑战新目标
```

#### 偏离风险

- 增加大量手动微操，使准备决策失去核心地位；
- 依赖隐藏公式，让玩家无法理解队伍表现；
- 通过每日任务占用主要会话时间；
- 角色成长只有数值提升，没有组合变化。

---

### 21.2 示例二：低压空间经营游戏

#### 玩家幻想

拥有并逐渐塑造一个属于自己的温暖空间。

#### 核心体验

通过收集、布置和照顾，让空间逐渐反映玩家偏好，并从缓慢变化中获得稳定的陪伴与满足。

#### 体验支柱

- 低压成长；
- 可见变化；
- 高表达空间；
- 失败可恢复；
- 节奏由玩家掌握。

#### 可能冲突

限时活动能够制造短期目标，但会破坏低压和自主节奏。

#### 处理

活动内容可以长期保留，限时部分只提供轻量表达奖励，不阻断核心内容。

---

### 21.3 示例三：高压生存游戏

#### 玩家幻想

在资源极度有限的危险环境中，依靠判断和准备坚持生存。

#### 核心体验

持续评估风险、分配资源并承担选择后果，在不确定环境中逐渐建立生存能力。

#### 体验支柱

- 风险可感知；
- 资源选择有代价；
- 失败具有信息；
- 环境保持威胁；
- 成功来自准备而不是纯随机。

#### 不适合的设计

- 完全无损失败；
- 隐藏关键风险；
- 随机永久损失且无法准备；
- 过度自动化核心资源决策。

---

### 21.4 示例四：功能筛选

#### 提案

为队伍管理游戏增加实时 PVP。

#### 评估

```text
Core Fit：
弱。项目核心是低压培养和观察，不是高压实时对抗。

玩家价值：
可能满足少数竞争型玩家。

复杂度成本：
极高，需要实时同步、平衡、匹配和反作弊。

注意力成本：
会将角色平衡和成长设计转向竞技公平。

体验风险：
可能破坏低压、表达和长期培养。

结论：
不作为核心功能。若需要社交比较，可优先探索异步挑战或非损失型展示。
```

## 22. 验证方法

### 22.1 自由描述

让玩家回答：

```text
你觉得这个游戏主要在做什么？
最吸引你的部分是什么？
你为什么愿意继续下一轮？
```

玩家描述不必与设计术语一致，但应与目标体验接近。

### 22.2 行为时间分布

观察玩家时间主要花在哪里。

如果项目声称核心是队伍决策，但玩家大部分时间用于：

- 领取奖励；
- 处理背包；
- 等待；
- 切换页面；
- 重复刷取；

说明实际体验可能已经偏离。

### 22.3 决策回忆

让玩家复述：

- 做过哪些重要决定；
- 哪个决定改变了结果；
- 为什么调整下一轮策略。

### 22.4 情绪曲线

记录玩家在不同阶段的：

- 兴奋；
- 紧张；
- 困惑；
- 无聊；
- 安心；
- 成就。

检查实际情绪是否与设计目标一致。

### 22.5 支柱验证

每项支柱都应对应可观察行为。

例如：

```text
支柱：战斗结果可解释。
验证：玩家能否指出队伍失败的主要原因。

支柱：角色组合支持表达。
验证：不同玩家是否形成可行且明显不同的队伍。

支柱：低压失败。
验证：失败后玩家是否愿意再次尝试，且没有明显损失焦虑。
```

### 22.6 偏离审查

定期统计：

- 核心循环时间占比；
- 管理流程时间占比；
- 新系统数量；
- 规则例外数量；
- 教学和界面增长；
- 玩家对项目的主要描述；
- 核心支柱相关问题。

### 22.7 原型验证

优先用低成本原型验证：

- 核心循环是否愿意重复；
- 主要决策是否真实存在；
- 反馈是否支持学习；
- 情绪目标是否能够产生。

如果核心体验在低表现原型中仍然不成立，增加美术、奖励和内容通常无法从根本上解决问题。

## 24. V1 完成标准

核心体验与玩家幻想体系可以被视为 V1 完成，当：

- 目标玩家和主要情境已经明确；
- 玩家幻想已定义身份、能力、影响和表达；
- 核心体验能够用具体陈述表达；
- 主体验、支持体验和情境体验已经区分；
- 基础、高峰、恢复和长期情绪已经定义；
- 体验支柱保持在可管理数量；
- 每项支柱具有明确设计含义和验证方式；
- 核心循环包含动机、决策、反馈和调整；
- Moment、Session、Meta 和长期循环相互支持；
- 每个主要系统能够说明体验贡献；
- 内容评审能够区分表现、参数和决策变化；
- 项目的显性与隐性体验承诺已经记录；
- 新功能具有 Core Fit 和复杂度评估；
- 体验偏离信号和复审机制已经建立；
- 核心体验能够通过原型、行为和玩家描述验证。

## 25. 总结

核心体验定义项目希望玩家长期重复获得什么。

玩家幻想定义玩家希望在其中成为谁、拥有何种能力，并如何影响世界。

情绪目标定义这些行为应当产生怎样的感受。

体验支柱将抽象体验转化为长期设计方向。

核心循环则让这些方向真正进入玩家反复执行的行为结构。

一个成熟的核心体验体系应做到：

```text
玩家身份明确；
主要行为明确；
核心决策明确；
反馈关系明确；
情绪目标明确；
支柱能够取舍；
循环能够变化；
系统共同支持；
内容持续强化；
承诺保持一致；
偏离能够被发现；
体验可以被验证。
```

核心体验的作用不是限制所有创意，而是确保新的创意能够共同回答同一个问题：

> 它是否让玩家更强烈、更清晰、更持续地获得这个项目最值得存在的体验？
