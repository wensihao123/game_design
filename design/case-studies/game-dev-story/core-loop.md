## 1. 文档目的

本文件聚焦《游戏发展国》的核心循环，回答：

```text
玩家为什么会不断开始下一款游戏；
外包工作与自研游戏分别承担什么职责；
一个游戏项目如何从立项进入开发、Debug、评论和销量；
项目结果如何转化为下一轮资源；
短周期、项目周期、公司周期和行业周期如何相互嵌套；
失败为什么通常不会终止循环；
中后期核心循环为什么会从探索转向优化。
```

本文件重点分析：

- 主循环结构；
- 外包循环；
- 自研游戏循环；
- 项目内部循环；
- 员工行动循环；
- Bug 与 Debug；
- 评论结算；
- 销量结算；
- 资源回流；
- 成长回流；
- 项目并行与节奏；

- 循环中的反馈；
- 循环驱动因素；
- 中后期循环衰减；
- 可复用循环模式。

## 2. 核心循环总定义

《游戏发展国》的核心循环可以压缩为：

```text
选择一个项目
→ 投入人员、资金与知识
→ 自动完成生产
→ 观察质量和风险
→ 发行并接受评价
→ 获得销量、收入和粉丝
→ 将结果投入下一项目
```

更完整地表示：

```text
观察公司状态
→ 决定做外包还是自研游戏
→ 为项目选择平台、类型、题材和方向
→ 分配员工或外部专家
→ 员工持续产出开发属性
→ 处理体力、随机事件和 Bug
→ 完成 Debug
→ 发行作品
→ 逐个揭晓评论
→ 观察周销量、排名和收入
→ 获得资金、研究、粉丝、经验和解锁
→ 招聘、训练、转职、授权或扩张
→ 开始下一项目
```

### 2.1 核心循环的本质

这不是：

```text
不断点击来生产资源
```

而是：

```text
反复发起一个有起点、有过程、有悬念、有结算的项目。
```

### 2.2 为什么项目是正确的循环单位

游戏开发这个主题天然适合以项目为单位，因为一个项目可以同时容纳：

- 创意选择；
- 人员配置；
- 资源投入；
- 时间推进；
- 风险事件；
- 质量成长；
- 市场评价；
- 商业结果。

这使每个项目都能形成完整的体验波形。

## 3. 循环层级

《游戏发展国》的循环至少可以分为五层。

### 3.1 员工行动循环

时间尺度：秒级。

```text
员工开始工作
→ 产生属性点
→ 消耗体力
→ 可能触发特殊状态
→ 回到工作或休息
```

作用：

- 让自动生产过程可见；
- 提供即时反馈；
- 让员工有角色感；
- 制造小波动。

### 3.2 开发阶段循环

时间尺度：短分钟级。

```text
阶段开始
→ 选择内部人员或外包
→ 产生对应属性
→ 处理结果
→ 进入下一阶段
```

作用：

- 把一个大项目切成多个小结点；
- 让不同员工能力有使用场景；
- 增加项目中的决策密度。

### 3.3 项目循环

时间尺度：数分钟。

```text
立项
→ 开发
→ Debug
→ 发行
→ 评论
→ 销售
```

作用：

- 构成核心游玩单位；
- 提供最完整的情绪结算；
- 连接全部主要系统。

### 3.4 公司循环

时间尺度：多个项目。

```text
项目回报
→ 员工成长
→ 公司扩张
→ 更高项目能力
→ 更高项目回报
```

作用：

- 提供长期复利；
- 改变资源约束；
- 解锁新策略。

### 3.5 行业循环

时间尺度：游戏年份。

```text
新平台出现
→ 平台增长
→ 平台成熟
→ 平台衰退
→ 新平台替代
```

作用：

- 改变外部环境；
- 防止核心循环完全静态；
- 推动公司迁移战略。

## 4. 主循环中的两条路径

玩家在许多时间点面对两种主要工作路径：

```text
外包工作
```

和：

```text
自研游戏
```

这两条路径不是平级内容，而是分别承担不同功能。

---

## 5. 外包循环

### 5.1 外包循环定义

```text
选择外包项目
→ 员工完成指定工作
→ 产生有限研究与收入
→ 恢复现金流
→ 维持公司运转
→ 重新准备自研项目
```

### 5.2 外包的主要投入

- 员工时间；
- 员工体力；
- 机会成本。

### 5.3 外包的主要产出

- 较稳定现金；
- 研究点；
- 少量员工实践；
- 过渡时间价值。

### 5.4 外包不提供的主要价值

通常不提供或弱化：

- 大规模销量；
- 粉丝复利；
- 项目品牌；
- 评论仪式；
- 奖项；
- 系列价值；
- 平台影响。

### 5.5 外包的系统职责

外包承担的是：

```text
安全循环
```

主要用于：

- 早期生存；
- 失败恢复；
- 等待员工恢复；
- 等待平台时机；
- 缓解工资压力；
- 在高风险项目之间过渡。

### 5.6 外包为何不会取代核心循环

如果外包收益过高，它会成为最优解，玩家会避免自研游戏。

因此外包通常具有：

- 收益上限低；
- 情绪结算弱；
- 长期复利弱；
- 品牌价值弱。

它帮助玩家生存，但不能帮助玩家快速成为行业巨头。

---

## 6. 自研游戏循环

### 6.1 自研循环定义

```text
选择平台、类型和题材
→ 投入资金与人员
→ 完成多个开发阶段
→ 累积作品属性与 Bug
→ Debug
→ 发行
→ 获得评论
→ 获得销量、收入、粉丝和经验
→ 将回报再投资
```

### 6.2 自研循环的主要投入

- 项目成本；
- 平台授权；
- 员工体力；
- 员工能力；
- 外包费用；
- 研究点；
- 宣传费用；
- 时间。

### 6.3 自研循环的主要产出

- 作品；
- 评价；
- 销量；
- 收入；
- 粉丝；
- 研究点；
- 类型经验；
- 题材经验；
- 员工成长；
- 奖项；
- 续作机会；
- 公司记录。

### 6.4 自研循环的角色

自研是：

```text
成长循环
```

只有通过自研作品，玩家才能获得真正的长期复利。

### 6.5 核心风险

- 组合选择不佳；
- 员工能力不足；
- 项目成本过高；
- 平台市场不足；
- 宣传浪费；
- 评论偏低；
- 销量不足；
- 现金回收慢。

---

## 7. 外包与自研的循环关系

```text
外包循环
   │
   ├─ 提供安全现金
   ├─ 提供研究点
   └─ 降低失败后无法继续的风险
          │
          v
自研循环
   │
   ├─ 提供高收益
   ├─ 提供粉丝和品牌
   ├─ 提供奖项和记录
   └─ 推动公司长期成长
```

### 7.1 两者的设计互补

外包保证：

```text
玩家不会因为一次失败而完全失去行动能力。
```

自研保证：

```text
玩家不能只靠安全行为获得全部长期成长。
```

### 7.2 风险梯度

```text
外包
→ 低成本自研
→ 成熟组合项目
→ 新组合项目
→ 高宣传项目
→ 新平台项目
→ 大型项目
→ 自研主机
```

这形成逐步升级的风险阶梯。

## 8. 项目立项循环

项目立项是每次自研循环的决策密度最高点。

### 8.1 平台选择

玩家判断：

- 当前用户量；
- 平台生命周期；
- 授权费用；
- 开发成本；
- 公司现金；
- 项目方向；
- 是否值得长期投入。

### 8.2 类型选择

影响：

- 作品定位；
- 组合关系；
- 公司专精；
- 经验成长；
- 潜在评价。

### 8.3 题材选择

影响：

- 组合评价；
- 市场接受；
- 创意表达；
- 作品身份。

### 8.4 方针选择

决定作品更偏向：

- 趣味；
- 创意；
- 画面；
- 音乐；
- 其他方向。

### 8.5 项目命名

命名主要提供：

- 归属感；
- 系列感；
- 记忆；
- 自我叙事。

### 8.6 立项循环的输入输出

输入：

- 公司状态；
- 平台状态；
- 员工状态；
- 已知组合；
- 当前现金；
- 长期目标。

输出：

- 项目定义；
- 初始成本；
- 风险水平；
- 预期市场；
- 后续人员需求。

## 9. 项目生产循环

### 9.1 项目生产总流程

```text
阶段启动
→ 选择负责人
→ 负责人产生基础贡献
→ 全体员工持续工作
→ 属性增长
→ 随机事件
→ Bug 增长
→ 阶段完成
→ 下一阶段
```

### 9.2 自动生产

大部分开发行为自动执行。

玩家不需要逐项安排：

- 每个员工每天做什么；
- 每个工时分配；
- 每项具体任务；
- 每个 Bug 的归属。

### 9.3 玩家参与

玩家仍会：

- 选择阶段负责人；
- 选择外包；
- 接受或拒绝挑战；
- 使用研究点；
- 使用道具；
- 观察员工状态；
- 等待恢复；
- 调整长期计划。

### 9.4 自动化的价值

自动化保留了：

```text
制作过程感
```

同时避免：

```text
项目管理微操。
```

## 10. 员工行动循环

### 10.1 基本行为

员工在办公桌前自动行动。

一次行为通常产生：

- 某项作品属性；
- 体力消耗；
- 视觉特效；
- 数值跳字；
- 有时产生 Bug 或特殊状态。

### 10.2 行动反馈

员工行动通过：

- 动画；
- 点数；
- 音效；
- 表情；
- 特殊特效；

让玩家看见其贡献。

### 10.3 体力循环

```text
工作
→ 体力下降
→ 效率或可用性下降
→ 休息
→ 恢复
→ 再工作
```

### 10.4 体力的设计作用

- 防止最强员工无限重复承担所有工作；
- 制造项目内节奏；
- 提供人员轮换价值；
- 使时间和恢复具有意义。

### 10.5 行动循环的局限

员工行动大多由属性决定，玩家对过程干预较少。

因此中后期容易变成：

```text
观察数值增长
```

而不是：

```text
持续做出新决策。
```

## 11. 开发阶段循环

项目通常被切分为若干主题阶段。

虽然不同版本的具体标签可能略有差异，但设计上通常包含：

- 企划或剧本；
- 主体开发；
- 画面；
- 音乐；
- Debug。

### 11.1 阶段差异

不同阶段调用不同员工能力。

这使员工价值不是单一总分。

### 11.2 阶段负责人选择

玩家可以：

- 使用内部员工；
- 使用外部专家；
- 在成本与能力之间权衡。

### 11.3 阶段循环作用

每个阶段都提供一次小型决策和小型结算。

这避免整个项目只有开头和结尾两个决策点。

### 11.4 阶段循环图

```text
阶段开始
→ 选择负责人
→ 产生基础属性
→ 全员持续开发
→ 阶段完成
→ 进入下一阶段
```

---

## 12. 内部制作循环

### 12.1 内部制作投入

- 员工体力；
- 员工机会成本；
- 当前能力。

### 12.2 内部制作产出

- 项目贡献；
- 员工经验；
- 公司内部能力利用。

### 12.3 长期价值

内部制作会强化：

```text
培养员工
→ 员工承担更多阶段
→ 减少外包依赖
→ 项目成本下降
→ 公司能力提高
```

### 12.4 风险

如果内部员工状态差，可能：

- 属性增长有限；
- 体力不足；
- 影响后续主体开发。

---

## 13. 外部专家循环

### 13.1 外部专家投入

- 一次性资金；
- 机会选择；
- 有时存在重复使用限制或效率变化。

### 13.2 外部专家产出

- 较高或较稳定的阶段贡献；
- 弥补团队短板；
- 降低当前项目风险。

### 13.3 外部专家的系统作用

```text
资金
→ 临时能力
```

它让资金不只用于扩张，也可以用于即时质量。

### 13.4 内部与外部的张力

```text
培养能力
vs
购买能力
```

这是项目循环中的重要选择。

---

## 14. 随机事件循环

### 14.1 事件触发

开发中员工可能提出挑战或进入特殊状态。

### 14.2 玩家选择

常见结构是：

```text
是否投入研究点或其他资源，
让员工尝试产生高额贡献。
```

### 14.3 成功

可能获得：

- 大量属性；
- 项目突破；
- 情绪高潮。

### 14.4 失败

可能产生：

- Bug；
- 资源损失；
- 体力压力；
- 项目风险。

### 14.5 设计意义

这类事件将原本自动的过程转化为：

```text
短暂的风险决策节点。
```

### 14.6 风险接受循环

```text
出现机会
→ 判断员工与资源
→ 接受或拒绝
→ 成功或失败
→ 更新项目预期
```

## 15. Bug 循环

### 15.1 Bug 产生

Bug 来自：

- 普通开发；
- 挑战失败；
- 特殊事件；
- 项目规模。

### 15.2 Bug 的负面作用

- 延长项目；
- 增加收尾压力；
- 消耗员工；
- 影响发行时机。

### 15.3 Debug

```text
开发完成
→ 进入 Debug
→ 员工逐步消除 Bug
→ 可能获得研究点
→ Bug 清零或达到发行条件
```

### 15.4 Bug 的主题作用

Bug 让开发流程更像真实软件项目。

### 15.5 Bug 的循环作用

Bug 防止：

```text
属性达成
→ 立即发行
```

而是增加最后一个收束阶段。

### 15.6 失败转化

Bug 在 Debug 中可以转化为研究价值。

这是：

```text
负面状态
→ 处理成本
→ 次级收益
```

的循环。

## 16. 项目完成循环

### 16.1 完成时的主要结果

- 最终属性；
- 最终 Bug；
- 项目成本；
- 项目时间；
- 作品身份；
- 组合状态；
- 平台状态。

### 16.2 完成不是最终结算

项目完成后还有：

- 评论；
- 销量；
- 排名；
- 奖项；
- 粉丝；
- 经验。

### 16.3 完成节点的情绪作用

开发完成提供：

- 生产释放；
- 结果期待；
- 对评论的过渡。

---

## 17. 评论循环

### 17.1 评论循环结构

```text
作品完成
→ 第一位评论者评分
→ 第二位评论者评分
→ 第三位评论者评分
→ 第四位评论者评分
→ 总体结果
```

### 17.2 输入

评论大致基于：

- 作品属性；
- 组合；
- 类型和题材水平；
- 项目质量；
- 其他隐藏修正。

### 17.3 输出

- 单项分数；
- 总体评价；
- 市场预期；
- 玩家情绪。

### 17.4 评论的循环职责

评论是：

```text
质量结算
```

它回答：

```text
这款作品做得好不好？
```

### 17.5 为什么不直接显示销售结果

评论在销量前出现，建立因果直觉：

```text
高质量
→ 更可能高销量
```

即使实际销量还受平台和市场影响。

## 18. 销量循环

### 18.1 销量循环结构

```text
作品发行
→ 首周销量
→ 周期销量
→ 排名变化
→ 收入回收
→ 销量衰减
→ 停止销售
```

### 18.2 销量输入

- 作品质量；
- 评论；
- 平台用户；
- 粉丝；
- 宣传；
- 组合；
- 公司品牌；
- 市场状态。

### 18.3 销量输出

- 现金；
- 粉丝；
- 记录；
- 排名；
- 成功里程碑；
- 公司发展机会。

### 18.4 销量的循环职责

销量是：

```text
商业结算
```

它回答：

```text
市场是否接受这款作品？
```

### 18.5 销量衰减

销量逐周下降，形成：

- 产品生命周期；
- 下一项目压力；
- 持续收入但非永久收入；
- 结算尾声。

---

## 19. 双重结算循环

《游戏发展国》的一个项目不是一次结算，而是两次。

### 19.1 第一层：质量结算

```text
开发属性
→ 评论
```

### 19.2 第二层：商业结算

```text
评论 + 平台 + 粉丝 + 宣传
→ 销量
```

### 19.3 双重结算的价值

- 延长情绪；
- 分离“作品好”和“卖得好”；
- 让市场系统具有独立作用；
- 让高评价后仍有期待。

### 19.4 第三层：长期结算

```text
销量与评价
→ 粉丝、经验、奖项、续作、公司记录
```

因此更准确地说，是：

```text
质量结算
→ 商业结算
→ 长期成长结算
```

## 21. 再投资循环

项目完成后的关键问题是：

```text
把回报投入哪里？
```

### 21.1 投入员工

- 升级；
- 训练；
- 转职；
- 招聘；
- 替换。

### 21.2 投入平台

- 新授权；
- 新主机；
- 平台迁移。

### 21.3 投入公司

- 扩大办公室；
- 增加员工容量。

### 21.4 投入下一项目

- 更高成本；
- 更强外包；
- 更多宣传；
- 新组合；
- 新平台。

### 21.5 保留现金

- 支付工资；
- 应对失败；
- 等待机会；
- 准备大型投资。

### 21.6 再投资循环的设计价值

项目结果不会停留为被动奖励。

它立刻成为新的决策资源。

## 22. 公司成长循环

```text
作品成功
→ 资金增加
→ 招聘和培训
→ 生产能力增加
→ 项目质量提高
→ 作品更成功
```

### 22.1 员工数量循环

```text
扩大办公室
→ 增加员工容量
→ 增加总产出
→ 支持更大项目
→ 获得更多收入
```

### 22.2 固定成本约束

办公室和员工增加后，工资也增加。

因此成长循环中仍存在：

```text
能力增加
+
固定支出增加
```

### 22.3 中后期问题

当收入增长远超工资后，固定成本失去约束。

## 23. 知识循环

### 23.1 玩家知识循环

```text
选择组合
→ 观察评价和销量
→ 形成经验
→ 下一次做出更好选择
```

### 23.2 游戏内知识循环

```text
重复开发
→ 类型和题材等级提高
→ 下一次作品潜力提高
```

### 23.3 双重知识循环

```text
玩家更懂系统
+
公司数值更懂该类型
```

### 23.4 知识循环的衰减

当玩家掌握全部最佳组合后：

- 试验减少；
- 失败减少；
- 选择趋同；
- 攻略替代探索。

## 24. 平台循环

### 24.1 外部平台循环

```text
平台公布
→ 玩家评估
→ 购买授权
→ 开发作品
→ 用户增长
→ 平台成熟
→ 平台衰退
→ 转移新平台
```

### 24.2 平台循环的作用

它在核心项目循环之外加入：

```text
时代压力。
```

### 24.3 决策变化

同一个组合和团队，在不同平台阶段可能有不同价值。

### 24.4 自研主机循环

```text
积累资金与人才
→ 研发自有主机
→ 发布主机
→ 为自家平台开发
→ 获得平台级影响
```

### 24.5 终局变化

自研主机将平台从外部变量变成公司资产。

## 25. 循环嵌套关系

```text
员工行动循环
    ↓
开发阶段循环
    ↓
项目循环
    ↓
公司成长循环
    ↓
行业时代循环
```

同时，结果会反向影响更低层循环：

```text
行业变化
→ 改变项目选择
→ 改变阶段人员需求
→ 改变员工使用
```

### 25.1 嵌套的价值

玩家每个时刻都有：

- 一个立即可观察的状态；
- 一个正在推进的项目；
- 一个公司成长目标；
- 一个行业长期方向。

## 26. 项目并行与节奏

### 26.1 非严格并行

玩家通常不能像真实工作室一样同时开发大量作品。

但游戏通过以下方式制造重叠：

```text
上一款作品仍在销售
+
下一款作品已经开发
```

### 26.2 重叠循环

```text
Project A：销售
Project B：开发
Company：员工培养
Industry：平台变化
```

多个时间层同时运行。

### 26.3 节奏价值

- 减少等待；
- 让旧项目继续产生价值；
- 让新项目立即有资金预期；
- 保持公司持续运转感。

### 26.4 销量结束信号

当旧作品销量下降并停止时，玩家通常已进入后续项目。

因此不会出现明显空档。

- 员工培养；

## 29. 循环中的资源压力

### 29.1 短期压力

- 当前现金；
- 当前体力；
- 当前 Bug；
- 当前项目成本。

### 29.2 中期压力

- 工资；
- 招聘；
- 培训；
- 平台授权；
- 宣传。

### 29.3 长期压力

- 办公室扩张；
- 高阶员工；
- 自研主机；
- 新平台迁移。

### 29.4 资源压力的变化

早期：

```text
现金是主要瓶颈。
```

中期：

```text
员工和解锁是主要瓶颈。
```

后期：

```text
目标与挑战本身成为主要瓶颈。
```

## 30. 循环中的风险结构

### 30.1 可预见风险

- 现金不足；
- 工资过高；
- 授权太贵；
- 平台用户太少；
- 员工太弱；
- 组合评价差。

### 30.2 半可预见风险

- 项目属性不足；
- 员工体力不足；
- 宣传回报不高；
- 销量不及预期。

### 30.3 不可完全控制风险

- 特殊事件；
- 评论波动；
- 市场波动；
- 员工爆发或失败。

### 30.4 风险缓解

- 使用成熟组合；
- 保留现金；
- 选择成熟平台；
- 培养员工；
- 外包关键阶段；
- 减少宣传；
- 做外包恢复。

---

## 31. 失败循环

### 31.1 项目失败循环

```text
项目评价低
→ 销量低
→ 现金增长有限
→ 扩张推迟
→ 回到保守项目或外包
→ 恢复资源
→ 再次尝试
```

### 31.2 失败不会阻断主要原因

- 外包提供收入；
- 旧员工仍保留；
- 研究与经验可能保留；
- 组合知识仍保留；
- 公司不会因单作失败立即清零。

### 31.3 失败仍然有信息价值

失败帮助玩家判断：

- 组合是否不佳；
- 平台是否不合适；
- 团队是否太弱；
- 投入是否不足；
- 时机是否错误。

### 31.4 恢复循环

```text
失败
→ 降低风险
→ 做外包或低成本项目
→ 积累资金
→ 重新进入成长循环
```

---

## 32. 成功循环

### 32.1 项目成功

```text
高属性
→ 高评论
→ 高销量
→ 高收入与粉丝
```

### 32.2 成功后的能力提升

- 更强招聘；
- 更多训练；
- 新平台授权；
- 更高宣传；
- 办公室扩张；
- 主机准备。

### 32.3 成功循环的加速

成功会提高下一次成功的概率。

### 32.4 滚雪球

```text
成功
→ 更多资源
→ 更强项目
→ 更大成功
```

### 32.5 设计风险

后期滚雪球可能使：

- 风险失效；
- 项目结果可预测；
- 外包循环失去意义；
- 现金选择失去权重。

---

## 33. 核心循环驱动因素

### 33.1 “再做一款”

最直接的驱动力。

每款作品都有短周期结算，因此下一轮启动成本低。

### 33.2 “再高一点”

玩家希望提高：

- 属性；
- 评论；
- 销量；
- 排名；
- 粉丝；
- 公司记录。

### 33.3 “再解锁一个”

玩家希望获得：

- 新类型；
- 新题材；
- 新职业；
- 新员工；
- 新平台；
- 新办公室。

### 33.4 “再培养一次”

员工成长需要多个项目。

### 33.5 “再等一个时代节点”

例如：

- 新平台；
- 奖项；
- 扩张；
- 主机开发。

## 36. 循环的清晰度

### 36.1 状态清楚

玩家知道当前处于：

- 立项；
- 开发；
- 某阶段；
- Debug；
- 评论；
- 销售；
- 培养；
- 等待。

### 36.2 输入清楚

每个阶段需要的决策有限。

### 36.3 输出清楚

项目最终产出：

- 评分；
- 销量；
- 收入；

## 37. 循环中的等待

### 37.1 必要等待

- 员工工作；
- 体力恢复；
- 销量推进；
- 平台成长；
- 工资周期。

### 37.2 等待中的可观察内容

- 动画；
- 数值；
- 事件；
- 销量；
- 新闻；
- 其他系统变化。

### 37.3 等待不是空白

游戏利用多层状态重叠填充等待。

### 37.4 潜在风险

当玩家已掌握系统后，开发观察可能变为纯等待。

## 38. 核心循环与内容循环

### 38.1 内容作为循环变体

新类型、题材和平台不会改变主循环结构。

它们改变：

- 输入；
- 风险；
-预期；
- 结果；
- 主题表达。

### 38.2 内容扩展方式

```text
同一循环
+
新组合
+
新平台
+
新员工
+
新目标
```

### 38.3 优点

- 易于扩展；
- 玩家无需重学；
- 生产成本低。

### 38.4 风险

如果新内容只改变数值，不改变决策，可能产生重复感。

---

## 39. 核心循环与成长循环

### 39.1 循环互相依赖

没有成长：

```text
核心循环会失去长期意义。
```

没有项目：

```text
成长系统没有资源来源和使用场景。
```

### 39.2 作品是连接点

作品把：

- 员工；
- 资金；
- 平台；
- 组合；
- 市场；
- 评价；
- 成长；

连接起来。

### 39.3 成长如何改变循环

成长提高：

- 项目规模；
- 成功概率；
- 内容选择；
- 投资上限；
- 风险承受；
- 身份层级。

---

## 40. 核心循环与时代循环

### 40.1 静态循环问题

如果平台不变化，玩家可能永久使用：

- 同一平台；
- 同一组合；
- 同一团队；
- 同一宣传方案。

### 40.2 时代循环干预

新平台通过：

- 新授权；
- 新用户曲线；
- 新成本；
- 新时机；

迫使项目循环变化。

### 40.3 自研主机后的变化

平台时代压力部分被玩家控制。

因此终局需要其他外部压力，否则策略会趋于稳定。

---

## 41. 中后期循环变化

### 41.1 早期循环

```text
资源紧张
→ 选择谨慎
→ 项目结果重要
→ 失败需要恢复
```

### 41.2 中期循环

```text
资源增长
→ 培养与平台选择增加
→ 项目策略更丰富
→ 成功开始复利
```

### 41.3 后期循环

```text
资源充足
→ 最优组合和员工稳定
→ 项目结果可预测
→ 循环转向记录优化
```

### 41.4 循环衰减原因

- 现金约束失效；
- 员工能力过强；
- 组合答案固定；
- 平台判断简化；
- 评论悬念下降；
- 外包恢复不再需要；
- 新决策层级不足。

---

## 42. 核心循环的主要优点

### 42.1 完整

每次项目都有开始、过程和结算。

### 42.2 紧凑

循环时间短。

### 42.3 易懂

项目主题与现实认知一致。

### 42.4 高反馈

每个阶段都有视觉和数值反馈。

### 42.5 可恢复

失败不会轻易终止游戏。

### 42.6 可扩展

新内容可以复用主循环。

### 42.7 强复利

结果会显著改变下一循环。

### 42.8 多层重叠

员工、项目、公司和行业同时推进。

---

## 43. 核心循环的主要取舍

### 43.1 自动化换取轻量性

优点：

- 操作负担低；
- 节奏快；
- 易上手。

代价：

- 中后期观察感变强；
- 项目过程策略有限。

### 43.2 固定项目结构换取稳定体验

优点：

- 清楚；
- 可预期；
- 易反馈。

代价：

- 长期重复。

### 43.3 强复利换取成长爽感

优点：

- 成长明显；
- 成功有价值。

代价：

- 后期风险下降。

### 43.4 半隐藏规则换取发现感

优点：

- 有学习；
- 有试验。

代价：

- 容易查表化。

---

## 44. 可复用核心循环模式

### 44.1 Safe Loop + Growth Loop

```text
低风险工作
```

## 47. 核心循环结论

### 47.1 结论一：项目是最小完整体验单位

员工行动很短，员工成长很长。

只有“制作一款游戏”同时拥有：

- 选择；
- 过程；
- 风险；
- 结算；
- 成长。

### 47.2 结论二：外包负责保底，自研负责成长

外包让玩家不容易被失败锁死。

自研让玩家愿意承担风险。

### 47.3 结论三：评论与销量构成双重高潮

评论回答：

```text
做得好不好。
```

销量回答：

```text
卖得好不好。
```

### 47.4 结论四：真正的奖励是改变下一循环

现金、粉丝、研究、员工和解锁都会改变下一项目的上限。

### 47.5 结论五：多层循环重叠避免等待

员工工作、作品销售、公司成长和平台时代同时推进。

### 47.6 结论六：失败被设计为减速

玩家可以退回安全循环，重新积累。

### 47.7 结论七：后期问题来自循环约束失效

当资金、员工和知识足够强后：

- 风险下降；
- 选择收敛；
- 结算悬念下降；
- 循环从探索转向优化。

## 49. 总结

《游戏发展国》的核心循环可以最终概括为：

```text
安全循环保证继续行动，
项目循环制造选择和悬念，
市场结算提供情绪回报，
成长循环让结果进入下一次生产，
行业循环迫使成熟策略继续变化。
```

其中最关键的结构是：

```text
立项
→ 自动生产
→ 风险波动
→ Debug
→ 评论
→ 销量
→ 多轨成长
→ 再投资
```

它用非常少的操作，持续让玩家经历：

```text
我做了一个决定
→ 我的团队正在执行
→ 结果还不确定
→ 市场给出了判决
→ 公司因此改变
→ 我现在可以做得更大
```

这就是《游戏发展国》能够持续驱动“再做一款”的核心原因。
