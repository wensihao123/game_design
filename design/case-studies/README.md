# Case Studies（案例分析）

> Status: V1  
> Path: `design/case-studies/README.md`  
> Scope: 对现有游戏的拆解与学习笔记  
> Purpose: 定义案例分析层的定位、方法、目录组织与登记方式，把 Philosophy 与 Systems 中的抽象理念放到真实商业游戏上做验证与提炼。

---

## 1. 这一层是什么

Case Studies 层负责回答：

```text
一款已经成立的商业游戏，
是如何把某种体验目标落成一套可运转的系统的？
它做对了什么、付出了什么代价、有哪些可迁移的模式？
```

它不是攻略，也不追求反编译内部公式，而是：

- 验证 Philosophy / Systems 文档能否用于分析真实游戏；
- 拆解核心体验、循环结构、系统耦合与经济模型；
- 提炼可复用（与不应照搬）的设计模式，回流到通用文档。

---

## 2. 分析方法

案例统一采用**可观察设计分析（Observable Design Analysis）**，不依赖源码，主要依据游戏内可见规则、玩家操作流程、官方说明与可重复验证的系统行为。

为区分事实与推断，证据分四级：

| 等级 | 含义 |
|---|---|
| Level A | 直接可观察事实 |
| Level B | 稳定社区知识（可在游戏中重复验证） |
| Level C | 设计推断（基于系统结构推导作用） |
| Level D | 待验证假设（无法仅凭公开信息确认） |

社区攻略与 Wiki 只作稳定机制与解锁信息的辅助来源，不直接当作官方设计意图。

## 4. 目录组织约定

- 每个案例独立一个子目录（kebab-case slug，如 `game-dev-story/`）。
- 子目录内 `README.md` 为落地页：写清分析目的、边界、证据等级、系统清单、阅读顺序与完成状态。
- 案例是**顺序通读**的，内容文档用 `01-`、`02-` 数字前缀排列，使目录顺序直接对齐阅读顺序；`README.md` 保留不加前缀（见根 README「约定」中的例外说明）。
- 文档头部统一带元信息块（标题、开发商、Analysis Type、Status、Path、Last Updated 等）。

---

## 5. 与其他层的关系

```text
Philosophy   为什么这样设计
Systems      产品如何运转
Case Studies 真实游戏如何做到 —— 验证上两层，并反哺可复用模式
```

案例中提炼的模式若具备通用价值，应回写到 Systems 或 Philosophy，而不是停留在单个案例里。
