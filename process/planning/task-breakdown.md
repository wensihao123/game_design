# Task Breakdown（任务拆分）

> 状态：草案  
> 适用范围：通用的软件与游戏开发项目  
> 目的：定义如何把一个功能、需求、缺陷或技术目标拆分为可执行、可验证、可追踪的任务。

---

## 1. 文档目的

Task Breakdown 用于把一个较大的工作项，转化为可以被实际执行和管理的任务集合。

它回答以下问题：

- 一个 Feature 应如何拆成任务；
- 一个任务多大才合适；
- 哪些内容应独立成 Task；
- 哪些内容不应拆得过细；
- 如何识别任务依赖；
- 如何区分设计任务、工程任务、测试任务和发布任务；
- 如何避免“任务全部完成，但功能仍然不能交付”；
- 如何保证任务拆分覆盖完整交付链路；
- 如何让任务状态能够真实反映进度。

Task Breakdown 的目标不是把工作拆成尽可能多的小步骤，而是形成一组：

> 边界清晰、输出明确、依赖可见、结果可验证、能够共同完成上层目标的任务。

---

## 2. 适用范围

任务拆分适用于：

- Feature；
- Story；
- Bug；
- Technical Work；
- Research / Spike；
- Asset Work；
- Release Work；
- Migration；
- Refactoring；
- Performance Improvement；
- Documentation Work。

不同类型的工作，应使用不同拆分方式。

---

## 3. Task Breakdown 与其他流程的关系

任务拆分位于功能生命周期中的 Plan 阶段。

```text
Ready
→ Design
→ Prototype（可选）
→ Plan
→ Build
```

它依赖前置输出：

- 问题已经明确；
- Scope 已经明确；
- Non-goals 已经明确；
- 主要行为已经明确；
- 关键依赖已经识别；
- 初步验收标准已经存在。

如果这些内容尚未明确，任务拆分通常会产生：

- 重复任务；
- 范围冲突；
- 隐藏依赖；
- 无法验证的任务；
- 实现中途大规模返工。

Task Breakdown 不替代：

- Feature Lifecycle；
- Definition of Ready；
- Definition of Done；
- Backlog Management；
- Technical Design；
- Testing Strategy。

它只负责：

> 把已经批准的工作转化为可执行计划。

---

## 4. 拆分的核心原则

### 4.1 任务应围绕结果，而不是动作

不合格任务：

```text
写代码
改 UI
做测试
处理数据
研究一下
```

合格任务：

```text
实现用户创建预设的保存逻辑
增加预设名称重复校验
补充删除预设的失败状态
验证旧版本预设数据迁移
```

任务标题应尽量表达：

> 完成后会产生什么可验证结果。

---

### 4.2 每个任务应有明确边界

一个任务应说明：

- 输入是什么；
- 输出是什么；
- 哪些内容包含；
- 哪些内容不包含；
- 依赖什么；
- 如何验证；
- 何时算完成。

任务边界不清，会导致：

- 多人重复工作；
- 功能遗漏；
- 责任不清；
- 任务无法关闭；
- Scope 不断扩张。

---

### 4.3 任务应可独立验证

一个任务完成后，应能回答：

```text
如何证明它已经完成？
```

例如：

```text
新增数据库字段
```

不如：

```text
新增 preset_name 字段，并验证创建、读取、更新和迁移均可正确使用该字段
```

---

### 4.4 拆分应覆盖完整交付链路

很多计划只拆工程实现，不拆：

- 设计；
- 数据；
- UI；
- 资产；
- 测试；
- 文档；
- 集成；
- 发布；
- 迁移；
- 监控。

结果是：

> 所有“开发任务”都完成了，但 Feature 仍然不能 Done。

任务拆分必须以 Definition of Done 为反向检查依据。

---

### 4.5 不要过度拆分

过度拆分会造成：

- 任务数量过多；
- 管理成本上升；
- 上下文切换频繁；
- 工作价值被微小步骤淹没；
- 状态更新成本高于执行成本。

不建议为以下内容单独创建 Task：

- 改一个变量名；
- 写一行日志；
- 调整一个 2px 间距；
- 修改一个局部常量；
- 完成同一目标所需的极小步骤。

这些内容可以作为：

- Checklist；
- Subtask；
- 实现说明；
- Code Review 修改项。

---

### 4.6 不要拆成无法单独产生意义的层

不合理：

```text
创建文件
创建类
添加变量
添加函数
调用函数
```

这些步骤只是同一个实现动作的内部过程。

更合理：

```text
实现可复用的预设存储服务
```

---

## 5. 任务层级

推荐使用：

```text
Feature
→ Task
→ Subtask / Checklist
```

复杂项目可以加入：

```text
Epic
→ Feature
→ Story
→ Task
→ Subtask
```

---

## 5.1 Feature

表示一个有独立用户价值或产品价值的能力。

例如：

```text
用户可以创建和管理自定义预设
```

---

## 5.2 Task

表示完成 Feature 所需的可执行工作单元。

例如：

```text
设计预设管理数据模型
实现预设创建接口
实现预设列表 UI
补充预设删除回归测试
```

---

## 5.3 Subtask

用于进一步拆分复杂 Task。

例如：

```text
Task：实现预设创建接口

Subtasks：
- 定义请求模型
- 增加名称校验
- 保存数据库记录
- 返回错误码
- 补充单元测试
```

Subtask 不应替代 Task 的目标和验收标准。

---

## 5.4 Checklist

用于不会独立管理的小步骤。

例如：

```markdown
- [ ] 正常输入
- [ ] 空名称
- [ ] 名称过长
- [ ] 重复名称
- [ ] 数据库失败
```

---

## 6. 任务类型

一个完整 Feature 通常包含多种任务类型。

---

## 6.1 Product / Design Task

用于明确：

- 业务规则；
- 玩法规则；
- 用户流程；
- 状态；
- 边界条件；
- 文案；
- 验收标准。

示例：

```text
定义预设重命名规则与冲突处理
设计删除确认流程
补充空状态和失败状态
```

---

## 6.2 UX / UI Task

用于：

- 页面结构；
- 交互流程；
- 原型；
- 状态设计；
- 响应式布局；
- 可访问性；
- 错误提示；
- 焦点和输入方式。

示例：

```text
设计预设列表的 Loading、Empty 和 Error 状态
实现预设重命名弹窗
验证键盘导航路径
```

---

## 6.3 Engineering Task

用于：

- 业务逻辑；
- 数据结构；
- API；
- 状态管理；
- 持久化；
- 集成；
- 性能；
- 安全；
- 构建。

示例：

```text
实现预设创建与更新接口
增加预设缓存失效逻辑
实现客户端预设状态同步
```

---

## 6.4 Data / Migration Task

用于：

- 数据库 Schema；
- 本地存档；
- 数据迁移；
- 数据修复；
- 兼容；
- 回滚。

示例：

```text
为旧预设数据增加默认名称
实现 v2 到 v3 的迁移脚本
验证迁移失败后的回滚
```

---

## 6.5 Asset Task

用于：

- 图标；
- 动画；
- 音效；
- 图片；
- 视频；
- 模型；
- 文案资源；
- 本地化资产。

示例：

```text
制作预设空状态插图
补充创建成功音效
输出正式尺寸图标资源
```

---

## 6.6 Test Task

用于：

- 单元测试；
- 集成测试；
- 端到端测试；
- 回归；
- 性能；
- 兼容；
- 正式构建验证。

示例：

```text
补充预设名称校验单元测试
验证旧版本数据迁移
完成正式构建回归测试
```

---

## 6.7 Documentation Task

用于更新：

- 功能设计；
- 技术设计；
- API；
- 数据结构；
- 使用说明；
- 迁移说明；
- 发布说明；
- 运维文档。

---

## 6.8 Release / Operations Task

用于：

- Feature Flag；
- 灰度；
- 监控；
- 日志；
- 发布配置；
- 回滚；
- 数据修复；
- 发布后验证。

---

## 6.9 Research / Spike Task

用于解决未知问题。

必须定义：

- 研究问题；
- 时间限制；
- 验证方法；
- 输出；
- 结束条件。

---

## 7. 一个好的任务应具备什么

建议每个 Task 至少具备以下属性：

```text
Clear
Bounded
Testable
Owned
Sized
Traceable
Independent Enough
```

对应：

```text
清晰
有边界
可验证
有负责人
规模适中
可追踪
足够独立
```

---

## 7.1 Clear：清晰

任务标题和描述能够让执行者理解：

- 为什么做；
- 要产生什么结果；
- 影响哪里。

---

## 7.2 Bounded：有边界

任务明确：

- 包含什么；
- 不包含什么；
- 不会无限扩张。

---

## 7.3 Testable：可验证

完成标准可以通过：

- 测试；
- Demo；
- 评审；
- 构建；
- 输出文件；
- 日志；
- 数据结果；

进行确认。

---

## 7.4 Owned：有负责人

每个活跃任务应有明确 Owner。

Owner 负责：

- 推进；
- 更新状态；
- 暴露阻塞；
- 确保输出；
- 发起评审；
- 完成关闭。

---

## 7.5 Sized：规模适中

任务不应：

- 大到无法估算；
- 大到跨越多个独立目标；
- 大到长时间没有中间结果；
- 小到管理成本高于执行成本。

---

## 7.6 Traceable：可追踪

任务应能关联到：

- Feature；
- 验收标准；
- Design；
- Bug；
- Milestone；
- Release。

---

## 7.7 Independent Enough：足够独立

任务不必完全独立，但应尽量减少无必要耦合。

如果两个任务必须一起完成，可能需要：

- 合并；
- 重新拆分；
- 明确依赖顺序；
- 定义共享接口。

---

## 8. 拆分维度

任务可以按不同维度拆分。

---

## 8.1 按用户流程拆分

例如：

```text
创建
查看
编辑
删除
分享
```

适合用户流程清晰的 Feature。

---

## 8.2 按行为状态拆分

例如：

```text
正常状态
Loading
Empty
Error
Disabled
Recovery
```

适合 UI 和异步流程。

---

## 8.3 按系统层拆分

例如：

```text
数据层
服务层
业务层
表现层
集成层
测试层
```

适合架构清晰的工程功能。

注意：只按技术层拆分，可能导致每个任务无法独立产生用户价值。

---

## 8.4 按风险拆分

例如：

```text
先验证性能风险
再验证数据迁移
最后完成常规实现
```

适合高风险 Feature。

---

## 8.5 按平台拆分

例如：

```text
Web
iOS
Android
Desktop
Console
```

适合平台差异较大的工作。

---

## 8.6 按数据范围拆分

例如：

```text
单用户
团队
组织
全局管理员
```

---

## 8.7 按发布阶段拆分

例如：

```text
Internal
Alpha
Beta
General Availability
```

---

## 8.8 按内容批次拆分

例如：

```text
首批 10 个模板
第二批 20 个模板
本地化批次
高级内容包
```

---

## 8.9 按依赖顺序拆分

例如：

```text
定义接口
实现服务端
实现客户端
完成集成
补充测试
```

---

## 9. Vertical Slice 与 Horizontal Slice

---

## 9.1 Horizontal Slice

按技术层拆分：

```text
完成全部后端
完成全部前端
完成全部测试
```

优点：

- 专业分工清楚；
- 同类工作集中。

缺点：

- 长时间没有完整结果；
- 集成风险延后；
- 很难提前验证真实价值；
- 容易在最后发现不匹配。

---

## 9.2 Vertical Slice

按完整价值切片：

```text
先完成一个最小但完整的创建流程
再完成编辑流程
再完成删除流程
```

每个 Slice 尽量包含：

- 必要 UI；
- 必要逻辑；
- 必要数据；
- 必要测试；
- 必要集成。

优点：

- 更早获得可运行结果；
- 更早暴露集成问题；
- 更容易调整；
- 更容易验证价值。

---

## 9.3 推荐原则

优先使用 Vertical Slice。

只有在以下情况才优先使用 Horizontal Slice：

- 基础设施必须先完成；
- 多个 Feature 共用同一底层；
- 专业边界非常明确；
- 依赖关系无法避免；
- 平台或团队结构要求分层交付。

---

## 10. 如何判断任务是否过大

以下信号表示任务可能过大：

- 无法在一个合理周期内完成；
- 包含多个独立用户价值；
- 验收标准超过一个清晰主题；
- 需要多个完全不同专业角色；
- 涉及多个不相关系统；
- 估算范围非常大；
- 无法在中途展示结果；
- 长时间保持 In Progress；
- 出现大量 Subtask；
- 无法清楚描述 Done。

---

## 10.1 过大任务示例

```text
实现用户管理系统
完成全部战斗系统
重构整个数据层
完成移动端适配
```

---

## 10.2 更合理拆分

```text
实现用户注册流程
实现密码重置流程
实现用户权限校验
实现管理员用户列表
```

---

## 11. 如何判断任务是否过小

以下信号表示任务可能过小：

- 执行时间远小于管理时间；
- 无法独立评审；
- 完成后没有独立输出；
- 只是同一实现的机械步骤；
- 需要频繁切换状态；
- 大量任务只能同时完成；
- 任务标题只是一个函数或变量。

---

## 11.1 过小任务示例

```text
创建类
添加字段
写 getter
写 setter
调用接口
```

---

## 11.2 合并后的任务

```text
实现用户配置模型及其读取、更新接口
```

---

## 12. 任务规模建议

不建议使用绝对时间作为唯一标准，但可以参考：

### 小型 Task

- 半天以内；
- 输出清楚；
- 适合独立执行；
- 不需要进一步拆分。

### 标准 Task

- 半天到两天；
- 有明确结果；
- 有少量依赖；
- 可以独立评审和验证。

### 大型 Task

- 超过两到三天；
- 风险较高；
- 通常应检查是否需要拆分。

具体阈值应根据：

- 团队经验；
- 项目复杂度；
- 技术栈；
- 交付节奏；
- 工作类型；

进行调整。

---

## 13. 任务依赖

每个 Task 应识别依赖。

常见关系：

```text
Blocks
Blocked By
Depends On
Can Run In Parallel
Related To
Duplicates
Supersedes
```

---

## 13.1 依赖类型

### 技术依赖

例如：

```text
API 完成后才能接入前端
```

### 数据依赖

例如：

```text
Schema 确认后才能编写迁移
```

### 设计依赖

例如：

```text
交互流程确认后才能实现 UI
```

### 资产依赖

例如：

```text
正式动画资源完成后才能最终验收
```

### 决策依赖

例如：

```text
权限规则未确认前无法实现校验
```

### 外部依赖

例如：

```text
等待第三方服务开通测试环境
```

---

## 13.2 依赖记录要求

必须说明：

- 依赖是什么；
- 是否阻塞；
- Owner；
- 预计完成时间；
- 临时替代方案；
- 超时处理方式。

---

## 13.3 避免循环依赖

例如：

```text
A 依赖 B
B 依赖 C
C 依赖 A
```

出现循环依赖时，应：

- 重新定义接口；
- 引入临时 Stub；
- 调整拆分方式；
- 合并任务；
- 建立基础设施任务。

---

## 14. Task 与 Acceptance Criteria 的映射

每个重要验收标准应至少被一个 Task 覆盖。

推荐建立映射：

| 验收标准 | 对应任务 |
|---|---|
| 用户可创建预设 | 创建 API、创建 UI、集成测试 |
| 非法名称被拒绝 | 校验逻辑、错误提示、测试 |
| 重启后数据保留 | 持久化、迁移、重启测试 |

如果某条验收标准没有任何对应任务，说明计划可能遗漏。

如果某个任务无法关联任何验收标准、风险或技术必要性，应重新判断是否真的需要。

---

## 15. 反向使用 Definition of Done

任务拆分完成后，应对照 DoD 检查是否遗漏。

至少覆盖：

```text
Scope
Acceptance
Integration
Quality
Experience
Data
Release
Documentation
Operations
```

例如：

- 是否有正式构建验证任务；
- 是否有数据迁移任务；
- 是否有失败状态任务；
- 是否有日志和监控任务；
- 是否有文档任务；
- 是否有发布后验证任务。

---

## 16. 研究任务与 Spike

Research / Spike 不应无限进行。

必须定义：

```text
问题
假设
时间上限
验证方法
输出
结束条件
```

---

## 16.1 不合格 Spike

```text
研究一下缓存
看看新框架
试试能不能优化
```

---

## 16.2 合格 Spike

```text
在 2 天内验证方案 A 是否能将 10,000 条记录的加载时间控制在 500ms 内，并输出测试数据、限制和推荐方案。
```

---

## 16.3 Spike 输出

```text
可行
有条件可行
不可行
需要进一步验证
```

以及：

- 证据；
- 限制；
- 风险；
- 建议；
- 是否需要正式实现任务。

## 17. Bug 的任务拆分

Bug 不一定只需要一个“修复”任务。

复杂 Bug 可能需要：

```text
复现
定位根因
修复
补充测试
检查数据影响
修复历史数据
回归
发布
监控
```

---

## 17.1 Bug 拆分示例

```text
复现重复订单问题
定位幂等校验缺失根因
增加服务端幂等键
补充并发测试
扫描受影响订单
修复历史异常数据
完成正式环境监控
```

---

## 18. Refactoring 的任务拆分

重构任务应避免：

```text
重构整个系统
```

应拆成：

- 定义目标边界；
- 建立回归保护；
- 抽取模块；
- 迁移调用方；
- 移除旧实现；
- 性能验证；
- 文档更新。

重构任务必须说明：

- 为什么重构；
- 哪些行为不能改变；
- 如何验证无回归；
- 是否分阶段迁移；
- 如何回滚。

---

## 19. Migration 的任务拆分

迁移通常包含：

```text
目标模型
兼容层
迁移脚本
数据校验
灰度
回滚
清理旧逻辑
```

示例：

```text
定义 v3 数据模型
实现 v2 与 v3 双读
编写批量迁移脚本
验证迁移完整性
启用 v3 写入
监控错误率
移除 v2 兼容层
```

## 20. Release 的任务拆分

发布不是单一任务。

可能包括：

- 版本确认；
- 构建；
- 签名；
- 配置；
- 发布说明；
- 数据迁移；
- 回滚演练；
- 发布；
- 冒烟测试；
- 监控；
- 关闭。

## 21. 任务命名规范

任务标题建议使用：

## 26. 常见失败模式

### 26.1 只拆开发任务

表现：

- 没有测试；
- 没有集成；
- 没有文档；
- 没有迁移；
- 没有正式构建验证。

结果：

> 任务全部完成，但 Feature 仍无法 Done。

---

### 26.2 任务标题过于模糊

例如：

```text
处理后端
完成 UI
测试一下
```

改进：

- 使用动词；
- 指定对象；
- 指定结果。

---

### 26.3 按代码文件拆分

例如：

```text
修改 user_service.py
修改 user_controller.py
修改 user_repository.py
```

这种拆分关注文件，而不是价值和结果。

更合理：

```text
实现用户冻结流程并覆盖服务、接口和数据层
```

---

### 26.4 大任务长期 In Progress

原因可能是：

- Scope 太大；
- 依赖未暴露；
- 多个目标混在一起；
- 没有中间结果；
- Owner 不明确。

处理：

- 重新拆分；
- 明确依赖；
- 提取 Spike；
- 降低 Scope；
- 采用 Vertical Slice。

---

### 26.5 任务拆得过细

表现：

- 数百个微小 Task；
- 状态更新耗时；
- 无法看出功能进度；
- 大量 Task 必须一起完成。

处理：

- 合并机械步骤；
- 使用 Checklist；
- 只保留可独立验证的 Task。

---

### 26.6 没有任务依赖

表现：

- 所有任务看起来都可并行；
- 开始后不断 Blocked；
- 接口和设计尚未确定；
- 最后才发现顺序错误。

处理：

- 建立依赖图；
- 明确 Blocks / Blocked By；
- 先处理高风险依赖。

---

### 26.7 Task Done 但无法被使用

表现：

- 代码存在，但没有接入；
- UI 完成，但没有数据；
- API 完成，但客户端无法调用；
- 测试只覆盖 Mock。

处理：

- 增加 Integration Task；
- 优先 Vertical Slice；
- 用 Feature DoD 检查计划。

---

### 26.8 Spike 没有结束条件

表现：

- 研究无限延长；
- 只收集资料；
- 没有决策；
- 没有输出。

处理：

- 设时间上限；
- 定明确问题；
- 定成功标准；
- 必须输出结论。

---

### 26.9 用估算代替拆分

给大任务一个更大的估算，并不能解决：

- 风险；
- 依赖；
- 不可验证；
- Scope 不清。

估算不能替代正确拆分。

---

在开始执行前，可以进行 Task Review。

评审问题：

```text

依赖是否可控？

```

```text

```

## 29. Task 级 Definition of Done

```markdown
- [ ] 任务目标已完成；
- [ ] 验收标准通过；
- [ ] 局部测试通过；
- [ ] 无明显回归；
- [ ] 依赖方可以使用输出；
- [ ] 临时内容已清理；
- [ ] 必要评审已完成；
- [ ] 必要文档已更新；
- [ ] Completion Notes 已填写；
- [ ] Follow-up 已创建。
```

Task Done 不代表 Feature Done。

## 30. 计划变更

执行过程中可能发现：

- 新依赖；
- Scope 变化；
- 任务过大；
- 技术路线变化；
- 原任务不再需要；
- 需要新增测试；
- 需要回退设计。

此时应：

1. 更新 Feature 记录；
2. 更新任务拆分；
3. 记录原因；
4. 调整依赖；
5. 检查验收覆盖；
6. 必要时重新进行 Ready Review。

不要在执行中偷偷增加任务，却不更新上层范围。

---

## 31. 任务拆分的完成标准

一次 Task Breakdown 可以被视为完成，当：

- Feature 目标明确；
- Scope 和 Non-goals 明确；
- 主要任务已识别；
- 高风险任务已识别；
- 关键依赖已识别；
- 任务顺序明确；
- 可并行工作明确；
- 所有验收标准有对应任务；
- 已覆盖集成和测试；
- 已覆盖数据、文档和发布；
- 任务规模合理；
- 第一批任务可以开始。

---

## 32. 最终原则

Task Breakdown 的本质不是：

> 把工作切成更多卡片。

它的本质是：

> 把一个复杂目标转化为一组能够被理解、执行、验证并最终共同完成交付的工作单元。

一个好的任务拆分应做到：

- 目标不丢失；
- 价值可追踪；
- 依赖透明；
- 风险提前暴露；
- 任务规模适中；
- 结果可验证；
- 完整覆盖 Definition of Done；
- 能够尽早形成可运行、可评估的结果。
