# Testing Checklist（测试检查清单）

> 状态：草案  
> 适用范围：通用的软件与游戏开发项目  
> 目的：定义功能、系统、内容与发布在进入 Done 或 Release 前，应覆盖的通用测试维度与检查方法。

---

## 1. 文档目的

Testing Checklist 用于帮助项目在验证阶段系统性检查：

- 功能是否按预期工作；
- 失败流程是否安全；
- 边界条件是否被覆盖；
- 数据是否正确；
- 集成是否稳定；
- 性能是否达标；
- 不同平台和环境是否兼容；
- 正式构建是否与开发环境一致；
- 旧功能是否被破坏；
- 发布后是否可观察、可恢复。

本清单不是某个功能的具体测试用例集合。

它提供的是一套通用测试框架，用于避免遗漏高频风险。

每个功能仍应根据自己的：

- Scope；
- Acceptance Criteria；
- Risk；
- Workflow Track；
- Target Platform；
- Data Impact；
- Release Strategy；

补充具体测试用例。

---

## 2. 测试的核心目标

测试不只是为了证明“功能能用”。

完整测试至少需要回答：

```text
它是否正确？
它在错误情况下是否安全？
它是否与其他系统兼容？
它是否在目标环境中稳定？
它是否会破坏旧功能？
它是否能够被观察和恢复？
```

测试的目标不是证明系统永远没有问题。

测试的目标是：

> 通过足够的证据，将未知风险降低到当前发布目标可以接受的程度。

---

## 3. 测试原则

### 3.1 验证行为，而不是只验证实现

测试应围绕：

- 用户行为；
- 系统结果；
- 数据变化；
- 状态转换；
- 可观察输出；

进行设计。

不应只检查：

- 某个函数被调用；
- 某个类存在；
- 某个文件被创建；
- 某行代码被执行。

---

### 3.2 正常流程不是全部

只验证 Happy Path，通常会遗漏真正高风险的问题。

测试至少应覆盖：

```text
正常流程
失败流程
边界条件
重复操作
中断恢复
状态切换
集成影响
正式构建
```

---

### 3.3 测试深度应与风险匹配

测试投入应根据：

- 用户影响；
- 数据影响；
- 安全风险；
- 可逆性；
- 复杂度；
- 发布成本；
- 平台数量；
- 依赖数量；
- 故障恢复难度；

进行调整。

---

### 3.4 尽早测试

测试不应只发生在开发完成后。

可以在不同阶段进行：

```text
Design：验证验收标准是否可测试
Prototype：验证高风险假设
Build：单元和局部验证
Integrate：系统级验证
Validate：正式验收
Release：正式构建与发布验证
Observe：发布后监控
```

---

### 3.5 测试应可重复

测试结果应尽量做到：

- 环境明确；
- 输入明确；
- 预期明确；
- 步骤明确；
- 结果可复现；
- 证据可保存。

---

### 3.6 测试证据应可追踪

关键测试应记录：

- 测试版本；
- 构建号；
- 环境；
- 平台；
- 测试时间；
- 测试人；
- 结果；
- 缺陷链接；
- 截图、日志或报告。

---

## 4. 测试层级

推荐从以下层级考虑测试。

---

## 4.1 Unit Test：单元测试

验证最小逻辑单元。

适合：

- 纯函数；
- 计算逻辑；
- 校验逻辑；
- 状态转换；
- 数据转换；
- 规则系统；
- 序列化；
- 迁移函数。

优点：

- 快；
- 稳定；
- 容易定位问题；
- 适合自动化。

---

## 4.2 Component Test：组件测试

验证一个组件或模块在受控环境中的行为。

适合：

- UI 组件；
- 服务模块；
- 游戏组件；
- 数据访问层；
- 状态机；
- 输入组件；
- 独立业务模块。

---

## 4.3 Integration Test：集成测试

验证多个系统之间是否正确协作。

适合：

- 客户端与服务端；
- UI 与业务逻辑；
- 数据库与服务；
- 存档与加载；
- 认证与权限；
- 场景切换；
- 第三方服务；
- 构建与资源加载。

---

## 4.4 End-to-End Test：端到端测试

验证真实用户流程。

例如：

```text
登录
→ 创建内容
→ 保存
→ 退出
→ 重新进入
→ 继续使用
```

端到端测试覆盖面广，但：

- 执行成本高；
- 容易受环境影响；
- 定位问题较慢。

不应只依赖端到端测试。

---

## 4.5 Acceptance Test：验收测试

用于确认功能是否满足 Acceptance Criteria。

验收测试回答：

> 功能是否按批准定义完成？

---

## 4.6 Regression Test：回归测试

用于确认新改动没有破坏已有功能。

回归测试应优先覆盖：

- 核心流程；
- 共享模块；
- 高风险依赖；
- 历史高频缺陷；
- 数据和存档；
- 发布关键路径。

---

## 4.7 Smoke Test：冒烟测试

快速验证构建或部署是否基本可用。

通常包括：

- 能否启动；
- 能否进入主流程；
- 核心入口是否可用；
- 关键依赖是否正常；
- 是否存在立即崩溃；
- 发布配置是否正确。

---

## 5. 测试准备

在正式测试前，应确认：

```markdown
- [ ] Scope 已明确；
- [ ] Acceptance Criteria 已明确；
- [ ] 测试环境可用；
- [ ] 测试数据已准备；
- [ ] 测试版本已固定；
- [ ] 构建号已记录；
- [ ] 依赖服务可用；
- [ ] 风险项已识别；
- [ ] 测试负责人明确；
- [ ] 缺陷记录方式明确；
```

# 6. Functional Testing：功能测试

## 6.1 正常流程

验证：

- 用户可以进入功能；
- 用户可以完成主要操作；
- 系统返回正确结果；
- 状态更新正确；
- 后续流程可继续；
- 成功反馈明确。

检查：

```markdown
- [ ] 主流程可完成；
- [ ] 输入被正确处理；
- [ ] 输出符合预期；
- [ ] 状态正确更新；
- [ ] 成功反馈正确；
- [ ] 后续流程正常；
```

---

## 6.2 失败流程

验证：

- 无效输入；
- 权限不足；
- 网络失败；
- 服务异常；
- 数据缺失；
- 资源加载失败；
- 第三方服务失败；
- 用户取消；
- 操作超时。

检查：

```markdown
- [ ] 失败不会导致崩溃；
- [ ] 错误信息可理解；
- [ ] 用户知道下一步怎么做；
- [ ] 失败不会错误修改数据；
- [ ] 支持重试、取消或恢复；
- [ ] 错误状态不会永久卡住；
```

---

## 6.3 边界条件

验证：

- 最小值；
- 最大值；
- 空值；
- 超长输入；
- 特殊字符；
- 0；
- 负数；
- 极大数据量；
- 单个元素；
- 无元素；
- 临界时间；
- 状态刚好切换时。

检查：

```markdown
- [ ] 最小值正确；
- [ ] 最大值正确；
- [ ] 空值正确；
- [ ] 超出范围时行为明确；
- [ ] 特殊字符被正确处理；
- [ ] 临界状态没有异常；
```

---

## 6.4 重复操作

验证：

- 连续点击；
- 重复提交；
- 快速切换；
- 多次保存；
- 多次删除；
- 重复支付；
- 重复加载；
- 多次触发同一事件。

检查：

```markdown
- [ ] 不会产生重复记录；
- [ ] 不会重复扣费或结算；
- [ ] 不会重复发放奖励；
- [ ] 不会重复执行不可逆操作；
- [ ] 幂等行为正确；
- [ ] UI 不会因快速操作失控；
```

---

## 6.5 取消与中断

验证：

- 用户主动取消；
- 页面关闭；
- 程序退出；
- 网络断开；
- 系统休眠；
- 切换账号；
- 场景或页面切换；
- 操作进行中被打断。

检查：

```markdown
- [ ] 中断后状态一致；
- [ ] 未完成操作不会被误判为成功；
- [ ] 临时数据被正确清理；
- [ ] 可以继续或重新开始；
- [ ] 不会产生半完成数据；
```


# 21. Regression Testing：回归测试

回归范围应优先包含：

- 被直接修改的功能；
- 共享模块；
- 相邻流程；
- 公共数据结构；
- 公共接口；
- 认证；
- 支付；
- 存档；
- 发布；
- 历史高频 Bug；
- 核心主流程。

检查：

```markdown
- [ ] 直接影响范围已回归；
- [ ] 相邻功能已回归；
```

# 38. 常见失败模式

## 38.1 只测 Happy Path

结果：

- 错误状态崩溃；
- 重复操作产生脏数据；
- 失败后无法恢复；
- 正式用户遇到未验证场景。

---

## 38.2 测试太晚开始

结果：

- 验收标准不可测试；
- 架构不利于测试；
- Bug 集中爆发；
- 修复成本高。

---

## 38.3 只在开发环境测试

结果：

- 正式构建资源缺失；
- 配置不同；
- 权限不同；
- 性能不同；
- 依赖不同。

---

## 38.4 只做手工测试

结果：

- 高频回归成本高；
- 重复性差；
- 容易遗漏；
- 发布速度受限。

---

## 38.5 只做自动化测试

结果：

- 视觉和体验问题被遗漏；
- 实际用户流程未被观察；
- Flaky Test 掩盖问题；
- 测试只验证实现细节。

---

## 38.6 缺陷没有环境信息

结果：

- 无法复现；
- 无法确认版本；
- 无法定位配置问题；
- 重复调查。

---

## 38.7 测试通过但没有证据

结果：

- 无法追踪；
- 无法复盘；
- 无法确认测试版本；
- Done Review 缺少依据。

---

## 38.8 回归范围过小

结果：

- 共享模块被破坏；
- 相邻流程出错；
- 历史 Bug 重新出现。

---

## 38.9 测试范围无限扩大

结果：

- 发布长期拖延；
- 无法判断何时结束；
- 非关键增强混入测试。

改进：

- 使用 Scope；
- 定 Exit Criteria；
- 风险驱动；
- 将新想法拆成 Follow-up。

---

# 39. 测试完成定义

一次测试活动可以被视为完成，当：

- 计划范围已覆盖；
- 关键场景已验证；
- 验收标准有结果；
- 主要风险有证据；
- 缺陷已记录；
- Blocker 和 Critical 已关闭；
- 回归已完成；
- 正式构建已验证；
- 已知限制已接受；
- 测试报告已归档；
- 下一步明确。

---

# 40. 最终原则

Testing Checklist 的本质不是：

> 把所有可能情况都列出来。

它的本质是：

> 用结构化方式识别高风险场景，建立足够证据，确认功能在真实条件下能够正确、稳定、安全地工作。

一个有效的测试过程应做到：

- 风险驱动；
- 验收驱动；
- 尽早开始；
- 正常与失败并重；
- 自动化与手工结合；
- 开发环境与正式环境结合；
- 结果可重复；
- 证据可追踪；
- 缺陷可管理；
- 退出条件明确。
