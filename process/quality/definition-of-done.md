# Definition of Done（完成定义）

> 状态：草案  
> 适用范围：通用的软件与游戏开发项目  
> 目的：定义一个功能、需求或工作项在被标记为 Done 前，必须满足的最低完成条件。

---

## 1. 文档目的

Definition of Done，简称 **DoD**，用于判断一个功能是否已经真正完成。

它回答的核心问题是：

> 这个功能是否已经达到可以被接受、交付、发布或进入后续阶段的质量标准？

DoD 的作用，是避免以下情况被误认为“完成”：

- 代码已经写完，但尚未集成；
- 主流程可用，但失败流程未处理；
- 功能在开发环境中可运行，但正式构建失败；
- UI 只完成默认状态，没有空状态、错误状态或禁用状态；
- 数据可以写入，但无法正确读取或迁移；
- 功能通过一次手工测试，但没有回归验证；
- 临时代码、调试入口和占位资源仍存在；
- 文档、配置、监控或发布准备尚未完成；
- 已知问题没有记录，却直接关闭任务；
- 功能进入主分支，但尚未对真实用户可用。

DoD 不是为了追求“绝对完美”，而是为了定义：

> 对当前项目、当前工作流等级和当前发布目标而言，什么叫作足够完整、足够可靠、可以被接受。

---

## 2. 适用范围

DoD 可以应用于不同层级：

- 单个 Task；
- User Story；
- Feature；
- Epic；
- Milestone；
- Release。

不同层级的完成标准不完全相同。

例如：

| 层级 | Done 的含义 |
|---|---|
| Task | 任务输出已完成并通过局部验证 |
| Feature | 功能已实现、集成、验证并达到体验标准 |
| Milestone | 该阶段目标和关键闭环已完成 |
| Release | 发布包已通过构建、回归和发布检查 |

本文件主要定义 **Feature 级别的 Definition of Done**，并提供 Task 和 Release 层级的补充说明。

---

## 3. Done 的定义

一个功能只有在以下条件基本成立时，才可以被标记为 Done：

- 功能范围已按批准内容实现；
- 验收标准通过；
- 已完成系统集成；
- 主要正常、失败和边界流程已验证；
- 不存在阻塞发布的问题；
- 数据、存档或迁移行为正确；
- 必要的 UI、反馈、文案、动画或音效已完成；
- 正式或接近正式的构建验证通过；
- 必要测试已完成；
- 回归风险处于可接受范围；
- 临时实现已清理或明确记录；
- 已知限制和技术债务已记录；
- 相关文档已更新；
- 必要的监控、日志、回滚或禁用方案已准备；
- 责任人已经确认接受该结果。

Done 的本质是：

> 功能已经不再依赖未完成的关键工作，可以安全进入待发布或发布流程。

---

## 4. Done 的设计原则

### 4.1 Done 不等于“代码写完”

代码完成只代表 Build 阶段的一部分完成。

完整功能通常还需要：

```text
Build
→ Integrate
→ Validate
→ Polish
→ [Done Gate]
```

如果一个功能尚未接入真实流程、尚未验证失败状态、尚未测试正式构建，那么它通常不能进入 Done。

---

### 4.2 Done 是质量 Gate

Done 应被视为一个明确的质量决策点。

推荐状态：

```text
Done
Done with Accepted Limitations
Needs Fixes
Needs More Validation
Return to Design
Cancelled
```

不建议使用模糊表述：

```text
基本完成
差不多好了
只剩一点小问题
开发完成
```

---

### 4.3 Done 不等于 Released

Done 表示：

> 功能已经达到可发布质量。

Released 表示：

> 功能已经进入目标用户实际使用的环境。

一个功能可以 Done，但尚未 Released，原因可能包括：

- 等待发布窗口；
- 等待其他功能；
- 等待完整回归测试；
- 等待平台认证；
- 等待部署审批；
- 等待商店审核；
- 等待版本冻结结束。

---

### 4.4 Done 标准应与风险匹配

不同工作流等级应使用不同严格程度的 DoD。

影响 DoD 严格程度的因素包括：

- 用户影响；
- 数据影响；
- 安全风险；
- 可逆性；
- 系统复杂度；
- 平台数量；
- 发布成本；
- 长期维护成本；
- 出错后的恢复难度。

---

### 4.5 Done 应当可验证

DoD 不能依赖纯主观判断。

不合格表述：

```text
体验不错
代码质量可以
性能应该没问题
大概不会影响旧功能
```

更合格的表述：

```text
所有关键验收标准通过；
正式构建中完成主流程；
无阻塞级缺陷；
旧版本数据可正常迁移；
性能指标未超过约定阈值。
```

---

## 5. Done 的核心维度

通用 DoD 建议从以下九个维度判断：

```text
Scope
Acceptance
Integration
Quality
Experience
Data
Release
Documentation
Operations
```

对应中文：

```text
范围
验收
集成
质量

发布

```

# 6. Scope：范围完成度

## 6.1 目标

确认批准范围内的内容已经完成，未完成内容没有被隐藏。

## 6.2 需要检查

- 已批准 Scope 是否全部完成；
- Non-goals 是否仍然保持在范围外；
- 是否出现未记录的范围变化；
- 是否有被临时跳过的关键项；
- 是否有“以后再补”但未记录的内容；
- 后续增强是否已拆分为独立工作项。

## 6.3 可接受情况

以下内容可以不在当前功能中完成：

- 已明确属于 Non-goals 的内容；
- 已被批准延期的非关键增强；
- 已拆分为独立 Follow-up Feature 的工作；
- 不影响当前验收和发布的优化项。

## 6.4 不可接受情况

- 主流程的一部分未完成；
- 关键失败状态缺失；
- 必要集成被跳过；
- 因时间不足删减范围但未记录；
- 验收标准中明确要求的能力尚未实现。

## 6.5 Done 条件

- [ ] 已批准 Scope 已完成；
- [ ] Non-goals 未被无意加入；
- [ ] 所有范围变化已记录；
- [ ] 未完成增强已拆分；
- [ ] 不存在隐藏的关键缺口。

---

# 7. Acceptance：验收标准

## 7.1 目标

确认功能是否满足预先定义的可观察结果。

## 7.2 验收要求

验收标准应：

- 具体；
- 可测试；
- 可观察；
- 覆盖主流程；
- 覆盖主要失败流程；
- 覆盖关键边界场景；
- 与真实用户或系统行为相关。

## 7.3 验收结果

建议记录：

```text
Pass
Fail
Not Applicable
Accepted Limitation
Deferred with Approval
```

## 7.4 验收证据

证据可以包括：

- 测试结果；
- 截图；
- 视频；
- 自动化测试报告；
- 构建号；
- 日志；
- 性能报告；
- QA 记录；
- 评审结论。

## 7.5 Done 条件

- [ ] 所有关键验收标准通过；
- [ ] 未通过项已处理或正式接受；
- [ ] 验收结果有记录；
- [ ] 验收由明确责任人确认；
- [ ] 不存在“尚未验证但先算通过”的关键项。

---

# 8. Integration：系统集成

## 8.1 目标

确认功能已经进入真实产品流程，而不是只在孤立环境中可用。

## 8.2 集成范围

- 主导航或主流程；
- 共享状态；
- 数据源；
- API；
- 存储；
- 权限；
- 认证；
- 页面或场景切换；
- 其他系统；
- UI；
- 日志；
- 埋点；
- 本地化；
- 平台能力；
- 构建配置。

## 8.3 需要验证

- 功能入口真实可达；
- 功能退出路径正常；
- 状态切换正确；
- 依赖系统正常；
- 不破坏已有流程；
- 场景切换或页面刷新后行为正确；
- 重启后数据状态正确；
- 真实配置下行为正确；
- 无调试环境依赖。

## 8.4 Done 条件

- [ ] 功能已接入目标产品流程；
- [ ] 依赖系统已连接；
- [ ] 共享状态行为正确；
- [ ] 不存在关键集成阻塞；
- [ ] 真实或接近真实环境验证通过；
- [ ] 无仅在测试环境中成立的隐藏依赖。

---

# 9. Quality：功能质量

## 9.1 目标

确认功能在正确性、稳定性、性能和兼容性方面达到要求。

## 9.2 质量维度

### 正确性

- 正常流程正确；
- 失败流程正确；
- 边界条件正确；
- 状态转换正确；
- 重复操作正确；
- 并发或异步行为正确。

### 稳定性

- 不崩溃；
- 不产生严重异常；
- 不造成不可恢复状态；
- 错误可以被处理；
- 异常状态可恢复。

### 性能

- 响应时间符合要求；
- 内存使用可接受；
- 帧率或吞吐符合目标；
- 不产生明显卡顿；
- 不引入不可接受的后台开销。

### 兼容性

- 目标平台正常；
- 支持的分辨率正常；
- 支持的输入方式正常；
- 旧数据兼容；
- 旧流程不被破坏；
- 依赖版本兼容。

## 9.3 缺陷等级

建议至少区分：

```text
Blocker
Critical
Major
Minor
Trivial
```

### Blocker / Critical

通常不能带入 Done。

### Major

需要根据影响判断，通常应修复或正式接受。

### Minor / Trivial

可以在记录后进入 Done，但需满足：

- 不影响主流程；
- 不影响数据；
- 不造成严重体验问题；
- 有清晰后续处理方式。

## 9.4 Done 条件

- [ ] 无 Blocker 或 Critical 缺陷；
- [ ] Major 缺陷已修复或正式接受；
- [ ] 核心测试通过；
- [ ] 回归风险可接受；
- [ ] 性能符合要求；
- [ ] 目标平台和环境验证通过；
- [ ] 已知问题已记录。

# 10. Experience：体验完整度

## 10.1 目标

确认功能不仅“能运行”，还达到当前交付目标所需的体验质量。

## 10.2 体验范围

- 信息层级；
- 操作反馈；
- 加载状态；
- 空状态；
- 失败状态；
- 禁用状态；
- 成功状态；
- 文案；
- 动画；
- 音效；
- 视觉一致性；
- 可访问性；
- 响应式布局；
- 多语言；
- 键盘和手柄导航；
- 错误提示；
- 恢复路径。

## 10.3 常见遗漏

- 只有默认状态，没有 Loading；
- 失败后没有重试；
- 无数据时页面空白；
- 按钮禁用但没有原因；
- 操作成功没有反馈；
- 错误信息不可理解；
- 关键操作没有确认或撤销；
- 小屏或高分辨率布局异常；
- 键盘焦点不可见；
- 文案仍是占位符。

## 10.4 Done 条件

- [ ] 所有必要 UI 状态完整；
- [ ] 关键操作有明确反馈；
- [ ] 错误信息可理解；
- [ ] 必要时有重试、取消或恢复路径；
- [ ] 占位资源和临时文案已清理；
- [ ] 体验与现有产品保持一致；
- [ ] 可访问性要求已满足或记录。

---

# 11. Data：数据、存档与迁移

## 11.1 目标

确认功能不会造成数据丢失、不一致或兼容问题。

## 11.2 适用场景

- 数据库存储；
- 本地存档；
- 云存储；
- 用户设置；
- 配置数据；
- 缓存；
- 账户状态；
- 内容版本；
- Schema 变化；
- 数据迁移。

## 11.3 需要验证

- 数据写入正确；
- 数据读取正确；
- 数据删除正确；
- 重启后状态正确；
- 失败时不会损坏数据；
- 重复操作不会产生异常数据；
- 旧版本数据可以迁移；
- 新版本数据不会被旧逻辑错误读取；
- 回滚策略明确；
- 数据备份或恢复策略可用。

## 11.4 Done 条件

- [ ] 数据读写验证通过；
- [ ] 持久化验证通过；
- [ ] 迁移逻辑验证通过；
- [ ] 旧数据兼容性验证通过；
- [ ] 数据失败场景已测试；
- [ ] 不存在未处理的数据丢失风险；
- [ ] 必要的版本号或 Schema 已更新；
- [ ] 回滚或恢复方式明确。

---

# 12. Release：正式构建与发布准备

## 12.1 目标

确认功能在接近生产环境的构建中正常，并具备进入发布流程的条件。

## 12.2 需要验证

- 正式构建成功；
- 目标平台构建成功；
- 功能在构建包中可用；
- 文件和资源完整；
- 配置正确；
- 依赖版本正确；
- 权限正确；
- 密钥和环境变量正确；
- 发布开关正确；
- 回滚方式可行；
- 灰度或禁用方案可用；
- 构建中不存在调试依赖。

## 12.3 Done 条件

- [ ] 正式或接近正式的构建通过；
- [ ] 目标平台验证通过；
- [ ] 构建包中功能可达；
- [ ] 资源和配置完整；
- [ ] 无开发环境专属依赖；
- [ ] 发布开关已确认；
- [ ] 必要时已有回滚或禁用方案；
- [ ] 发布阻塞项为零。

# 13. Documentation：文档

## 13.1 目标

确认功能相关知识已经被更新并可被后续维护者理解。

## 13.2 可能需要更新的文档

- 功能设计；
- 技术设计；
- API；
- 数据结构；
- 存档格式；
- 配置说明；
- 使用说明；
- 测试说明；
- 部署说明；
- 发布说明；
- Changelog；
- 运行手册；
- 已知问题；
- 决策记录。

## 13.3 文档完成原则

不是所有功能都需要大量文档。

应根据影响范围判断：

- 是否改变公共接口；
- 是否改变数据；
- 是否改变流程；
- 是否增加维护成本；
- 是否引入新工具；
- 是否有复杂配置；
- 是否有特殊排错方式。

## 13.4 Done 条件

- [ ] 相关设计文档已更新；
- [ ] 技术文档已更新；
- [ ] 数据或迁移说明已更新；
- [ ] 使用或操作说明已更新；
- [ ] 已知限制已记录；
- [ ] Changelog 或版本说明已准备；
- [ ] 文档与当前实现一致。

---

# 14. Operations：运维、监控与维护

## 14.1 目标

确认功能在发布后能够被观察、维护、禁用或恢复。

## 14.2 适用内容

- 日志；
- 监控；
- 埋点；
- 告警；
- Feature Flag；
- 灰度发布；
- 回滚；
- 数据修复；
- 客服支持；
- 运行手册；
- 所有权；
- 维护责任。

## 14.3 需要考虑

- 发布后如何知道它是否正常；
- 如何发现错误；
- 如何定位问题；
- 是否可以关闭功能；
- 是否可以回滚；
- 谁负责处理事故；
- 用户遇到问题时如何恢复；
- 是否需要数据补偿。

## 14.4 Done 条件

- [ ] 必要日志已添加；
- [ ] 必要监控或埋点已添加；
- [ ] 错误能够被发现和定位；
- [ ] 必要时可禁用或回滚；
- [ ] 维护 Owner 明确；
- [ ] 事故处理方式明确；
- [ ] 数据修复或恢复方案已考虑。

# 15. 不同 Workflow Track 的 Done 标准

---

## 15.1 Fast Track

### 适用范围

- 文案修改；
- 小型样式调整；
- 明确的小型 Bug；
- 单个配置修改；
- 单个资源替换；
- 影响范围有限且易回滚的改动。

### 最低 Done 条件

- [ ] 改动已完成；
- [ ] 目标结果已验证；
- [ ] 不影响相邻功能；
- [ ] 无阻塞缺陷；
- [ ] 可以安全回滚；
- [ ] 必要记录已更新。

### 可简化内容

- 完整技术文档；
- 正式性能测试；
- 复杂监控；
- 完整迁移方案；
- 多层级评审。

---

## 15.2 Standard Track

### 适用范围

- 普通产品功能；
- 小型系统扩展；
- 常见 UI 流程；
- 中等复杂度集成；
- 新内容类型。

### Done 条件

- [ ] Scope 完成；
- [ ] 验收标准通过；
- [ ] 集成完成；
- [ ] 正常和失败流程通过；
- [ ] 无阻塞级缺陷；
- [ ] 必要 UI 状态完整；
- [ ] 数据行为正确；
- [ ] 正式构建验证通过；
- [ ] 相关文档更新；
- [ ] 已知问题已记录。

---

## 15.3 Full Track

### 适用范围

- 核心系统；
- 大型架构调整；
- 数据和存档结构变化；
- 安全敏感功能；
- 大型 UI 重构；
- 跨多个系统的功能；
- 难以回滚的改动。

### 附加 Done 条件

除 Standard Track 条件外，还应包括：

- [ ] 完整回归测试；
- [ ] 正式性能测试；
- [ ] 数据迁移和回滚验证；
- [ ] 安全与隐私检查；
- [ ] 多平台验证；
- [ ] 灰度、禁用或回滚方案；
- [ ] 监控和告警；
- [ ] 运维说明；
- [ ] 风险接受记录；
- [ ] 发布审批；
- [ ] 发布后验证计划。

---

# 16. Done 状态

建议使用以下状态。

---

## 16.1 Done

所有必要完成条件已满足，可以进入发布流程。

---

## 16.2 Done with Accepted Limitations

存在少量已知限制，但：

- 不影响主流程；
- 不构成数据风险；
- 不构成安全风险；
- 不构成发布阻塞；
- 已正式记录；
- 已有后续处理计划；
- 已由责任人接受。

---

## 16.3 Needs Fixes

存在明确缺陷，需要修复后再次评审。

---

## 16.4 Needs More Validation

实现可能正确，但证据不足。

常见情况：

- 尚未测试正式构建；
- 尚未测试数据迁移；
- 尚未验证目标平台；
- 尚未完成性能测试；
- 尚未完成关键回归。

---

## 16.5 Return to Design

发现问题不是实现缺陷，而是设计方向、范围或验收定义本身存在问题。

---

## 16.6 Cancelled

功能不再继续。

应记录：

- 取消原因；
- 当前产物如何处理；
- 是否需要回滚；
- 是否需要清理资源；
- 是否存在可复用成果。

---

# 17. 必须阻止 Done 的情况

以下情况通常不应允许进入 Done：

- 关键验收标准未通过；
- 主流程存在阻塞缺陷；
- 失败流程会导致数据损坏；
- 尚未完成系统集成；
- 只在开发环境可用；
- 正式构建无法运行；
- 旧数据无法迁移；
- 存在未处理的安全风险；
- 存在未记录的高风险技术债务；
- UI 关键状态缺失；
- 占位资源仍在正式流程中；
- 无法回滚或禁用高风险功能；
- 关键文档未更新；
- 负责人尚未确认接受；
- 测试结果缺失；
- 发布阻塞项尚未关闭。

---

# 18. 可以带入 Done 的已知限制

不是所有问题都必须在 Done 前解决。

可接受限制通常满足：

- 不影响主流程；
- 不影响数据正确性；
- 不影响安全；
- 不影响兼容性；
- 不导致严重体验问题；
- 不阻止发布；
- 有明确记录；
- 有 Owner；
- 有后续计划或明确决定不处理。

示例：

- 非关键动画细节；
- 次要文案优化；
- 小型视觉间距；
- 非关键性能优化；
- 极少出现且可恢复的 Minor 问题；
- 不影响使用的辅助功能缺失。

---

# 19. Done Review

## 19.1 形式

Done Review 可以是：

- 文档评审；
- QA 验收；
- Demo；
- 代码与产品联合检查；
- 发布前检查；
- 个人项目中的正式自检。

## 19.2 评审问题

```text
批准范围是否完成？
验收标准是否通过？
是否完成真实集成？
失败和边界流程是否验证？
是否存在发布阻塞问题？
数据和迁移是否正确？
正式构建是否验证？
UI 和反馈是否完整？
已知问题是否记录？
文档是否更新？
是否具备回滚、禁用或恢复方式？
是否可以安全进入发布流程？
```

## 19.3 输出

```text
Done
Done with Accepted Limitations
Needs Fixes
Needs More Validation
Return to Design
Cancelled
```

# 20. 重新进行 Done Review 的触发条件

功能通过 Done 后，若发生以下变化，应重新评审：

- Scope 改变；
- 验收标准改变；
- 新增关键功能；
- 技术架构改变；
- 数据结构改变；
- 迁移逻辑改变；
- 平台范围改变；
- 发布配置改变；
- 新发现 Blocker 或 Critical 缺陷；
- 功能长期未发布；
- 依赖版本发生重大变化；
- 发布环境发生变化；
- 原来的测试证据已失效。

# 24. 常见失败模式

## 24.1 把代码完成当作 Done

表现：

- 未集成；
- 未验证正式构建；
- 无失败状态；
- 无数据测试；
- 无回归测试。

改进：

- 强制区分 Build Complete 和 Done；
- 保留 Integrate、Validate、Polish 和 Done Gate。

---

## 24.2 DoD 过于宽松

表现：

- 每个功能都可以轻易通过；
- 缺陷和限制不记录；
- 测试证据缺失；
- 发布后问题频发。

改进：

- 明确阻塞条件；
- 强制验收和正式构建验证；
- 高风险功能使用 Full Track。

---

## 24.3 DoD 过于严格

表现：

- 小改动也要求完整发布流程；
- 文档和测试成本超过改动价值；
- 团队绕过流程；
- Done 永远无法达到。

改进：

- 使用 Fast、Standard、Full Track；
- 根据风险和影响调整标准；
- 允许 Done with Accepted Limitations。

---

## 24.4 已知问题未记录

表现：

- 团队口头知道问题；
- 发布后没人记得限制；
- 用户遇到已知问题却没有处理方案；
- 相同问题反复被发现。

改进：

- 所有已接受限制必须记录；
- 标明影响、Owner 和后续计划。

---

## 24.5 无限 Polish

表现：

- 功能无法关闭；
- 视觉细节不断调整；
- 发布被非关键优化拖延；
- 没有明确质量目标。

改进：

- 在开始 Polish 前定义目标标准；
- 将后续增强拆分为 Follow-up；
- 以 Done 条件判断是否继续。

---

## 24.6 测试只覆盖主流程

表现：

- 成功路径正常；
- 失败路径崩溃；
- 重复操作产生错误数据；
- 边界状态未处理；
- 断网或异常状态不可恢复。

改进：

- 强制失败流程和边界检查；
- 在验收标准中提前定义异常行为。

---

## 24.7 Done 后继续偷偷改范围

表现：

- 已标记 Done，但仍不断加入功能；
- 测试证据失效；
- 发布包与 Done Review 不一致。

改进：

- Done 后新增能力必须重新进入流程；
- 重大变化触发重新评审。

# 25. Task 级 Done

对于普通 Task，可以使用更轻量的完成定义。

```markdown
## Task Done Checklist

- [ ] 任务目标已完成；
- [ ] 局部验证通过；
- [ ] 无明显回归；
- [ ] 代码或资产已提交；
- [ ] 必要评审已完成；
- [ ] 临时内容已清理；
- [ ] 相关文档或注释已更新；
- [ ] 任务输出可被上层 Feature 使用。
```

Task Done 不代表 Feature Done。

# 26. Release 级 Done

一个 Release 可以被视为 Done，通常需要：

```markdown
## Release Done Checklist

- [ ] 计划功能已通过 Done Gate；
- [ ] 回归测试完成；
- [ ] 正式构建完成；
- [ ] 目标平台验证完成；
- [ ] 数据迁移验证完成；
- [ ] 发布说明完成；
- [ ] 版本号和构建号正确；
- [ ] 回滚方案可用；
- [ ] 发布后验证完成；
- [ ] 已知问题已公开记录；
- [ ] 监控和支持已准备。
```

# 27. 最终判断标准

一个功能可以被认定为 Done，当：

> 它已经完成批准范围，通过关键验收，完成真实集成，在质量、体验、数据、构建、文档和运维方面达到当前交付标准，并且不存在阻止其进入发布流程的关键问题。

Done 的本质不是：

> 开发已经没有任何事情可做。

Done 的本质是：

> 这个功能已经足够完整、可靠、可接受，可以安全交付。
