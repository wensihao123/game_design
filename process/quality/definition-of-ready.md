# Definition of Ready（准备就绪定义）

> 状态：草案  
> 适用范围：通用的软件与游戏开发项目  
> 目的：定义一个功能、需求或工作项在进入正式开发前，必须满足的最低准备条件。

---

## 1. 文档目的

Definition of Ready，简称 **DoR**，用于判断一个功能或需求是否已经具备进入正式开发的条件。

它的核心问题是：

> 这个工作是否已经足够清楚、可控、可验证，可以正式投入设计、实现和交付？

DoR 的目的不是增加审批环节，也不是要求所有细节在开发前全部确定。

它的作用是防止以下问题过早进入开发：

- 问题定义不清；
- 用户价值不明确；
- 范围持续变化；
- 核心流程存在冲突；
- 关键依赖尚未准备；
- 技术风险未被识别；
- 验收标准缺失；
- 负责人和决策方式不明确；
- 开发完成后才发现无法集成；
- 因基础信息不足导致大量返工。

一个功能只有在满足 Ready 条件后，才应进入正式的设计、计划或实现阶段。

---

## 2. 适用范围

DoR 适用于：

- 新功能；
- 重要系统改动；
- UI 或交互流程；
- 数据结构变化；
- 存档或持久化变化；
- 技术架构调整；
- 跨模块改动；
- 具有较高风险或成本的 Bug 修复；
- 需要多个角色协作的工作；
- 进入正式里程碑的工作项。

对于以下低风险工作，可以使用轻量版 Ready 标准：

- 文案修正；
- 小型视觉调整；
- 简单配置变更；
- 明确且易回滚的小型 Bug；
- 单个资源替换；
- 局部重命名；
- 不影响架构、数据和发布的微小调整。

---

## 3. Ready 的定义

一个工作项在以下条件成立时，可以被认定为 Ready：

- 目标明确；
- 价值明确；
- 范围明确；
- 核心行为明确；
- 依赖可控；
- 关键风险已识别；
- 验收方式明确；
- 负责人明确；
- 剩余未决问题不会阻碍下一阶段；
- 当前项目优先级允许正式投入。

Ready 不代表所有问题都已解决。

Ready 代表：

> 剩余的不确定性可以在后续工作中安全处理，不会推翻目标、范围、核心流程或主要交付方案。

---

## 4. Ready 的设计原则

### 4.1 Ready 是 Gate，不是普通阶段

Ready 是一个决策门槛，而不是一个持续执行工作的阶段。

推荐理解：

```text
Capture
→ Clarify
→ Assess
→ [Ready Gate]
→ Design / Plan / Build
```

Ready Gate 的结果应是一个明确判断：

```text
Ready
Ready with Conditions
Ready for Prototype
Not Ready
Deferred
Rejected
```

---

### 4.2 Ready 判断“是否足够开始”

DoR 不要求：

- 所有 UI 已完成；
- 所有技术细节已确定；
- 所有资产已准备；
- 所有任务已精确估算；
- 所有边界条件都有最终答案；
- 所有风险都已完全消失。

它要求的是：

- 方向不会因基础问题频繁重做；
- 主要工作范围不会持续膨胀；
- 开发者知道自己要解决什么；
- 结果可以被验证；
- 关键依赖不会在中途突然阻塞；
- 高风险项已有处理方案。

---

### 4.3 Ready 不等于 Design Complete

Ready 与设计完成是两个不同概念。

| 概念 | 核心问题 |
|---|---|
| Ready | 是否已经具备正式开始解决问题的条件？ |
| Design Complete | 解决方案是否已经明确到可以稳定实现？ |

对于大型功能：

```text
Ready Gate
→ Design
→ Prototype
→ Plan
→ Build
```

对于小型功能，Ready、Design 和 Plan 可能会非常接近。

---

### 4.4 Ready 不等于高优先级

优先级和准备状态是两个独立维度。

```text
Priority = 这件事现在有多重要
Ready = 这件事现在能不能稳定开始
```

一个功能可以：

```text
高优先级，但 Not Ready
```

也可以：

```text
Ready，但当前优先级较低
```

高优先级不应成为绕过 Ready Gate 的理由。

如果高优先级功能尚未 Ready，应优先解决它的阻塞问题，而不是直接进入混乱开发。

---

### 4.5 Ready 标准应根据风险调整

不同工作不应使用完全相同的 Ready 要求。

建议根据以下因素调整严格程度：

- 影响范围；
- 复杂度；
- 可逆性；
- 技术风险；
- 用户风险；
- 数据风险；
- 发布风险；
- 依赖数量；
- 出错成本；
- 长期维护成本。

---

## 5. Ready 的核心维度

通用 DoR 建议从以下八个维度判断：

```text
Value
Scope
Behavior
Dependencies
Risk
Validation
Ownership
Feasibility
```

对应中文：

```text
价值
范围
行为
依赖
风险
验证
责任
可行性
```

---

# 6. Value：价值

## 6.1 目标

确认为什么要做这项工作，以及它为谁创造价值。

## 6.2 需要明确

- 解决什么问题；
- 服务哪个用户或使用场景；
- 为什么现在需要解决；
- 预期产生什么结果；
- 不做会造成什么影响；
- 是否与当前目标或里程碑一致。

## 6.3 合格示例

```text
用户需要重复配置相同选项，导致操作成本较高。
本功能允许保存并复用配置，以减少重复操作。
```

## 6.4 不合格示例

```text
感觉这个功能不错。
其他产品都有。
以后可能会用到。
可以让产品更完整。
```

## 6.5 Ready 条件

- [ ] 问题已经明确描述；
- [ ] 目标用户或场景明确；
- [ ] 预期价值明确；
- [ ] 与当前目标或里程碑的关系明确。

---

# 7. Scope：范围

## 7.1 目标

明确本次工作包含什么，以及明确不包含什么。

## 7.2 需要明确

- 本次交付内容；
- 最小交付边界；
- 主要功能范围；
- 明确排除的内容；
- 与其他功能的边界；
- 是否存在明显范围冲突。

## 7.3 Scope 示例

```markdown
## 范围

- 创建预设；
- 编辑预设；
- 删除预设；
- 应用预设。

## 非目标

- 预设分享；
- 云同步；
- 团队协作；
- 公共市场。
```

## 7.4 为什么必须定义 Non-goals

Non-goals 用于防止以下问题：

- 开发中不断加入“顺便做一下”；
- 多个功能边界混在一起；
- 任务无法关闭；
- 验收标准持续变化；
- 原本的小功能变成大型系统。

## 7.5 Ready 条件

- [ ] Scope 明确；
- [ ] Non-goals 明确；
- [ ] 最小交付边界明确；
- [ ] 没有关键范围冲突；
- [ ] 后续增强内容已与本次范围分离。

---

# 8. Behavior：核心行为

## 8.1 目标

确认功能的主要行为已经清楚到可以开始设计和实现。

## 8.2 至少需要明确

- 谁触发；
- 在什么条件下触发；
- 系统如何响应；
- 成功状态；
- 失败状态；
- 重要边界状态；
- 是否涉及异步、等待或加载；
- 是否涉及重复操作；
- 是否涉及取消或撤销。

## 8.3 简单行为流

```text
用户发起操作
→ 系统验证输入
→ 验证成功则执行并保存
→ 验证失败则保留输入并提示原因
```

## 8.4 常见状态

```text
Idle
Loading
Success
Empty
Failed
Disabled
Cancelled
```

## 8.5 Ready 条件

- [ ] 主流程明确；
- [ ] 主要成功状态明确；
- [ ] 主要失败状态明确；
- [ ] 关键边界状态已识别；
- [ ] 不存在互相冲突的核心行为定义。

---

# 9. Dependencies：依赖

## 9.1 目标

确认该功能所依赖的系统、数据、资产、工具或外部条件不会在开发中途突然造成阻塞。

## 9.2 常见依赖

- 前置功能；
- 共享系统；
- API；
- 数据源；
- 第三方服务；
- UI 设计；
- 美术资产；
- 音频资产；
- 内容配置；
- 平台能力；
- 权限；
- 外部团队；
- 构建环境；
- 技术决策。

## 9.3 依赖可被接受的状态

每个关键依赖至少应满足以下之一：

```text
已经可用
有明确交付时间
有临时替代方案
不阻碍当前阶段
```

## 9.4 Not Ready 的依赖情况

- 不知道数据从哪里来；
- 接口尚未定义；
- 依赖功能没有计划；
- 第三方能力未确认；
- 负责人不明确；
- 关键资产规格不存在；
- 测试环境不可用；
- 发布条件未知。

## 9.5 Ready 条件

- [ ] 主要依赖已列出；
- [ ] 关键依赖状态明确；
- [ ] 数据来源明确；
- [ ] 依赖负责人明确；
- [ ] 不存在无计划的关键阻塞依赖。

---

# 10. Risk：风险

## 10.1 目标

在正式投入前识别可能导致失败、返工或高成本的问题。

## 10.2 常见风险类型

- 技术可行性；
- 性能；
- 安全；
- 隐私；
- 数据丢失；
- 数据迁移；
- 向后兼容；
- 平台兼容；
- 法律与合规；
- 内容生产成本；
- 依赖不稳定；
- 用户体验；
- 可访问性；
- 发布和回滚；
- 长期维护。

## 10.3 Ready 对风险的要求

Ready 不要求风险全部消失，但要求：

- 关键风险已被识别；
- 风险影响已被理解；
- 高风险项已有验证或缓解计划；
- 不存在完全未知的致命风险；
- 必要时已决定先做 Prototype 或 Spike。

## 10.4 风险处理方式

```text
Accept
Mitigate
Prototype
Spike
Transfer
Avoid
Defer
Reject
```

## 10.5 Ready 条件

- [ ] 关键风险已识别；
- [ ] 高风险项已有处理方案；
- [ ] 必要的 Prototype 或 Spike 已计划；
- [ ] 不存在未经评估的致命风险；
- [ ] 回滚或禁用需求已考虑。

---

# 11. Validation：验证

## 11.1 目标

确认项目已经知道如何判断这个功能是否完成和正确。

## 11.2 验收标准要求

验收标准应当：

- 具体；
- 可观察；
- 可测试；
- 与用户或系统行为有关；
- 不依赖主观表述；
- 能覆盖主要成功和失败场景。

## 11.3 不合格示例

```text
功能可以正常使用。
界面看起来不错。
性能没有问题。
```

## 11.4 合格示例

```text
用户可以创建、编辑和删除预设；
非法输入不会覆盖已有数据；
保存结果在重新启动后仍然存在；
重复提交不会创建重复记录；
功能在正式构建中可以完成主要流程。
```

## 11.5 Ready 条件

- [ ] 初步验收标准已写出；
- [ ] 预期结果可以被观察和测试；
- [ ] 正常流程有验证方式；
- [ ] 主要失败流程有验证方式；
- [ ] 必要的性能、兼容或迁移验证已识别。

---

# 12. Ownership：责任

## 12.1 目标

明确谁负责推动、决策、实现、验收和发布。

## 12.2 至少需要明确

- 谁负责推动功能；
- 谁决定范围；
- 谁处理冲突；
- 谁负责实现；
- 谁负责验收；
- 谁决定是否发布；
- 谁负责后续维护。

个人项目中，同一个人可以承担所有角色，但仍应区分角色责任。

## 12.3 角色示例

| 角色 | 责任 |
|---|---|
| Product / Design | 目标、价值、范围和体验 |
| Engineering | 技术方案、实现和集成 |
| QA | 验收、边界与回归 |
| Release | 构建、发布和回滚 |
| Owner | 最终决策和优先级 |

## 12.4 Ready 条件

- [ ] 功能 Owner 明确；
- [ ] 范围决策人明确；
- [ ] 验收责任明确；
- [ ] 冲突处理方式明确；
- [ ] 后续维护责任明确。

---

# 13. Feasibility：可行性

## 13.1 目标

判断当前是否具备现实条件开始推进。

## 13.2 需要考虑

- 是否有足够时间；
- 是否具备必要技能；
- 是否有可用工具；
- 是否有测试环境；
- 是否符合当前里程碑；
- 是否可以承担长期维护；
- 成本是否在可接受范围；
- 是否有无法控制的外部因素。

## 13.3 Ready 条件

- [ ] 当前资源足以开始；
- [ ] 所需工具或环境可用；
- [ ] 技术能力可获得；
- [ ] 工作符合当前计划；
- [ ] 长期维护成本可接受；
- [ ] 不存在明显无法承担的成本。

---

# 14. Workflow Track 的 Ready 标准

不同工作流等级使用不同 Ready 要求。

---

## 14.1 Fast Track

### 适用范围

- 文案调整；
- 小型视觉修复；
- 明确的小型 Bug；
- 单个配置修改；
- 单个资源替换；
- 易于回滚的局部调整。

### Ready 最低条件

- [ ] 目标明确；
- [ ] 影响范围明确；
- [ ] 预期结果明确；
- [ ] 不存在关键依赖；
- [ ] 验证方式明确；
- [ ] 可以安全回滚。

### 不强制要求

- 完整 Feature Brief；
- 正式风险评估；
- 详细技术设计；
- 独立 Prototype；
- 完整多阶段评审。

---

## 14.2 Standard Track

### 适用范围

- 普通产品功能；
- 小型系统扩展；
- 常见 UI 流程；
- 新内容类型；
- 中等复杂度集成。

### Ready 条件

- [ ] 问题和价值明确；
- [ ] Scope 和 Non-goals 明确；
- [ ] 主流程明确；
- [ ] 主要依赖明确；
- [ ] 主要风险已识别；
- [ ] 初步验收标准存在；
- [ ] Owner 明确；
- [ ] 与当前优先级或里程碑一致。

---

## 14.3 Full Track

### 适用范围

- 核心系统；
- 大型架构调整；
- 数据结构变化；
- 持久化变化；
- 安全敏感功能；
- 大型 UI 重构；
- 跨多个系统的改动；
- 难以回滚的决策。

### 附加 Ready 条件

除 Standard Track 条件外，还应包括：

- [ ] 正式风险评估；
- [ ] 数据和迁移影响明确；
- [ ] 安全和合规影响明确；
- [ ] 技术可行性已初步判断；
- [ ] Prototype 或 Spike 计划明确；
- [ ] 回滚或禁用方案明确；
- [ ] 关键外部依赖已有承诺；
- [ ] 发布和运维影响明确；
- [ ] 验收策略较完整；
- [ ] 成本和时间范围已被接受。

---

# 15. Ready 状态

建议使用以下状态。

---

## 15.1 Ready

没有关键阻塞，可以进入目标阶段。

---

## 15.2 Ready with Conditions

存在少量未决事项，但不会推翻主要目标、范围或核心方案。

可接受条件示例：

- 非关键文案未最终确定；
- 次要动画规格待补充；
- 非关键测试数据尚未准备；
- 局部视觉细节待确认；
- 不影响结构的参数尚未定稿。

条件必须：

- 被明确记录；
- 有负责人；
- 有完成时间或触发条件；
- 不影响关键路径；
- 不会导致重大返工。

---

## 15.3 Ready for Prototype

问题值得继续，但关键体验、技术或性能风险尚未验证。

此状态允许进入：

```text
Prototype
Spike
Research
Experiment
```

不允许直接进入正式 Build。

---

## 15.4 Not Ready

存在会导致阻塞、误解或高概率返工的问题。

---

## 15.5 Deferred

该功能可能有价值，但当前不投入。

应记录：

- 延后原因；
- 重新评估条件；
- 可能的目标时间；
- 是否依赖其他工作。

---

## 15.6 Rejected

明确决定不做。

应记录：

- 拒绝原因；
- 评估依据；
- 是否存在替代方案；
- 未来什么条件变化时可以重新考虑。

---

# 16. 必须判定为 Not Ready 的情况

以下情况通常应阻止正式开发：

- 无法清楚说明要解决的问题；
- 不知道服务哪个用户或场景；
- Scope 持续冲突；
- Non-goals 缺失且范围不断扩大；
- 主流程存在互斥版本；
- 关键决策人未确定；
- 依赖系统没有计划；
- 数据来源未知；
- 外部接口未定义；
- 存在未评估的重大安全或数据风险；
- 不知道如何验收；
- 成本明显超出可承受范围；
- 当前里程碑不容纳该工作；
- 技术可行性完全未知，却准备直接 Build；
- 发布或回滚方式完全未知；
- 功能与现有系统存在未解决冲突。

DoR 的价值来自能够明确地说：

> 当前还不应该开始正式开发。

如果所有工作都能自动通过 Ready，DoR 就失去了意义。

---

# 17. 可以保留的未决问题

不是所有未决问题都会阻止 Ready。

可以保留的问题通常具有以下特点：

- 不改变功能目标；
- 不扩大主要范围；
- 不推翻核心架构；
- 不改变验收方式；
- 可以局部调整；
- 容易回滚；
- 不影响其他系统；
- 不影响关键路径。

可保留示例：

- 非关键颜色；
- 小型间距；
- 内部命名；
- 次要动画速度；
- 可延期的表现增强；
- 非关键提示文案；
- 不影响数据结构的参数；
- 局部优化方案。

这些内容应记录在：

```text
Open Questions
Conditions
Follow-up
```

而不是被假装为已经解决。

---

# 18. Ready Review

## 18.1 形式

Ready Review 可以是：

- 文档自检；
- 异步评审；
- 短会；
- Backlog Refinement 的一部分；
- 个人项目中的阶段检查。

不要求使用正式会议。

## 18.2 评审问题

```text
问题是否清楚？
价值是否清楚？
范围是否清楚？
Non-goals 是否清楚？
主流程是否清楚？
依赖是否可控？
关键风险是否处理？
验收标准是否存在？
Owner 是否明确？
是否使用了正确的 Track？
是否可以正式投入？
```

## 18.3 输出

评审结果必须明确：

```text
Ready
Ready with Conditions
Ready for Prototype
Not Ready
Deferred
Rejected
```

避免使用模糊结论：

```text
基本可以
差不多
应该没问题
先做起来再说
```

---

# 19. 重新进行 Ready Review 的触发条件

功能通过 Ready 后，若发生重大变化，应重新评审。

常见触发条件：

- 目标用户变化；
- 问题定义变化；
- 主流程变化；
- Scope 明显扩大；
- Non-goals 被纳入当前范围；
- 新增关键外部依赖；
- 技术方案发生架构级变化；
- 数据或存档模型变化；
- 安全、隐私或合规影响变化；
- 成本显著增加；
- 交付时间显著变化；
- 验收标准被重新定义；
- 功能暂停时间过长；
- 外部环境发生变化；
- 原先关键假设被推翻。

核心判断：

> 原先作出投入承诺时所依据的条件，是否仍然成立？

小型实现调整不需要重新评审。

- [ ] 主流程明确；

- [ ] 长期维护成本可接受。

- [ ] Workflow Track 已确定；
- [ ] Ready Review 已完成；
- [ ] 未决问题和条件已记录；
- [ ] 下一阶段明确；

# 23. 常见失败模式

## 23.1 Checklist 形式化

表现：

- 所有选项都被机械勾选；
- 内容只写“已确认”；
- 没有实际证据；
- 评审只检查是否填写，不检查质量。

改进：

- 重要项链接到具体说明；
- 关键风险必须写清楚；
- 验收标准必须可测试；
- Reviewer 检查信息质量，而不是表格完整度。

---

## 23.2 Ready 标准过于宽松

表现：

- 所有需求都能进入开发；
- Scope 持续变化；
- 验收标准在最后才补；
- 关键依赖中途暴露；
- DoR 没有拦截作用。

改进：

- 明确 Not Ready 条件；
- 强制 Scope 和 Non-goals；
- 强制初步验收标准；
- 高风险功能先进入 Prototype。

---

## 23.3 Ready 标准过于严格

表现：

- 小改动也需要大量文档；
- 所有设计必须提前完成；
- 团队不愿执行流程；
- 文档成本超过实际价值；
- Ready 变成小型瀑布。

改进：

- 使用 Fast、Standard 和 Full Track；
- 根据风险调整要求；
- 允许 Ready with Conditions；
- 只要求足够开始，不要求全部完成。

---

## 23.4 将 Ready 当成永久承诺

Ready 表示：

```text
根据当前信息，值得且可以开始。
```

不表示：

```text
无论未来发现什么都必须完成。
```

新证据出现后，功能仍可以：

- 回退；
- 缩小范围；
- 延后；
- 取消；
- 重新设计。

---

## 23.5 把“大家都知道”当作正式说明

口头理解和个人记忆不是稳定信息。

关键内容应被记录：

- 目标；
- Scope；
- Non-goals；
- 依赖；
- 风险；
- 验收标准；
- 决策；
- Owner。

---

## 23.6 用高优先级绕过 Ready

表现：

```text
这件事很急，所以直接开始做。
```

结果通常是：

- 中途停工；
- 需求冲突；
- 大量返工；
- 无法验收；
- 紧急程度反而被放大。

正确做法：

> 优先解决使它 Not Ready 的问题，而不是跳过 Ready。

---

# 24. 例外

可以简化 Ready 流程的情况：

- 影响范围极小；
- 风险低；
- 容易回滚；
- 行为清楚；
- 不涉及数据和迁移；
- 不影响核心架构；
- 验证简单；
- 不依赖外部系统。

即使简化，也至少需要：

```text
明确目标
明确影响范围
明确验证方式
明确回滚方式
```

---

# 25. 最终判断标准

一个功能可以被认定为 Ready，当：

> 它的目标、价值、范围、核心行为、依赖、风险、责任和验收方式已经足够清楚，并且剩余不确定性不会阻碍下一阶段，也不会造成高概率的重大返工。

Ready 的本质不是“所有问题都解决了”。

Ready 的本质是：

> 现在开始，是一个合理、可控、可验证的决定。
