# Review Checklist（评审检查清单）

> 状态：草案  
> 适用范围：通用的软件与游戏开发项目  
> 目的：定义设计、代码、数据、内容、资产、测试和发布在进入下一阶段前，应进行的通用评审检查。

---

## 1. 文档目的

Review Checklist 用于帮助项目在评审阶段系统性检查：

- 目标是否仍然清楚；
- 方案是否满足需求；
- 实现是否符合架构和规范；
- 是否引入隐藏依赖；
- 是否遗漏失败流程与边界情况；
- 是否存在数据、性能、安全或兼容性风险；
- 是否满足测试和发布要求；
- 是否留下未记录的技术债务；
- 是否可以安全进入下一阶段。

评审的目标不是寻找完美方案，也不是用个人偏好阻止交付。

评审的目标是：

> 在成本仍然可控时，尽早发现会导致返工、缺陷、维护困难或发布风险的问题。

---

## 2. 适用范围

本检查清单可以用于：

- Feature Review；
- Design Review；
- Architecture Review；
- Code Review；
- UI / UX Review；
- Content Review；
- Asset Review；
- Data Review；
- Security Review；
- Test Review；
- Release Review；
- Process Review。

不同类型的评审，应选择相关部分使用，而不是机械执行全部检查项。

---

## 3. Review 与 Testing 的区别

Review 和 Testing 关注点不同。

| 类型 | 核心问题 |
|---|---|
| Review | 方案和实现是否合理、完整、可维护 |
| Testing | 功能在真实条件下是否正确运行 |

Review 主要依赖：

- 文档；
- 设计；
- 代码；
- 数据结构；
- 配置；
- 资产；
- 测试计划；
- 变更记录。

Testing 主要依赖：

- 可运行版本；
- 输入；
- 输出；
- 实际行为；
- 测试证据。

二者不能互相替代。

代码评审通过，不代表功能一定正确。

功能测试通过，也不代表实现一定可维护。

---

## 4. 评审设计原则

### 4.1 评审应基于目标和标准

评审不应变成：

```text
我不喜欢这种写法
我会用另一种方式
这个看起来不够优雅
```

评审意见应尽量关联：

- Feature Goal；
- Scope；
- Acceptance Criteria；
- Architecture；
- Coding Standard；
- Security Requirement；
- Performance Target；
- Definition of Done。

---

### 4.2 先判断正确性，再讨论偏好

评审优先级应为：

```text
Correctness
→ Risk
→ Maintainability
→ Consistency
→ Style
```

先确认：

- 是否正确；
- 是否安全；
- 是否会破坏系统；
- 是否可维护。

最后再讨论：

- 命名偏好；
- 格式；
- 局部写法；
- 非关键风格。

---

### 4.3 评审应尽早发生

越早发现问题，修复成本越低。

推荐在以下阶段进行评审：

```text
Clarify：目标和范围评审
Design：方案评审
Prototype：结论评审
Plan：任务拆分评审
Build：实现评审
Validate：测试结果评审
Done：完成评审
Release：发布评审
```

---

### 4.4 评审不应一次解决所有问题

大型变更不应等到最后一次性评审。

更合理：

```text
先评审方向
再评审设计
再评审关键接口
再评审实现
最后评审交付
```

---

### 4.5 评审意见应可执行

不合格：

```text
感觉不太好
这里有问题
可以再优化
代码不够优雅
```

合格：

```text
该模块直接依赖全局状态，导致独立测试困难。建议通过构造参数或初始化方法注入依赖。
```

---

### 4.6 评审应区分阻塞与建议

建议使用以下等级：

```text
Blocker
Required
Suggestion
Question
Nit
Praise
```

对应：

- Blocker：必须解决，否则不能合并或进入下一阶段；
- Required：重要问题，应解决；
- Suggestion：建议改进，不阻塞；
- Question：需要澄清；
- Nit：小型风格或细节；
- Praise：值得保留的优秀做法。

## 5. 通用评审准备

正式评审前，应确认：

```markdown
- [ ] 评审对象明确；
- [ ] 目标明确；
- [ ] Scope 明确；
- [ ] Non-goals 明确；
- [ ] 相关文档可访问；
- [ ] 变更范围可理解；
- [ ] 关键风险已标记；
- [ ] 评审人具备必要上下文；
- [ ] 评审标准明确；
- [ ] 目标阶段明确；
```

---

# 6. Goal Review：目标评审

## 6.1 核心问题

- 当前方案是否仍然解决原问题；
- 是否偏离 Feature Goal；
- 是否引入了不必要的附加目标；
- 是否存在更小、更直接的解决方案；
- 是否有目标被实现细节替代。

## 6.2 检查项

```markdown
- [ ] 问题定义仍然成立；
- [ ] 目标用户或场景明确；
- [ ] 预期价值明确；
- [ ] 方案与目标一致；
- [ ] 没有无关 Scope 膨胀；
- [ ] Non-goals 仍然成立；
- [ ] 当前方案不是只为了技术便利；
```

---

# 7. Scope Review：范围评审

## 7.1 需要检查

- 是否完成批准范围；
- 是否新增未批准能力；
- 是否有关键内容被悄悄删除；
- 是否将 Follow-up 混入当前工作；
- 是否存在“顺便做一下”；
- 是否需要重新进行 Ready Review。

## 7.2 检查项

```markdown
- [ ] Scope 与批准内容一致；
- [ ] Non-goals 未被无意纳入；
- [ ] 范围变化已记录；
- [ ] 新增需求已拆分；
- [ ] 不存在隐藏的未完成项；
- [ ] 变更规模仍符合 Workflow Track；
```

---

# 8. Requirement Review：需求评审

## 8.1 核心问题

- 需求是否可理解；
- 是否存在互相冲突的规则；
- 是否遗漏失败状态；
- 是否遗漏边界情况；
- 是否可测试；
- 是否存在模糊词。

## 8.2 常见模糊词

```text
快速
友好
稳定
智能
高性能
合理
适当
无缝
简单
正常工作
```

这些词需要转化为可观察标准。

## 8.3 检查项

```markdown
- [ ] 主流程明确；
- [ ] 成功状态明确；
- [ ] 失败状态明确；
- [ ] 边界条件明确；
- [ ] 重复操作行为明确；
- [ ] 中断行为明确；
- [ ] 权限行为明确；
- [ ] 验收标准可测试；
- [ ] 不存在互相冲突的需求；
```

---

# 9. Design Review：设计评审

## 9.1 设计评审目标

确认方案：

- 能解决目标问题；
- 与现有系统一致；
- 可实现；
- 可测试；
- 可维护；
- 风险可接受；
- 不会制造不必要复杂度。

## 9.2 设计完整度

```markdown
- [ ] 主要流程已覆盖；
- [ ] 状态已定义；
- [ ] 异常流程已定义；
- [ ] 依赖已识别；
- [ ] 数据影响已识别；
- [ ] 平台影响已识别；
- [ ] 性能影响已识别；
- [ ] 安全影响已识别；
- [ ] 测试方式已考虑；
```

---

## 9.3 方案合理性

```markdown
- [ ] 方案直接解决问题；
- [ ] 没有过度设计；
- [ ] 没有不必要的新抽象；
- [ ] 没有重复已有能力；
- [ ] 复杂度与价值匹配；
- [ ] 关键决策有依据；
- [ ] 替代方案已被合理考虑；
```

---

## 9.4 可扩展性

```markdown
- [ ] 当前设计能支持已知近期需求；
- [ ] 没有为了未知未来过度抽象；
- [ ] 扩展点清楚；
- [ ] 数据结构可演进；
- [ ] 公共接口稳定；
- [ ] 未来迁移成本可接受；
```

---

# 10. Architecture Review：架构评审

## 10.1 模块边界

```markdown
- [ ] 模块职责清楚；
- [ ] 单一模块没有承担过多责任；
- [ ] 公共接口明确；
- [ ] 内部实现没有泄漏；
- [ ] 模块之间依赖方向合理；
- [ ] 没有循环依赖；
```

---

## 10.2 依赖管理

```markdown
- [ ] 依赖已显式声明；
- [ ] 没有隐藏全局依赖；
- [ ] 没有不必要的单例依赖；
- [ ] 外部依赖可替换或隔离；
- [ ] 测试替身可注入；
- [ ] 依赖生命周期明确；
```

---

## 10.3 数据流

```markdown
- [ ] 数据来源清楚；
- [ ] 数据所有权清楚；
- [ ] 单一事实来源明确；
- [ ] 状态同步方式合理；
- [ ] 缓存失效策略明确；
- [ ] 异步更新不会产生竞态；
- [ ] 数据流可追踪；
```

---

## 10.4 生命周期

```markdown
- [ ] 对象或服务生命周期清楚；
- [ ] 初始化顺序安全；
- [ ] 释放和清理明确；
- [ ] 重启或重载行为正确；
- [ ] 长期对象不会持有失效引用；
- [ ] 资源不会意外泄漏；
```

---

## 10.5 错误边界

```markdown
- [ ] 错误不会无限传播；
- [ ] 模块能处理可预期错误；
- [ ] 不可恢复错误有升级路径；
- [ ] 错误上下文不会丢失；
- [ ] 降级方式明确；
```

---

---

## 11.3 可维护性

```markdown
- [ ] 重复逻辑已合理处理；
- [ ] 没有过度抽象；
- [ ] 模块边界清楚；
- [ ] 修改范围可控；
- [ ] 公共接口稳定；
- [ ] 技术债务已记录；
- [ ] 临时代码已标记；
```

---

## 11.4 复杂度

```markdown
- [ ] 函数规模合理；
- [ ] 类职责合理；
- [ ] 条件分支可理解；
- [ ] 状态机没有隐式跳转；
- [ ] 没有过度使用继承；
- [ ] 没有过度使用全局状态；
- [ ] 没有不必要的反射或动态调用；
```

---

## 11.5 错误处理

```markdown
- [ ] 可预期错误有处理；
- [ ] 错误不会被静默吞掉；
- [ ] 错误信息有上下文；
- [ ] 重试策略合理；
- [ ] 失败不会造成脏状态；
- [ ] 不会因异常重复执行不可逆操作；
```

---

## 11.6 边界与空值

```markdown
- [ ] Null / None / 空值已考虑；
- [ ] 空集合已考虑；
- [ ] 最小值和最大值已考虑；
- [ ] 越界已处理；
- [ ] 类型转换安全；
- [ ] 默认值合理；
```

---

## 11.7 并发与异步

```markdown
- [ ] 共享状态访问安全；
- [ ] 锁粒度合理；
- [ ] 不存在明显死锁；
- [ ] 取消行为正确；
- [ ] 超时行为正确；
- [ ] 回调不会重复执行；
- [ ] 旧请求结果不会覆盖新状态；
```

---

## 11.8 资源与性能

```markdown
- [ ] 没有明显重复计算；
- [ ] 没有高频不必要分配；
- [ ] 缓存使用合理；
- [ ] I/O 时机合理；
- [ ] 大资源不会同步阻塞关键路径；
- [ ] 资源能够释放；
- [ ] 性能风险已测量而非猜测；
```

---

## 11.9 安全

```markdown
- [ ] 输入已校验；
- [ ] 权限检查位于可信边界；
- [ ] 敏感数据未写入日志；
- [ ] 密钥未硬编码；
- [ ] 文件路径安全；
- [ ] 外部输入不会直接执行；
- [ ] 错误信息不泄露敏感内容；
```

---

## 11.10 测试

```markdown
- [ ] 新逻辑有测试；
- [ ] Bug 修复有回归测试；
- [ ] 测试覆盖关键分支；
- [ ] 测试不依赖脆弱实现细节；
- [ ] 测试数据隔离；
- [ ] 测试可重复；
- [ ] 没有明显 Flaky Test；
```

# 21. Review 输出等级

建议使用以下评审结果：

```text
Approved
Approved with Suggestions
Approved with Conditions
Changes Required
Blocked
Rejected
```

---

## 21.1 Approved

没有阻塞问题，可以进入下一阶段。

---

## 21.2 Approved with Suggestions

存在非阻塞建议，可以后续改进。

---

## 21.3 Approved with Conditions

可以继续，但必须满足明确条件。

条件必须：

- 可执行；
- 有 Owner；
- 有截止时间或触发条件；
- 不影响关键路径；
- 有复核方式。

---

## 21.4 Changes Required

存在必须修改的问题。

---

## 21.5 Blocked

缺少关键信息、依赖或决策，无法继续评审。

---

## 21.6 Rejected

当前方案不适合继续，应重新设计或取消。

# 26. 常见失败模式

## 26.1 评审变成风格争论

表现：

- 大量讨论命名和格式；
- 忽略正确性和风险；
- 评审时间被低价值细节占用。

改进：

- 使用自动格式化；
- 使用静态检查；
- 明确优先级；
- 先处理 Blocker 和 Required。

---

## 26.2 评审太晚

表现：

- 完整实现后才发现方向错误；
- 修改成本巨大；
- 团队产生对立；
- 交付延期。

改进：

- 提前评审目标、设计和接口；
- 大型变更分阶段评审。

---

## 26.3 评审范围过大

表现：

- 一次提交包含多个 Feature；
- Reviewer 无法建立上下文；
- 问题被遗漏；
- 评审时间过长。

改进：

- 拆分 PR；
- 拆分设计；
- 使用 Vertical Slice；
- 单次评审聚焦一个目标。

---

## 26.4 只找问题，不确认优点

结果：

- 团队只感受到否定；
- 好做法无法沉淀；
- 评审关系恶化。

改进：

- 使用 Praise；
- 指出值得复用的模式；
- 记录优秀实践。

---

## 26.5 意见没有等级

表现：

- Reviewer 的每条建议都像必须修改；
- Author 不知道哪些阻塞；
- 反复沟通。

改进：

- 使用 Blocker / Required / Suggestion / Nit。

---

## 26.6 意见不可执行

表现：

```text
这里不好
需要优化
感觉不对
```

改进：

- 说明原因；
- 说明影响；
- 提供方向；
- 标记是否阻塞。

---

## 26.7 Author 没有提供上下文

表现：

- Reviewer 只能猜目标；
- 评审变成重新理解需求；
- 问题反复出现。

改进：

- 提供 Why；
- 提供 Scope；
- 提供风险；
- 提供测试证据；
- 提供截图或 Demo。

---

## 26.8 只评审代码，不评审交付

表现：

- 代码质量不错；
- 但没有集成；
- 没有失败状态；
- 没有正式构建；
- 没有迁移。

改进：

- 使用完整 Review Checklist；
- 对照 Definition of Done。

---

## 26.9 Reviewer 直接重写方案

表现：

- Reviewer 用个人偏好替代目标；
- Author 没有参与决策；
- 讨论失去边界。

改进：

- 指出问题和约束；
- 允许 Author 选择解决方式；
- 只有在安全或架构红线时强制方案。

---

## 26.10 评审意见长期不关闭

表现：

- 旧意见失效；
- PR 长期停滞；
- 条件无人跟踪。

改进：

- 每条 Required 有 Owner；
- 明确复核方式；
- 过期后重新评估；
- 不再适用则关闭并记录原因。

---

# 27. Reviewer 的责任

Reviewer 应：

- 理解目标；
- 识别风险；
- 提供可执行意见；
- 区分阻塞与建议；
- 避免个人偏好主导；
- 及时反馈；
- 对关键通过结论负责；
- 不批准自己无法理解的变更。

Reviewer 不应：

- 只检查格式；
- 无理由扩大 Scope；
- 要求无关重构；
- 用模糊意见阻塞；
- 代替 Owner 做所有实现决策。

---

# 28. Author 的责任

Author 应：

- 提供上下文；
- 控制变更范围；
- 提供测试证据；
- 说明风险；
- 回应评审意见；
- 记录范围变化；
- 不隐藏已知问题；
- 完成必要修正；
- 明确哪些意见不接受及原因。

---

# 29. 评审完成定义

一次评审可以结束，当：

- 评审范围已覆盖；
- Blocker 已关闭；
- Required 已关闭或正式接受；
- 条件已记录；
- 风险已明确；
- 评审结果已记录；
- 下一阶段明确；
- 相关 Owner 已确认；
- 无未处理的关键问题。

---

# 30. 最终原则

Review Checklist 的本质不是：

> 尽可能多地找问题。

它的本质是：

> 在合适的时间，用明确标准检查目标、方案、实现和交付风险，并通过可执行反馈提高最终质量。

一个有效的评审过程应做到：

- 目标驱动；
- 风险驱动；
- 尽早发生；
- 分阶段进行；
- 结论明确；
- 意见可执行；
- 阻塞与建议分离；
- 证据可追踪；
- 不被个人偏好主导；
- 能够支持下一阶段决策。
