# 功能生命周期（Feature Lifecycle）

> 状态：草案  
> 适用范围：通用的软件与游戏开发项目  
> 目的：定义一个功能如何从初始想法推进到正式发布，并在发布后完成评估与学习。

---

## 1. 文档目的

功能生命周期用于定义一个功能从提出、澄清、评估、设计、实现、验证、发布到复盘的完整工作流。

它的核心目标是：

- 减少因需求不清造成的返工；
- 将探索阶段与正式开发投入分离；
- 明确每个阶段的质量要求；
- 区分“实现完成”和“可以发布”；
- 为阶段流转提供清晰的进入条件与退出条件；
- 同时支持轻量流程和完整流程；
- 沉淀重要决策、关键假设与经验结论。

本流程管理的是一个**功能（Feature）**的演进，而不是每一个具体开发任务的执行状态。

一个功能通常是能够独立产生用户价值或产品价值的工作单元，例如：

- 一个新的玩法系统；
- 一条新的用户操作流程；
- 一项重要的 UI 能力；
- 一个新的内容系统；
- 一项面向用户的技术能力。

而以下工作通常只是功能下的执行任务，不需要独立走完整生命周期：

- 新增一个按钮；
- 重命名一个函数；
- 替换一个图标；
- 调整一段文案；
- 修改一个局部参数。

---

## 2. 设计原则

功能生命周期基于以下原则设计。

### 2.1 逐步降低不确定性

生命周期的每个阶段，应重点解决一种不同类型的不确定性：

| 不确定性类型 | 核心问题 |
|---|---|
| 价值不确定性 | 这件事值得做吗？ |
| 范围不确定性 | 我们到底要做什么？ |
| 体验不确定性 | 用户能否理解、使用并获得价值？ |
| 技术不确定性 | 这件事能否安全、稳定地实现？ |
| 交付不确定性 | 它能否正确接入完整产品？ |
| 质量不确定性 | 它是否达到可发布标准？ |
| 结果不确定性 | 它是否达成了原本目标？ |

流程不应因为“已经做了一段时间”就自动进入下一阶段。

只有当当前阶段的主要不确定性被降低到可接受程度后，功能才应继续推进。

---

### 2.2 使用 Gate 控制投入与质量

整个生命周期中有两个最重要的 Gate：

- **Ready Gate**：决定是否正式投入开发；
- **Done Gate**：决定功能是否达到可发布质量。

Gate 不是普通工作阶段，而是明确的决策点。

---

### 2.3 允许回退与迭代

生命周期不是严格的瀑布流程。

当新信息推翻旧假设时，功能可以退回前一个阶段。

例如：

```text
Prototype → Clarify
Build → Design
Validation → Build
Release → Follow-up Iteration
```

发生重大回退时，应记录：

- 新发现了什么；
- 哪个假设被证明是错误的；
- 哪些范围或规则发生了变化；
- 验收标准是否需要调整；
- 下一步准备如何处理。

---

### 2.4 区分功能阶段与任务状态

功能阶段表示这个功能在产品和交付层面所处的位置。

例如：

```text
Design
Build
Validation
```

任务状态表示某个具体执行任务的当前状态。

例如：

```text
Todo
In Progress
Blocked
Done
```

一个功能可能处于 Build 阶段，但其下：

- 一部分任务已经 Done；
- 一部分任务正在 In Progress；
- 一部分任务处于 Blocked。

功能阶段与任务状态不应混成同一套状态系统。

---

### 2.5 根据风险选择流程长度

不是所有改动都必须执行完整生命周期。

流程复杂度应取决于：

- 影响范围；
- 不确定性；
- 可逆性；
- 技术风险；
- 涉及系统数量；
- 发布风险；
- 出错成本。

---

## 3. 生命周期总览

```text
CAPTURE
   ↓
CLARIFY
   ↓
ASSESS
   ↓
[READY GATE]
   ↓
DESIGN
   ↓
PROTOTYPE（可选）
   ↓
PLAN
   ↓
BUILD
   ↓
INTEGRATE
   ↓
VALIDATE
   ↓
POLISH
   ↓
[DONE GATE]
   ↓
RELEASE
   ↓
OBSERVE & LEARN
   ↓
CLOSE / ITERATE
```

整个生命周期可以归纳为四个大的区段：

```text
Discovery
→ Commitment
→ Development & Quality
→ Delivery & Learning
```

对应中文可理解为：

```text
探索
→ 承诺投入
→ 开发与质量
→ 交付与学习
```

---

# 4. Discovery：探索阶段

Discovery 用于判断：

- 这个功能是否值得做；
- 它解决什么问题；
- 它的范围是什么。

探索阶段包括：

- Capture；
- Clarify；
- Assess。

---

## 4.1 Capture：记录想法

### 目标

记录一个想法或问题，但不代表已经承诺开发。

### 常见输入

- 用户反馈；
- 产品想法；
- 设计机会；
- 技术机会；
- 重复出现的问题；
- 市场或研究发现；
- 团队内部观察。

### 最低记录要求

至少应包括：

```text
功能名称
一句话描述
问题或机会
预期用户价值或产品价值
来源
可能影响的范围
```

### 推荐模板

```markdown
## 概要

## 问题

## 预期价值

## 来源

## 可能影响
```

### 进入条件

- 发现了一个可能有价值的问题、想法或机会。

### 退出条件

- 想法已经被清晰记录；
- 重复或明显无价值内容已被过滤；
- 可以进入澄清阶段。

### 可能结果

```text
进入 Clarify
与其他功能合并
归档
拒绝
```

---

## 4.2 Clarify：澄清功能

### 目标

把一个模糊想法转化为可以讨论、评估和限定范围的功能。

### 需要回答的问题

- 谁会使用这个功能？
- 它解决什么问题？
- 它在什么场景中使用？
- 主要用户流程是什么？
- 输入是什么？
- 输出是什么？
- 成功状态是什么？
- 失败状态是什么？
- 涉及哪些系统或团队？
- 哪些内容包含在本次范围内？
- 哪些内容明确不做？
- 哪些假设尚未被验证？

### 必要输出

```text
问题定义
目标用户或使用场景
功能目标
初步范围
非目标
主要流程
依赖
已知风险
初步验收方向
```

### Scope 与 Non-goals

Non-goals 非常重要。

它用于防止功能在开发过程中无意识膨胀。

示例：

```markdown
## 范围

- 用户可以创建并保存自定义预设；
- 用户可以编辑和删除已有预设。

## 非目标

- 预设分享；
- 云同步；
- 团队协作；
- 公共市场。
```

### 进入条件

- 想法已经被记录；
- 该问题或机会值得进一步调查。

### 退出条件

- 问题定义清楚；
- 范围和非目标清楚；
- 主流程可理解；
- 关键依赖和主要风险已暴露；
- 可以进行有效评估。

### 可能结果

```text
进入 Assess
退回 Capture
与其他功能合并
延后
拒绝
```

---

## 4.3 Assess：评估是否投入

### 目标

决定这个功能是否值得获得正式开发资源。

### 评估维度

- 用户价值；
- 产品价值；
- 战略一致性；
- 紧急程度；
- 实现成本；
- 设计复杂度；
- 技术风险；
- 运营风险；
- 依赖是否准备好；
- 可逆性；
- 长期维护成本；
- 机会成本；
- 是否可以先做一个更小版本。

### 常见决策

```text
Commit
Prototype First
Reduce Scope
Defer
Reject
Merge with Another Feature
Request More Research
```

### 进入条件

- 功能已经被充分澄清；
- 主要范围边界已明确；
- 预期价值可以被评估。

### 退出条件

- 已经做出决定；
- 决策理由已记录；
- 下一步明确。

---

# 5. Ready Gate：正式投入门槛

Ready Gate 表示功能从探索阶段进入正式开发阶段。

通过 Ready Gate 的功能，被认为已经具备进入计划和实施的条件。

---

## 5.1 Ready 条件

一个功能进入 Ready，通常需要满足：

- 问题定义清楚；
- 预期价值清楚；
- 功能范围明确；
- 非目标明确；
- 主要依赖已知；
- 关键风险已识别；
- 目标结果可以被理解；
- 初步验收标准存在；
- 不存在会阻塞开发的关键未决问题；
- 功能符合当前优先级或里程碑。

---

## 5.2 Ready 不代表什么

Ready 不代表：

- 所有实现细节都已经确定；
- 后续绝对不会调整设计；
- 所有风险都已经消失；
- 所有任务都已经精确估算；
- 无论发生什么都必须完成。

Ready 只代表：

> 当前信息已经足够充分，项目可以合理地承诺正式投入开发。

---

## 5.3 Gate 结果

```text
Approved
Approved with Conditions
Needs Clarification
Needs Prototype
Deferred
Rejected
```

---

# 6. Development & Quality：开发与质量阶段

这一部分负责把一个已承诺的功能，转化为已集成、已验证、已达到发布质量的产品能力。

包括：

- Design；
- Prototype；
- Plan；
- Build；
- Integrate；
- Validate；
- Polish。

---

## 6.1 Design：设计方案

### 目标

形成一套足以支持正式实现的解决方案。

### 设计内容

根据功能不同，设计可能包含以下部分。

#### 产品或玩法设计

- 规则；
- 行为；
- 状态；
- 限制；
- 成功条件；
- 失败条件；
- 与已有系统的关系。

#### 用户体验设计

- 用户流程；
- 导航；
- 操作反馈；
- 空状态；
- 错误状态；
- 禁用状态；
- 可访问性。

#### 技术设计

- 系统边界；
- 数据结构；
- API；
- 依赖；
- 持久化；
- 数据迁移；
- 性能；
- 安全性；
- 异常处理。

#### 内容与资产设计

- 视觉资产；
- 音频；
- 文案；
- 动画；
- 本地化；
- 内容规格。

#### 验证设计

- 验收标准；
- 核心场景；
- 测试策略；
- 可衡量的成功条件。

### 进入条件

- 功能已通过 Ready Gate。

### 退出条件

- 方案整体一致；
- 主要行为已定义；
- 关键技术风险已处理；
- 验收标准足够具体；
- 可以开始开发，而不需要反复重新讨论基础问题。

### 可能结果

```text
进入 Prototype
进入 Plan
退回 Clarify
退回 Assess
取消
```

---

## 6.2 Prototype：原型验证

### 目标

在完整实现前，验证高风险假设。

Prototype 是可选阶段。

### 适用场景

当存在以下情况时，建议先做原型：

- 核心体验不确定；
- 多种方案无法通过文档判断；
- 性能可行性不明确；
- 技术集成风险较高；
- 视觉或交互模式未经验证；
- 正式实现后的返工成本很高。

### 原型可接受的特征

原型可以使用：

- 临时 UI；
- 占位资源；
- 硬编码数据；
- 简化逻辑；
- 独立测试环境；
- 不完整异常处理；
- 可丢弃代码。

原型默认不应被视为正式实现。

### 原型应回答的问题

例如：

- 用户能否理解这套交互？
- 这个概念是否有价值？
- 是否能达到性能要求？
- 技术路线是否可行？
- 内容是否清晰可读？
- 流程是否过于复杂？
- 多个方案中哪一个更好？

### 必要输出

```text
验证的问题
使用的方法
结果
发现的风险
最终决策
新增约束
建议下一步
```

### 退出条件

- 目标假设已被验证；
- 结果已记录；
- 已经做出明确决策。

### 可能结果

```text
Validated
Continue with Constraints
Run Another Experiment
Change Direction
Return to Design
Reject Feature
```

---

## 6.3 Plan：实施计划

### 目标

把已批准的设计转换为可执行的交付计划。

Plan 应是一个短阶段，而不是漫长的文档阶段。

### 计划活动

- 拆分任务；
- 识别依赖；
- 明确实现顺序；
- 识别可并行工作；
- 识别技术 Spike；
- 识别资产需求；
- 识别内容需求；
- 识别测试工作；
- 识别集成工作；
- 定义最小可用版本；
- 识别发布约束。

### 任务类型

一个功能可能包含：

```text
设计任务
工程任务
UI 任务
资产任务
内容任务
数据任务
测试任务
集成任务
文档任务
发布任务
```

### 好任务的特征

一个好的任务应尽量：

- 聚焦单一目标；
- 可以验证；
- 规模适中；
- 输出明确；
- 依赖清楚；
- 能关联到验收标准。

### 退出条件

- 实施工作已被合理拆分；
- 依赖已暴露；
- 主要风险已有负责人或缓解方案；
- 执行顺序明确；
- 第一批工作可以直接开始。

---

## 6.4 Build：正式实现

### 目标

根据已批准的设计和计划完成正式实现。

### Build 包括

- 代码；
- 数据；
- UI；
- 内容；
- 视觉资产；
- 音频；
- 配置；
- 工具；
- 测试；
- 数据迁移；
- 文档更新。

### 实现原则

- 遵循现有架构和规范；
- 避免不必要的全局依赖；
- 区分临时代码与正式代码；
- 保持范围与已批准定义一致；
- 范围变化必须显式记录；
- 在实现过程中持续测试；
- 关键决策变化时同步更新功能记录。

### 范围变更控制

以下小变动通常不需要回退生命周期：

- 局部实现细节；
- 不影响目标的小型重构；
- 不影响验收标准的技术调整。

以下重大变化通常应回退到前一阶段：

- 功能目标变化；
- 主用户流程变化；
- 验收标准变化；
- 技术架构变化；
- 发布风险显著增加；
- Scope 或 Non-goals 变化；
- 成本显著超出原预期。

### 退出条件

- 计划中的主要组成部分已完成；
- 核心实现任务已完成；
- 局部和组件级检查通过；
- 已知阻塞问题已解决或被接受；
- 可以进入完整产品集成。

---

## 6.5 Integrate：系统集成

### 目标

把功能接入完整产品，并验证系统级行为。

一个功能在独立环境中可以运行，并不代表它已经完成集成。

### 集成范围

- 主流程；
- 已有系统；
- 共享数据；
- 持久化；
- 认证或权限；
- 页面或场景切换；
- UI 状态；
- 分析或埋点；
- 本地化；
- 音频与视觉系统；
- 平台行为；
- 部署或配置；
- 向后兼容。

### 需要回答的问题

- 功能是否能在真实产品流程中运行？
- 是否破坏了已有行为？
- 是否正确使用共享状态？
- 持久化是否正常？
- 重启、刷新、切换后是否正常？
- 是否能处理暂停、离线、错误和降级状态？
- 是否在接近生产环境的构建中工作？

### 退出条件

- 功能已进入预期产品流程；
- 所有必要依赖已接入；
- 系统行为足够稳定，可以正式验证；
- 不存在关键集成阻塞。

---

## 6.6 Validate：功能验证

### 目标

确认功能正确，并满足验收标准。

Validation 回答的是：

> 功能是否按照定义正确工作？

### 验证范围

- 正常流程；
- 失败流程；
- 边界条件；
- 重复操作；
- 非法输入；
- 状态切换；
- 持久化；
- 数据迁移；
- 兼容性；
- 权限；
- 性能；
- 可访问性；
- 本地化；
- 接近生产环境的构建；
- 回归问题。

### 验收标准要求

验收标准应当：

- 具体；
- 可观察；
- 可测试；
- 与用户行为或系统行为相关；
- 在验证前已经写好。

示例：

```markdown
## 验收标准

- 合法用户可以完成主要流程；
- 非法输入不会修改已保存数据；
- 重复提交不会产生重复记录；
- 状态在重启后仍然保留；
- 旧版本数据可以被正确加载；
- 功能在正式构建中可以运行。
```

### 验证输出

```text
通过 / 失败结果
发现的缺陷
已知限制
回归问题
性能结果
剩余风险
```

### 退出条件

- 所有关键验收标准通过；
- 阻塞问题已解决；
- 回归风险可接受；
- 已知限制已记录；
- 可以进入 Polish 或 Done Gate。

---

## 6.7 Polish：体验完善

### 目标

把一个“功能正确”的功能，提高到“可以交付给用户”的质量。

Polish 回答的是：

> 这个功能是否已经足够清晰、一致、可用和完整？

### Polish 范围

- 操作反馈；
- 动画；
- 音效；
- 视觉层级；
- 加载状态；
- 空状态；
- 禁用状态；
- 错误提示；
- 文案；
- 可访问性；
- 响应式布局；
- 易用性；
- 节奏与时机；
- 与现有产品体验的一致性；
- 清理临时资源和调试 UI。

### Polish 不应做什么

Polish 不应用来掩盖根本性的设计问题。

如果功能的使用逻辑或体验方向本身有问题，应退回 Design 或 Clarify，而不是继续装饰当前方案。

### 退出条件

- 功能达到目标体验标准；
- 临时表现已清理；
- 所有必要状态和反馈完整；
- 易用性问题在可接受范围内；
- 可以进入 Done Gate。

---

# 7. Done Gate：完成门槛

Done Gate 用于确认功能是否达到可发布质量。

---

## 7.1 Done 条件

一个功能可以通过 Done Gate，通常需要满足：

- 验收标准通过；
- 集成完成；
- 不存在发布阻塞问题；
- 必要测试已完成；
- 必要 Polish 已完成；
- 持久化和迁移行为正确；
- 接近生产环境的构建验证通过；
- 临时实现已清理或被记录；
- 已知技术债务已登记；
- 相关文档已更新；
- 运维和维护影响已明确；
- 必要时已有回滚或禁用方案。

---

## 7.2 Done 不等于 Released

一个功能可以已经 Done，但仍未 Released。

可能原因包括：

- 等待发布窗口；
- 等待其他功能；
- 等待完整回归测试；
- 等待审批；
- 等待打包；
- 等待平台认证；
- 等待部署协调。

---

## 7.3 Gate 结果

```text
Done
Done with Accepted Limitations
Needs Fixes
Needs More Validation
Return to Design
Cancelled
```

---

# 8. Delivery & Learning：交付与学习

这一部分负责把完成的功能交付给用户，并评估真实结果。

包括：

- Release；
- Observe & Learn；
- Close or Iterate。

---

## 8.1 Release：发布

### 目标

把功能交付到目标用户或目标环境。

### 发布活动

- 确认目标版本；
- 确认发布说明；
- 完成回归测试；
- 验证迁移；
- 构建正式产物；
- 发布或部署；
- 验证生产配置；
- 监控发布过程；
- 确认回滚路径；
- 执行发布后冒烟测试。

### 发布条件

- 已通过 Done Gate；
- 发布包已获批准；
- 运维依赖已准备好；
- 生产验证步骤已定义；
- 必要时有回滚或禁用方案。

### Released 的定义

Released 表示：

> 功能已经在目标用户实际使用的环境中可用。

Released 不等于：

- 已合并到主分支；
- 已在测试环境完成；
- 已在本地构建可用；
- 已在任务工具中标记 Done。

---

## 8.2 Observe & Learn：观察与学习

### 目标

评估功能是否真正达成预期结果。

### 需要回答的问题

- 用户是否发现这个功能？
- 用户是否理解它？
- 用户是否按照预期使用？
- 它是否解决了原问题？
- 它是否制造了新问题？
- 它是否影响性能、稳定性或平衡？
- 实现成本是否值得？
- 哪些假设是正确的？
- 哪些假设是错误的？

### 可使用的信息来源

- 数据分析；
- 埋点；
- 用户反馈；
- 客服问题；
- Bug 报告；
- 可用性测试；
- 运营指标；
- 性能数据；
- 手工观察；
- 内部复盘。

### 输出

```text
结果评估
非预期行为
后续问题
调优机会
扩展机会
回滚建议
流程经验
```

---

## 8.3 Close or Iterate：关闭或继续迭代

### 目标

根据发布后的信息，决定该功能下一步如何处理。

### 可能结果

```text
Close
Tune
Expand
Rework
Rollback
Deprecate
Create Follow-up Feature
```

### Close 条件

一个功能可以关闭，当：

- 预期目标已经达成或结果已被接受；
- 没有关键遗留问题；
- 经验已被记录；
- 剩余工作已单独进入后续项；
- 后续维护责任明确。

功能不应因为“未来还有可能继续改进”而永远保持 In Progress。

后续增强通常应作为新的 Follow-up Feature 管理。

---

# 9. 工作流等级

生命周期可以根据风险采用不同严格程度。

---

## 9.1 Fast Track：快速流程

### 适用范围

- 文案调整；
- 小型视觉修复；
- 明确且低风险的 Bug；
- 小型配置修改；
- 单个资源替换；
- 影响范围有限且易于回滚的改动。

### 流程

```text
Clarify
→ Build
→ Verify
→ Done
→ Release
```

### 最低要求

即使使用 Fast Track，也必须具备：

- 明确目标；
- 已知影响范围；
- 基础验证；
- 可追踪的修改记录。

---

## 9.2 Standard Track：标准流程

### 适用范围

- 普通产品功能；
- 小型系统扩展；
- 常见 UI 流程；
- 新内容类型；
- 中等集成复杂度功能。

### 流程

```text
Capture
→ Clarify
→ Assess
→ Ready
→ Design
→ Plan
→ Build
→ Integrate
→ Validate
→ Done
→ Release
→ Learn
```

需要时加入 Prototype。

---

## 9.3 Full Track：完整流程

### 适用范围

- 核心产品系统；
- 大型架构调整；
- 持久化格式变化；
- 安全敏感功能；
- 高风险平台工作；
- 大规模 UI 重构；
- 跨多个系统的功能；
- 难以回滚或代价高昂的决策。

### 流程要求

使用完整生命周期，并包括：

- 正式评估；
- 明确 Ready Gate；
- Prototype 或技术 Spike；
- 完整计划；
- 文档化验证；
- 明确 Done Gate；
- 发布和回滚计划；
- 发布后复盘。

---

# 10. 功能主记录

每个功能应维护一份主要记录，并在整个生命周期中持续更新。

主记录应链接到更详细的设计、技术、资产、测试和发布文档，而不是重复它们。

## 推荐模板

```markdown

> 流程等级：

```

# 11. 阶段流转规则

功能只有在当前阶段退出条件满足后，才能进入下一阶段。

不要为了让状态看起来更健康，而提前推进功能。

---

## 11.1 向前流转

向前流转时应记录：

- 已完成的输出；
- 尚未解决但已接受的风险；
- 当前负责人；
- 下一阶段。

---

## 11.2 向后流转

向后流转时应记录：

- 回退原因；
- 被推翻的假设；
- 受影响范围；
- 更新后的决策；
- 下一步行动。

---

## 11.3 暂停状态

暂停的功能应明确标记为：

```text
Blocked
Deferred
Cancelled
```

不应让一个功能长期停留在活跃阶段，却没有解释。

---

# 12. 例外与阶段跳过

当以下条件全部满足时，可以跳过部分阶段：

- 改动风险低；
- 易于回滚；
- 影响范围小；
- 决策已经足够明确；
- 跳过不会掩盖关键不确定性。

例如：

- 交互和技术路线已经被验证时，可以跳过 Prototype；
- 内部工具可以降低 Polish 要求；
- 紧急 Bug 可以合并 Capture、Clarify 和 Assess；
- 小型改动可以使用轻量 Plan。

跳过主要阶段的原因，应能够从功能记录中直接理解。

---

# 13. 常见失败模式

## 13.1 未澄清就开始 Build

表现：

- 需求反复变化；
- 验收标准很晚才出现；
- 范围持续扩大；
- 技术实现反过来决定产品设计。

预防方式：

- 强制明确 Scope、Non-goals 和 Ready 条件。

---

## 13.2 把 Prototype 当成正式实现

表现：

- 临时硬编码长期存在；
- 实验结构变成正式架构；
- 缺少错误处理；
- 占位资源进入发布版本。

预防方式：

- 原型结束时明确决定：丢弃、重写还是正式化。

---

## 13.3 把 Build 当成 Done

表现：

- 功能只在孤立环境中可用；
- 没有持久化；
- 没有错误状态；
- 没有正式构建测试；
- 没有回归测试；
- UI 状态不完整。

预防方式：

- 保留 Integrate、Validate、Polish 和 Done 的独立含义。

---

## 13.4 无限 Polish

表现：

- 没有明确发布标准；
- 外观调整持续进行但没有明显价值；
- 核心问题被表现优化掩盖。

预防方式：

- 在进入 Polish 前定义目标质量和 Done 条件。

---

## 13.5 在 Validation 阶段重新扩展范围

表现：

- 测试期间不断加入新想法；
- Bug 和增强需求混在一起；
- 发布被无关改进持续拖延。

预防方式：

- 将缺陷和后续增强分开；
- 除非出现关键问题，否则保护已批准范围。

---

## 13.6 发布后立即关闭

表现：

- 没有结果验证；
- 用户困惑反复出现；
- 错误假设没有记录；
- 后续项目重复相同问题。

预防方式：

- 在 Close 前加入 Observe & Learn。

---

# 14. 生命周期指标

指标应服务于改进流程，而不是只追求表面速度。

可参考的指标包括：

- Capture 到 Ready 的时间；
- Ready 到 Done 的时间；
- Ready 后发生重大范围变化的次数；
- 向后流转次数；
- Validation 阶段发现的缺陷数；
- Release 后发现的缺陷数；
- 需要返工的功能比例；
- Prototype 的结论分布；
- 回滚率；
- 验收标准失败率；
- Blocked 时间；
- 预期结果与实际结果的差异。

不要把单一指标作为唯一绩效目标。

例如，只追求缩短生命周期，可能会导致：

- 过早进入 Ready；
- Validation 不充分；
- 发布质量下降。

---

# 15. 总结

功能生命周期基于四个核心思想：

1. **先探索，再承诺投入**  
   Capture、Clarify 和 Assess 用于降低价值和范围不确定性。

2. **使用明确 Gate**  
   Ready 控制开发投入，Done 控制发布质量。

3. **区分实现、集成和验证**  
   Build 完成，不等于功能完整。

4. **发布后继续学习**  
   只有在真实环境中观察结果后，才能真正理解功能价值。

标准生命周期为：

```text
Capture
→ Clarify
→ Assess
→ Ready Gate
→ Design
→ Prototype
→ Plan
→ Build
→ Integrate
→ Validate
→ Polish
→ Done Gate
→ Release
→ Observe & Learn
→ Close or Iterate
```

这套生命周期应始终保持：

- 可调整；
- 风险驱动；
- 证据驱动；
- 阶段清晰；
- 不过度僵化。
