# Documentation Workflow（文档工作流）

> 状态：草案  
> 适用范围：通用的软件、游戏、媒体与数字产品项目  
> 目的：定义项目文档从创建、评审、发布、维护、更新、废弃到归档的完整流程。

---

## 1. 文档目的

Documentation Workflow 用于统一项目中文档的创建、组织、维护和淘汰方式，确保文档能够：

- 解决明确问题；
- 面向明确读者；
- 保持结构清晰；
- 与实际实现一致；
- 具备明确 Owner；
- 能够被搜索和引用；
- 避免内容重复与互相冲突；
- 在变更发生时及时更新；
- 在失效后被正确标记和归档；
- 长期支持设计、开发、测试、发布和维护。

它回答以下问题：

- 什么内容值得写成文档；
- 应选择哪种文档类型；
- 文档应放在哪里；
- 如何命名；
- 如何建立单一事实来源；
- 文档什么时候需要 Review；
- 谁负责维护；
- 如何处理过期内容；
- 文档之间如何建立链接；
- 如何避免多个版本同时存在；
- 什么时候应 Deprecated；
- 什么时候应 Archived；
- 文档是否需要版本号；
- 如何判断文档质量。

文档工作流的核心目标不是“写更多文档”。

它的核心目标是：

> 用最少但足够的文档，稳定保存项目中需要被共享、执行、验证和长期维护的信息。

---

## 2. 什么内容值得文档化

以下内容通常值得写入文档：

- 长期有效的规则；
- 重要设计决策；
- 架构边界；
- 工作流程；
- 接口契约；
- 数据结构；
- 存档与迁移；
- 发布步骤；
- 复杂操作；
- 难以从代码直接理解的背景；
- 关键约束；
- 重复发生的问题；
- 新成员需要知道的信息；
- 事故和复盘结论；
- 外部依赖；
- 授权与来源；
- 需要跨角色协作的信息。

以下内容不一定需要独立文档：

- 一次性临时沟通；
- 很快失效的信息；
- 可以直接从代码和工具中可靠获得的事实；
- 过度细化的实现过程；
- 无明确读者的笔记；
- 与已有文档重复的内容。

---

## 3. 核心原则

### 3.1 文档必须有目的

每份文档都应回答：

```text
为什么存在？
为谁服务？
读者读完后能做什么？
```

如果无法回答，文档可能不值得创建。

---

### 3.2 单一事实来源

同一类核心信息应只有一个主要维护位置。

例如：

```text
引擎版本 → Godot 项目元数据
发布步骤 → Release Workflow
功能完成标准 → Definition of Done
存档格式 → Save Format Documentation
```

其他文档可以引用，但不应复制大量同样内容。

---

### 3.3 链接优先于复制

复制内容容易造成：

- 多份版本不一致；
- 一处更新，其他地方过期；
- 读者不知道哪份正确；
- 维护成本增加。

推荐：

```text
摘要 + 链接
```

而不是重复整段规则。

---

### 3.4 文档与工作流绑定

文档更新应成为工作的一部分。

例如：

```text
Feature 变更
→ 实现更新
→ 测试更新
→ 文档更新
→ Done
```

文档不应长期依赖“以后有空再补”。

---

### 3.5 越接近事实源，越应自动化

例如：

- API 文档从代码生成；
- 配置表从 Schema 生成；
- Changelog 从 Release 记录生成；
- 构建信息由 CI 注入；
- 版本号由工具统一维护。

人工文档适合保存：

- 原因；
- 约束；
- 决策；
- 使用方式；
- 例外；
- 风险；
- 背景。

---

### 3.6 文档应可被找到

文档如果无法被找到，等同于不存在。

每份文档应：

- 放在合理目录；
- 命名清楚；
- 在 README 或索引中可发现；
- 使用统一标签或分类；
- 提供相关链接；
- 避免只存在个人设备。

---

## 4. 文档生命周期

```text
IDENTIFY NEED
  ↓
CHOOSE TYPE
  ↓
DRAFT
  ↓
REVIEW
  ↓
PUBLISH
  ↓
USE
  ↓
MAINTAIN
  ↓
REVIEW FOR FRESHNESS
  ↓
UPDATE / DEPRECATE / ARCHIVE
```

可能回退：

```text
Review → Draft
Use → Update
Deprecated → Replace
Archive → Restore
```

# 5. 文档类型

## 5.1 README

用于：

- 解释目录；
- 说明入口；
- 提供导航；
- 概括用途；
- 链接详细文档。

README 不应承载所有细节。

---

## 5.2 Process Documentation

用于定义：

- 生命周期；
- 工作流；
- Gate；
- Checklist；
- 角色；
- 状态；
- 模板。

---

## 5.3 Design Documentation

用于定义：

- 目标；
- 用户；
- 规则；
- 流程；
- Scope；
- Non-goals；
- 验收标准；
- 体验原则。

---

## 5.4 Technical Design

用于定义：

- 架构；
- 模块；
- 接口；
- 数据流；
- 生命周期；
- 风险；
- 方案对比；
- 迁移；
- 性能；
- 安全。

---

## 5.5 API Documentation

用于说明：

- Endpoint；
- 输入；
- 输出；
- 错误码；
- 示例；
- 版本；
- 权限；
- 限制。

---

## 5.6 Data Documentation

用于说明：

- Schema；
- 字段；
- 类型；
- 默认值；
- ID；
- 状态；
- 版本；
- 迁移；
- 保留策略；
- 所有权。

---

## 5.7 Operational Documentation

用于：

- 部署；
- 发布；
- 回滚；
- 监控；
- 告警；
- 故障处理；
- 运行手册；
- 恢复步骤。

---

## 5.8 Decision Record

用于记录：

- 背景；
- 选项；
- 决策；
- 原因；
- 后果；
- 复审条件。

---

## 5.9 Tutorial

用于教读者完成一项具体工作。

特点：

- 目标明确；
- 步骤顺序明确；
- 有示例；
- 有验证结果。

---

## 5.10 How-to Guide

用于解决一个具体问题。

例如：

```text
如何生成 Release Build
如何恢复损坏存档
如何新增本地化语言
```

---

## 5.11 Reference

用于快速查询事实。

例如：

- 命名规范；
- 输入动作表；
- 碰撞层表；
- 状态码；
- 配置字段；
- 术语表。

---

## 5.12 Troubleshooting

用于：

- 症状；
- 原因；
- 检查方法；
- 解决步骤；
- 预防方式。

---

## 5.13 Postmortem / Retrospective

用于记录：

- 发生了什么；
- 为什么发生；
- 如何处理；
- 学到了什么；
- 后续行动。

---

# 6. 文档层级

推荐：

```text
README
→ Overview
→ Detailed Guide
→ Reference
→ Template / Checklist
```

示例：

```text
process/README.md
→ workflow/feature-lifecycle.md
→ quality/definition-of-ready.md
→ templates/feature-template.md
```

不同层级应避免重复。

---

# 7. 文档目录设计

示例：

```text
docs/
├── README.md
├── design/
├── architecture/
├── process/
├── engineering/
├── operations/
├── api/
├── data/
├── decisions/
├── tutorials/
├── troubleshooting/
├── templates/
└── archive/
```

目录应基于读者和内容职责，而不是作者姓名。

不推荐：

```text
docs/alice/
docs/bob/
docs/misc/
docs/others/
```

---

# 8. 文件命名

推荐使用：

```text
kebab-case.md
```

例如：

```text
feature-lifecycle.md
release-workflow.md
save-format.md
```

避免：

```text
Doc1.md
new.md
final.md
notes.md
misc.md
```

## 8.1 文件名原则

- 表达主题；
- 稳定；
- 可搜索；
- 不包含日期，除非是事件文档；
- 不使用作者名；
- 不使用 final；
- 不使用模糊缩写。

---

# 9. 文档标题

文件内 H1 应清楚表达正式名称。

例如：

```markdown
# Release Workflow（发布流程）
```

文件名和标题可以略有差异，但主题必须一致。

---

# 10. 文档元数据

推荐在文档顶部包含：

```markdown
> 状态：
> Owner：
> 适用范围：
> 最后更新：
> 下次复审：
> 替代文档：
```

## 10.1 状态

推荐：

```text
Draft
In Review
Approved
Active
Deprecated
Archived
```

---

## 10.2 Owner

Owner 负责：

- 内容准确；
- 变更更新；
- 处理反馈；
- 定期复审；
- 决定废弃或归档。

Owner 不一定是原作者。

---

## 10.3 最后更新

用于判断时效。

但不能仅靠日期判断是否有效，还应检查内容。

---

## 10.4 下次复审

适合：

- 流程规范；
- 平台文档；
- 依赖说明；
- 合规文档；
- 发布手册。

---

# 11. 文档状态

## 11.1 Draft

- 内容不完整；
- 允许大幅变化；
- 不应被当作正式规则。

---

## 11.2 In Review

- 草稿基本完成；
- 正在评审；
- 可能仍需修改。

---

## 11.3 Approved

- 已通过评审；
- 可以正式使用；
- 进入维护阶段。

---

## 11.4 Active

- 当前正式有效；
- 是主要事实来源。

---

## 11.5 Deprecated

- 不再推荐使用；
- 仍可能被旧系统引用；
- 必须指向替代文档；
- 应说明停止使用日期。

---

## 11.6 Archived

- 不再参与当前工作；
- 只保留历史参考；
- 不应被新工作引用。

---

# 12. 文档创建前检查

创建新文档前应确认：

```markdown
- [ ] 已搜索是否存在类似文档；
- [ ] 新文档有明确读者；
- [ ] 新文档有明确目的；
- [ ] 内容不会更适合加入已有文档；
- [ ] Owner 明确；
- [ ] 存放目录合理；
- [ ] 文档类型明确；
- [ ] 预计维护成本可接受；
```

---

# 13. 文档 Draft

## 13.1 推荐结构

```markdown
# 标题

> 状态：
> 适用范围：
> 目的：

## 1. 文档目的

## 2. 定义

## 3. 原则

## 4. 流程 / 内容

## 5. 检查清单

## 6. 模板

## 7. 常见失败模式

## 8. 总结
```

并非所有文档必须使用完全相同结构，但应保持层次清楚。

---

## 13.2 Draft 原则

- 先写目标和边界；
- 再写主要内容；
- 最后补模板和示例；
- 避免一开始追求完美措辞；
- 先解决结构，再解决格式；
- 将未决问题明确标记。

---

# 14. 文档 Review

## 14.1 评审维度

```text
Correctness
Completeness
Clarity
Consistency
Usability
Freshness
Ownership
Discoverability
```

---

## 14.2 正确性

```markdown
- [ ] 与当前实现一致；
- [ ] 与当前流程一致；
- [ ] 无事实错误；
- [ ] 示例正确；
- [ ] 链接正确；
```

---

## 14.3 完整性

```markdown
- [ ] 目标清楚；
- [ ] 适用范围清楚；
- [ ] 关键步骤完整；
- [ ] 例外已说明；
- [ ] 失败处理已说明；
- [ ] Owner 明确；
```

---

## 14.4 清晰性

```markdown
- [ ] 标题明确；
- [ ] 层级合理；
- [ ] 术语统一；
- [ ] 无模糊表述；
- [ ] 示例足够；
- [ ] 不需要额外口头解释；
```

---

## 14.5 一致性

```markdown
- [ ] 与相关文档不冲突；
- [ ] 命名一致；
- [ ] 状态一致；
- [ ] 术语一致；
- [ ] 链接到事实源；
```

---

## 14.6 可用性

```markdown
- [ ] 读者可以直接执行；
- [ ] 步骤顺序合理；
- [ ] 检查清单可操作；
- [ ] 模板可复用；
- [ ] 结果可验证；
```

---

# 15. Review 角色

根据文档类型，Reviewer 可包括：

- Subject Matter Expert；
- 使用者；
- 工程负责人；
- 设计负责人；
- QA；
- Operations；
- Security；
- Legal；
- 新成员。

只由作者自己评审，容易遗漏上下文假设。

---

# 16. Publish：发布文档

文档 Approved 后，应：

- 更新状态；
- 更新索引；
- 添加相关链接；
- 通知目标读者；
- 移除 Draft 标记；
- 明确生效时间；
- 标记替代关系；
- 确保权限可访问。

文档发布不等于产品发布。

这里的 Publish 指：

> 文档正式成为当前工作可引用的有效内容。

---

# 17. 文档索引

每个主要目录应有 README 或 Index。

索引应包括：

```text
文档名称
用途
状态
适用读者
主要入口
```

示例：

```markdown
| 文档 | 用途 | 状态 |
|---|---|---|
| feature-lifecycle.md | 功能生命周期 | Active |
| definition-of-ready.md | 开始条件 | Active |
```

---

# 18. 交叉链接

文档应链接：

- 上层文档；
- 下层详细文档；
- 相关流程；
- 模板；
- 示例；
- 替代文档；
- 决策记录。

避免形成孤立文档。

---

# 19. Anchor 与内部链接

长文档应使用清楚的标题层级。

需要时提供目录。

内部链接应：

- 稳定；
- 不依赖频繁变化的标题；
- 定期检查；
- 避免深层脆弱链接。

---

# 20. 单一事实来源策略

对于每类核心信息，应明确事实源。

示例：

| 信息 | Source of Truth |
|---|---|
| 引擎版本 | project metadata |
| 版本号 | release config |
| API Schema | API spec |
| Roadmap | roadmap document |
| Bug 状态 | issue tracker |
| 发布记录 | release report |
| 架构决策 | ADR |

文档不应手工复制动态数据。

---

# 21. 文档与代码

## 21.1 应放在代码中的内容

- 类型；
- 函数；
- 参数；
- 局部约束；
- 不明显的算法原因；
- 边界条件；
- 使用示例。

## 21.2 应放在外部文档中的内容

- 系统背景；
- 架构；
- 跨模块关系；
- 操作流程；
- 发布；
- 迁移；
- 维护；
- 决策；
- 风险；
- 团队规范。

---

# 22. 注释原则

注释应说明：

```text
为什么这样做
为什么不能用更简单方式
有哪些重要约束
哪些行为不能改变
```

不应重复代码：

```gdscript
# 将 count 加 1
count += 1
```

---

# 23. 文档与 Issue Tracker

Issue Tracker 用于：

- 当前状态；
- Owner；
- Priority；
- 执行任务；
- Bug；
- 时间线；
- 讨论。

长期文档用于：

- 稳定规则；
- 设计；
- 架构；
- 流程；
- 背景；
- 决策；
- 使用方式。

不要把长期设计只留在已关闭 Issue 中。

---

# 24. 文档与聊天记录

聊天适合：

- 快速沟通；
- 临时讨论；
- 澄清；
- 协调。

重要结论应转入：

- 正式文档；
- ADR；
- Feature Record；
- Release Record；
- Bug Record。

聊天记录不应成为唯一事实来源。

---

# 25. 文档与会议

会议结束后，应提取：

- 决策；
- Action；
- Owner；
- 截止时间；
- 风险；
- 未决问题。

不需要完整逐字记录所有讨论。

---

# 26. Documentation as Code

建议文档：

- 使用 Markdown；
- 进入版本控制；
- 通过 Pull Request Review；
- 与代码变更一起更新；
- 支持 Diff；
- 支持自动检查；
- 支持静态站点生成。

优点：

- 可追踪；
- 可评审；
- 可回滚；
- 与代码同步；
- 易于自动化。

---

# 27. 文档版本控制

对于 Git 管理文档，不一定需要手工版本号。

可以依赖：

- Commit；
- Tag；
- Release；
- History。

需要显式版本号的情况：

- 对外发布规范；
- 合同；
- 合规；
- API；
- 数据格式；
- 需要并行维护多个版本；
- 用户手册。

---

# 28. 文档变更

重要变更应记录：

```text
Date
Change
Reason
Owner
Impact
```

可以使用：

- Changelog；
- Git History；
- 文档内 Change Log；
- ADR。

---

# 29. 文档维护触发器

以下事件应触发文档检查：

- Feature 变化；
- API 变化；
- 数据变化；
- 架构变化；
- 工作流变化；
- 发布；
- 引擎升级；
- 插件升级；
- Incident；
- 用户反馈；
- 新平台；
- 新语言；
- 权限变化；
- Owner 变化。

---

# 30. Freshness Review：时效检查

文档应定期检查：

- 内容是否仍然正确；
- 链接是否有效；
- 工具版本是否变化；
- 示例是否可运行；
- Owner 是否仍存在；
- 替代文档是否出现；
- 读者是否仍需要；
- 是否应合并或归档。

---

## 30.1 建议频率

根据类型：

| 文档类型 | 建议复审 |
|---|---|
| 流程规范 | 每 6–12 个月 |
| 架构文档 | 每次重大变更 |
| 发布手册 | 每次 Release |
| API 文档 | 每次接口变化 |
| 故障手册 | 每次 Incident |
| Roadmap | 每月或每季度 |
| 教程 | 每次工具升级 |

---

# 31. 过期文档信号

- 与当前实现不一致；
- 链接失效；
- 示例无法运行；
- Owner 不明；
- 长期无人引用；
- 内容与其他文档冲突；
- 使用已废弃工具；
- 多个文档描述同一事实；
- 读者持续误解；
- 依赖旧版本。

---

# 32. Deprecated

文档进入 Deprecated 时，应在顶部明确：

```markdown
> 状态：Deprecated  
> 替代文档：  
> 停止使用日期：  
> 原因：
```

Deprecated 文档仍可保留一段时间，用于：

- 旧版本；
- 迁移；
- 历史引用；
- 过渡。

---

# 33. Archived

文档进入 Archive 时：

- 从主要索引移除；
- 不再作为当前事实源；
- 保留历史链接；
- 标记归档日期；
- 标记替代文档；
- 避免被新工作引用。

---

# 34. 删除文档

可以删除：

- 垃圾文件；
- 无历史价值的误创建；
- 纯重复且已合并；
- 无引用的临时草稿；
- 敏感错误内容。

有决策价值或历史价值的文档应优先归档而非删除。

---

# 35. 合并文档

出现以下情况时应考虑合并：

- 内容高度重复；
- 同一读者；
- 同一主题；
- 分开后难以理解；
- 更新经常必须同步。

合并后应：

- 保留原链接重定向或说明；
- 更新索引；
- 更新引用；
- 归档旧文档。

---

# 36. 拆分文档

文档过大时可以拆分。

过大的信号：

- 主题过多；
- 不同读者；
- 更新频率不同；
- 超过合理导航长度；
- 内容难以复用；
- Review 责任不同。

拆分时保留一个 Overview 作为入口。

---

# 37. 模板管理

模板应：

- 有明确用途；
- 不包含项目特定过时示例；
- 定期更新；
- 与流程文档一致；
- 保持最小必要字段；
- 不鼓励机械填写。

模板不是为了让所有文档看起来一样，而是减少遗漏。

---

# 38. 术语管理

建议维护 Glossary。

记录：

```text
Term
Definition
Avoided Synonyms
Context
Owner
```

术语一致可以减少：

- 多义；
- 误解；
- 重复概念；
- 文档冲突。

---

# 39. 文档语言

应根据目标读者确定主语言。

如果支持多语言文档，应明确：

- 主版本；
- 翻译版本；
- 同步机制；
- 更新时间；
- 翻译 Owner。

不要让多个语言版本都被当作独立事实源。

---

# 40. 示例与代码片段

示例应：

- 可运行；
- 简短；
- 与当前版本一致；
- 明确是否为伪代码；
- 避免包含敏感信息；
- 不使用即将废弃 API。

---

# 41. 截图与视频

截图容易过期。

应：

- 标记版本；
- 仅在视觉信息必要时使用；
- 避免用截图代替可复制文本；
- UI 变化后更新；
- 避免暴露敏感数据；
- 使用清楚标注。

---

# 42. 文档自动检查

可以自动检查：

- 断链；
- 拼写；
- 标题层级；
- 文件命名；
- Front Matter；
- 重复标题；
- 无 Owner；
- 过期日期；
- Markdown 格式；
- 代码片段；
- 敏感信息。

---

# 43. Documentation CI

示例流程：

```text
Lint
→ Link Check
→ Spell Check
→ Build Docs
→ Validate Metadata
→ Publish Preview
```

---

# 44. 文档安全

文档中不应包含：

- 密钥；
- Token；
- 密码；
- 私密 URL；
- 真实用户数据；
- 客户信息；
- 内部敏感架构细节；
- 未授权个人信息；
- 未公开财务信息。

需要时使用：

- 脱敏；
- 示例值；
- 占位符；
- 权限控制；
- 私有文档库。

---

# 45. 权限与可见性

文档可以分为：

```text
Public
Internal
Restricted
Confidential
```

应根据内容设置访问权限。

限制访问的文档仍应有可发现的存在说明，避免重复创建。

---

# 46. Documentation Owner

Owner 负责：

- 准确性；
- Review；
- 更新；
- 失效处理；
- 合并；
- 归档；
- 回应问题。

如果 Owner 离开，应重新分配。

---

# 47. Reader Responsibility

读者也应：

- 检查状态；
- 检查最后更新；
- 使用 Active 文档；
- 发现错误时反馈；
- 不引用 Archived 文档作为当前规则。

---

# 48. 文档优先级

不是所有文档都同等重要。

优先级较高：

- 发布；
- 回滚；
- 数据迁移；
- 安全；
- 架构边界；
- 核心流程；
- 高风险操作；
- 新成员入口。

优先级较低：

- 很少使用的辅助技巧；
- 极少变化的背景介绍；
- 已有工具自动生成的重复信息。

---

# 49. 文档债务

Documentation Debt 包括：

- 缺失文档；
- 过期文档；
- 重复文档；
- 冲突文档；
- 无 Owner；
- 无索引；
- 无替代链接；
- 示例失效；
- 重要知识只存在个人记忆。

应像技术债务一样管理。

---

# 50. 文档更新与 Feature DoD

如果 Feature 改变以下内容，DoD 应要求更新文档：

- 用户流程；
- API；
- 数据；
- 配置；
- 架构；
- 发布；
- 存档；
- 权限；
- 运维；
- 故障处理。

# 55. 常见失败模式

## 55.1 文档越多越好

结果：

- 重复；
- 冲突；
- 过期；
- 难以搜索；
- 维护成本过高。

---

## 55.2 文档只有作者看得懂

原因：

- 假设太多；
- 缺少背景；
- 缩写未解释；
- 步骤不完整。

---

## 55.3 所有信息都写在 README

结果：

- README 过长；
- 导航困难；
- 更新冲突；
- 主题混乱。

README 应主要负责入口和导航。

---

## 55.4 只创建，不维护

结果：

- 文档逐渐不可信；
- 团队回到口头沟通；
- 新成员被误导。

---

## 55.5 多个事实源

例如：

- 两份引擎版本文档；
- 三份发布步骤；
- 多个 Roadmap；
- 多个命名规范。

结果：

- 不知道哪份有效；
- 更新不同步。

---

## 55.6 文档状态不清楚

读者无法判断：

- 是草稿还是规则；
- 是否已经过期；
- 是否仍然有效；
- 是否有替代文档。

---

## 55.7 文档只描述理想流程

问题：

- 没有例外；
- 没有失败处理；
- 没有现实约束；
- 无法实际执行。

---

## 55.8 文档与工作分离

表现：

```text
功能完成了，之后再补文档。
```

结果通常是文档永远不补。

---

## 55.9 复制聊天记录当文档

结果：

- 结构差；
- 结论不清；
- 噪音过多；
- 无法执行。

---

## 55.10 截图代替文本

结果：

- 无法搜索；
- 无法复制；
- 容易过期；
- 可访问性差。

---

## 55.11 文档没有 Owner

结果：

- 无人更新；
- 无人决定废弃；
- 长期过期。

---

## 55.12 Deprecated 文档仍出现在主索引

结果：

- 新成员继续使用旧流程；
- 过渡永远结束不了。

---

# 56. Documentation Metrics

可参考：

- Active 文档数量；
- 无 Owner 文档数量；
- 过期文档数量；
- 断链数量；
- 重复文档数量；
- 平均复审周期；
- 文档更新时间；
- 搜索成功率；
- 新成员常见问题；
- 文档引发的错误数量；
- Deprecated 未替换数量；
- Archive 数量；
- 文档 Review 通过率。

指标应帮助改善：

- 可发现性；
- 准确性；
- 维护性；
- 使用效果。

不应追求：

- 文档数量越多越好；
- 每份文档越长越好；
- 所有知识都文档化。

# 57. 文档维护节奏

## 每次 Feature

- 更新相关设计；
- 更新 API；
- 更新数据；
- 更新使用说明；
- 更新测试说明。

## 每次 Release

- 更新 Release Notes；
- 更新 Changelog；
- 更新迁移说明；
- 更新 Known Issues；
- 更新运行手册。

## 每月或每季度

- 检查过期文档；
- 检查无 Owner 文档；
- 检查重复文档；
- 检查断链；
- 检查索引。

## 每次重大变更

- 更新架构；
- 更新 ADR；
- 更新流程；
- 更新依赖；
- 更新升级说明。

# 58. 文档完成定义

一份文档可以被视为完成，当：

- 目的明确；
- 读者明确；
- 内容正确；
- 结构清楚；
- 相关链接完整；
- 与当前实现一致；
- Owner 明确；
- 状态明确；
- Review 已完成；
- 索引已更新；
- 替代和过期关系已处理；
- 不包含敏感信息；
- 可以被目标读者独立使用。

---

# 59. 最终原则

Documentation Workflow 的本质不是：

> 建立一个存放大量 Markdown 文件的仓库。

它的本质是：

> 确保项目中的关键知识、规则、决策和操作方式能够被正确记录、找到、理解、执行、维护和淘汰。

一个健康的文档体系应做到：

- 目的明确；
- 读者明确；
- 单一事实来源；
- 链接优先于复制；
- 状态清楚；
- Owner 清楚；
- 与实际工作同步；
- 可搜索；
- 可执行；
- 可评审；
- 可更新；
- 可废弃；
- 可归档。
