# FSM 细则 G:叙事/任务状态机(Narrative / Quest FSM)

> **索引归属**:`fsm-index.md` §2 中的 G 类。共享概念引用索引 `[§3.x]`;版本迁移机制建立在 C 篇 §4.4 之上并推向极限。
>
> **本类地位**:数据驱动 FSM(第 4 档 `[§3.1]`)的完全体。核心矛盾与其他类别都不同:**状态图本身是内容**——由内容侧批量生产、随版本频繁改动、且与玩家存档的纠缠最深。本篇一半在讲"图作为数据"的运行时,另一半在讲"图作为内容资产"的生产与演化。

---

## 1. 定位

### 1.1 坐标

- **节奏**:事件驱动,纯粹(无 tick 出边;时限任务的超时也经由时间系统发事件)
- **层**:逻辑层,进存档——且是**跨局、跨版本长期存活**的存档大户

### 1.2 覆盖范围与本质差异

任务链、剧情节点网、成就、教程步骤、世界事件(声望阶段、阵营剧情)。与 C 类的镜像关系在 C 篇 1.3 已定义,这里补全另一半:

| | C 实体生命周期 | G 叙事/任务 |
|---|---|---|
| 图与实例 | **一图 × 海量实例** | **海量图 × 每图少量实例**(通常 1) |
| 图的作者 | 程序侧(架构) | **内容侧**(策划/写作者) |
| 图的变更频率 | 低(随系统重构) | **极高**(随内容更新,每版本必改) |
| 运行时瓶颈 | 批量结算 | **监听成本**(§4.3) |
| 演化难题 | 状态名迁移 | **整图结构迁移**(§4.5) |

"图的作者是内容侧"推出 G 类的全部架构决策:图必须是数据(第 4 档别无选择)、条件/效果必须是封闭分类法(内容侧不写代码)、必须有静态校验工具(内容侧会填错表)、必须有版本迁移策略(内容侧改图不看存档死活)。

### 1.3 与相邻类别的边界补充

- **对话树是不是 G?** 结构上是图,但多数对话**不进存档**(中断即废,重新开始)且严格是"游标沿图行走"——按 `[§3.6]` 用图游标实现,不上 FSM 全套。对话的**后果**(选了哪个选项、解锁了什么)才进 G:对话结束时向 G 层发效果。分界线:**会话内瞬时结构归对话系统,跨会话持久事实归 G**
- **教程**:线性教程是序列(游标);可自由触发、可跳过、有依赖关系的教学点集合才是 G 的退化形态

## 2. 状态与转换的典型形态

### 2.1 任务实例的状态集(近乎全行业通用)

```
unavailable   前置未满足,玩家不可见
available     可承接(任务板上出现)
active        进行中
completable   目标已达成,待交付
completed     终态(奖励已发)
failed        终态(可选:可重试则回 available)
```

**这张图是架构不是内容**——所有任务共用这六态骨架(C 类式的一图多实例!),内容侧生产的不是状态图,而是**挂在转换上的条件与效果**:

| 转换 | 内容侧填写 |
|------|-----------|
| unavailable → available | 解锁条件(前置任务、等级、声望、flag) |
| available → active | 承接条件 + 承接效果(给任务道具、刷目标) |
| active → completable | **目标定义**(§4.2) |
| completable → completed | 交付条件(找谁交)+ 奖励效果 |
| active → failed | 失败条件(时限、关键 NPC 死亡) |

这个分解是 G 类最重要的设计决策:**状态图收敛为稳定骨架,内容自由度全部下放到条件/效果分类法**。图不再随内容变,§4.5 的迁移难题就砍掉了最重的一半。真正需要自定义图结构的(多结局剧情网),作为例外走 §5.1 的升级通道。

### 2.2 条件与效果:封闭分类法

内容侧可用的条件/效果是**封闭集合**——对分类法关闭,对组合开放(与 SOLID 篇 OCP 处的封闭 tag 分类法同一思想,此处是它在叙事域的实例):

```
条件类型(示例集):flag_set / quest_in_state / item_count≥ / entity_in_state /
                   stat≥ / time_reached / location_entered
效果类型(示例集):set_flag / grant_item / grant_currency / unlock_quest /
                   force_quest_state / spawn_entity / play_cutscene
```

条件支持 and/or/not 组合树。运行时是一个通用解释器:`eval(condition_tree, world_query) -> bool`、`apply(effect_list, world_command)`——`world_query/world_command` 是注入的窄接口(黑板思想的世界版),解释器本身零游戏知识,headless 全量可测。**新增条件/效果类型是程序侧动作**(加一个 case + 更新校验器 + 更新编辑器下拉框),有意保持一点摩擦——分类法的膨胀速度就是系统腐化速度。

## 3. 实现选型

第 4 档,无悬念。分层归属按 `[§3.7]`:

| 部件 | 归属 |
|------|------|
| 六态骨架 + 通用转换逻辑 | 系统层(一份,程序侧) |

## 4. 特色问题精讲

### 4.1 目标(objectives)是计数器,不是状态

"杀 10 只狼 + 采 5 株草 + 与 NPC 对话"——把每个目标建成状态会产生进度组合的状态积。正确建模:**active 态内部的一组并行计数器**:

```
objectives: [
  { id: kill_wolves, type: kill, target_tag: wolf, need: 10, current: 0 },
  { id: gather_herb, type: collect, item: herb, need: 5, current: 0 },
  { id: talk_elder,  type: talk, npc: elder, need: 1, current: 0 }
]
```

`active → completable` 的条件固定为"全部(或声明的子集)计数器达标"。计数器进度是运行时状态(逐项进存档);目标的可选/隐藏/顺序解锁(目标 B 在 A 达标后才出现)用目标上的 `visible_when` 条件表达,复用 §2.2 的条件树——不为它发明第二套机制。这是 `[§3.6]`"进度不是状态"的 G 类版本,与 A 篇攻击相位游标同源。

### 4.2 监听成本:千级任务不能都竖着耳朵

每只狼死亡都去问全部 3000 个任务定义"你关心吗?"——G 类的运行时瓶颈在此,不在转换本身。两级削减:

1. **只有活实例听**:unavailable/终态的任务不注册任何监听。承接时注册,离开 active 注销
2. **触发索引**:注册不是"订阅一切",而是按目标类型建倒排索引 `{(kill, wolf): [quest_a, quest_c], (collect, herb): [quest_a]}`——狼死亡事件只投递给两个任务实例。索引在承接/交付时增量维护

解锁检查(unavailable → available)同理不轮询:解锁条件里引用的每个 flag/任务/等级变更时,经**反向依赖索引**(构建期从全部任务定义静态生成,见 §4.4)只唤醒受影响的任务重评估。**G 类引擎的本体不是状态机,是这两张索引**——状态机部分反而平凡。

### 4.3 跨任务依赖:图之上的图

任务之间的解锁关系(A 完成开 B、B/C 完成开 D)构成一张**元图**。三条纪律:

- **构建期检测环**:任务 A 要求 B 完成、B 要求 A——内容侧一定会写出来,校验工具(§7.1)在导出时拓扑排序,有环即拒绝出包
- **依赖只许引用 id,禁止反向硬编码**:B 的定义里写"前置:A",A 的定义里**不写**"完成后解锁 B"——单向声明,反向索引由构建期生成。双向手写必然漂移
- **强制状态效果(force_quest_state)是受控后门**:剧情杀需要"强制任务 X 失败"时用它,但它绕过条件检查,校验工具对每处使用要求标注理由,并在 §4.5 迁移时重点核查

### 4.4 版本演化:G 类的终极难题

C 篇 §4.4 处理"状态名改了";G 类面对的是**图的语义整个变了**:玩家存档里 quest_x 进行中,新版本删了它/改了目标/拆成两个任务。C 式迁移表不够用,需要一套策略体系:

**(a) 定义打版本戳**:每个任务定义带 `version: int`,内容侧改动语义时必须 +1(改错别字不加)。存档记录每实例承接时的版本号,读档比对即知"此实例过期"。

**(b) 每任务声明迁移策略**(定义层字段,内容侧选择):

| 策略 | 行为 | 适用 |
|------|------|------|
| reset | 退回 available,进度清零 | 默认;目标结构大改 |
| autocomplete | 直接完成并补发奖励 | 任务被删除/大幅简化,不让玩家吃亏 |
| best_effort | 按 objective id 对齐搬运进度,对不上的清零 | 小改(数量调整、加一个目标) |
| custom | 指向一个迁移函数 id | 极少数复杂重构 |

**(c) 孤儿实例兜底**:存档里的任务 id 在新版定义中已不存在 → 全局策略(默认 autocomplete + 日志),永不崩溃、永不静默丢弃(玩家会记得"我有个任务做到一半")。

**(d) 铁律:终态不迁移**。completed/failed 的历史事实永久保留原样——它们是其他任务解锁条件的依据,重写历史会引发解锁级联错乱。由此推出内容侧红线:**已上线任务的 id 与其终态语义永不复用**——废弃 id 进墓碑名单,校验工具阻止新任务占用。

这套体系的成本不小,但它是把"内容更新"与"玩家存档"解耦的唯一通路——没有它,每次改任务都在赌玩家没接过这个任务。

### 4.5 内容校验:把 code review 做成编译器

内容侧以表格/编辑器批量生产,错误模式高度固定,全部可静态捕获。校验器(构建期强制运行,红即拒绝出包)至少查:

引用完整性(条件引用的 flag/item/quest/npc id 全部存在)、依赖无环(§4.3)、可达性(每个任务从初始状态存在至少一条解锁路径——最常见的内容 bug 是"写了个永远解不开的任务")、奖励与目标非空、版本戳合规(定义 diff 出语义变更但 version 未 +1 时告警)、force_quest_state 理由标注、墓碑 id 未被复用。

**校验器是 G 类工程投入回报率最高的组件**——它把"上线后玩家卡剧情"翻译成"出包时红一行",这类工具与游戏本体解耦、输入输出全是数据,也最适合交给 AI 代理独立实现和维护。

## 5. 升级路径

### 5.1 骨架不够用:自定义图 + 脚本逃生舱

多结局剧情网、狼人杀式的复杂事件——六态骨架容不下时,升级为**每内容一张自定义图**(状态/转换全在数据里,即第 4 档完全体),条件/效果分类法照用。分类法也不够表达时,最后的逃生舱是 SOLID 篇 OCP 处的策略对象:转换上挂脚本 id。纪律:**脚本是逃生舱不是正门**——每个挂脚本的转换在校验报告中单列,数量是仪表盘指标;脚本比例持续上升说明分类法该扩编了(把高频脚本模式吸收为新效果类型),而不是继续放任。

### 5.2 走向涌现叙事:storylet / 质量驱动

预写任务图的天花板是组合爆炸(想让事件因玩家状态自由涌现时,图不可枚举)。**storylet** 范式:内容原子 = {前置条件, 内容, 效果} 三元组,无图——运行时按当前世界状态筛出全部可触发原子,加权抽取。它其实是把 §2.2 的条件/效果分类法保留、把"图"扔掉:任务图的骨架消失,叙事结构从"作者铺路"变为"条件涌现"。识别信号:内容侧开始大量生产"结构相同、仅条件与文本不同"的微型任务——那就是 storylet 在敲门。两者共存无碍:主线用任务图,随机世界事件用 storylet 池,共享同一套条件/效果解释器。

## 6. 坏味道清单(G 类特有)

1. ❌ **任务逻辑写成代码**(每个任务一个 .gd)—— 内容侧无法生产、校验器无从查起,G 类第一死罪
2. ❌ **目标建成状态**(§4.1 状态积)
3. ❌ **事件广播全量轮询任务列表**(§4.2)
4. ❌ **双向手写依赖**(A 写解锁 B,B 又写前置 A,§4.3)
5. ❌ **改任务语义不 bump version**(§4.4a)—— 迁移体系被静默架空
6. ❌ **复用已废弃任务 id**(§4.4d)
7. ❌ **对话中途状态进存档** —— 会话瞬时结构与持久事实混淆(§1.3)
8. ❌ **脚本逃生舱常态化**(§5.1)

## 7. 测试要点

1. **静态校验即测试**:§4.5 校验器全绿是 CI 第一关——G 类多数"测试"根本不需要运行游戏
2. **模拟通关测试**:headless 下用效果指令直接驱动世界状态(set_flag、发 kill 事件),脚本化走完关键任务链,断言状态序列与奖励——G 版的场景剧本测试(B 篇 §7.2 同款形态)
3. **迁移矩阵测试**:旧版存档样本 × 四种策略逐一回放,断言无崩溃、无孤儿静默丢失、终态未被改写(§4.4)。**每次发版前用真实旧档跑一遍**
4. **索引一致性测试**:随机承接/放弃/完成序列后,断言触发索引与实际 active 集合一致(§4.2 增量维护的验收)
5. **可达性回归**:校验器的可达性检查对全部任务通过,作为内容合入的门禁

## 8. AI 代理协作约定(供 CLAUDE.md 摘录)

1. 新任务/剧情内容一律以定义数据(条件/效果/目标)表达;发现需求超出现有分类法时,停止实现并上报"扩编分类法 vs 挂脚本"的选择建议,不得擅自写内容专属代码。
2. 条件/效果解释器与两张索引(触发索引、反向依赖索引)为纯对象,经 world_query/world_command 窄接口访问世界;解释器代码中出现具体任务 id 即为实现错误。
3. 修改任何已上线任务定义时:判定是否语义变更,是则 version +1 并选定迁移策略;删除任务时将 id 移入墓碑名单。
4. 校验器与游戏代码同仓维护;新增条件/效果类型时,解释器 case、校验规则、编辑器枚举三处必须同一提交内完成。
5. 交付内容批次必须附带:校验器全绿报告、涉及任务链的模拟通关脚本、(若有语义变更)迁移矩阵测试结果。
