# FSM 细则 A:角色/动作状态机(Character Action FSM)

> **索引归属**:`fsm-index.md` §2 中的 A 类。共享概念引用索引 `[§3.x]`,不重复。
>
> **本类地位**:十类中唯一以"手感"为最高验收标准的一类。手感的本质是**毫秒级时序的宽容度设计**——本篇一半篇幅在讲各种"宽限窗口"。也是 `[§3.3]` 中"状态自治式"这一例外分支的正当使用场。

---

## 1. 定位

### 1.1 坐标

- **节奏**:帧驱动(通常物理帧),状态含 `physics_update`,每帧评估转换
- **层**:逻辑层,但与表现层的耦合是十类中最紧的(动画帧数据直接参与规则判定,见 §4.4)

### 1.2 覆盖范围与本质差异

玩家角色与重要单位的动作控制:idle/run/jump/fall/dash/attack/hurt/dead。与 C 类的本质差异在转换条件的性质:C 的条件是**离散事实**(资源够不够、tick 满没满),A 的条件是**连续物理量与瞬时输入的组合**(速度、是否着地、这一帧按没按跳)。这决定了 A 的一系列反常规选择:状态自治式、每帧全量评估、逻辑与动画数据互相渗透。

### 1.3 与相邻类别的边界

| 相邻类 | 区分标准 |
|--------|----------|
| B AI 行为 | A 管"身体怎么动"(动作执行层),B 管"脑子想干嘛"(决策层)。AI 单位通常**两台机都有**:B 决定"追击",产出移动/攻击意图;A 消费意图驱动身体。玩家角色则是"输入系统替代了 B"。二者绝不合并——合并的症状是出现 `chase_while_jumping` 这类决策×动作的状态积 `[§3.5]` |
| H 动画 | A 是真相,H 是投影——但 A 类是唯一允许**反向读 H 侧数据**的:攻击的相位时长来自动画帧数据(§4.4)。读的是数据不是节点,边界仍然成立 |
| J 输入识别 | J 把原始输入流识别为语义指令(搓招→"发波");A 消费语义指令。连招游戏中 J 是 A 的前置过滤器 |

## 2. 状态与转换的典型形态

### 2.1 典型状态集(2D 平台跳跃)

```
idle / run          地面态
jump / fall         空中态(上升/下降分开:两者的重力、控制力、可转出集合都不同)
dash                冲刺(定时长,锁其他输入)
attack              攻击(内部有相位,见 §4.4)
hurt                受击硬直
dead                终态
```

### 2.2 为什么转换表在这里不好用

试写一条:`run --[速度≈0 ∧ 无方向输入 ∧ 着地]--> idle`。A 类的"事件"不是离散消息,而是**每帧对连续量的谓词求值**;一个状态往往有四五条出边,谓词间还有优先级(受击 > 跳跃 > 攻击 > 移动)。硬套转换表的结果是把大量逻辑塞进 guard,表退化成谓词的花名册,失去"可审计的图"这个核心收益。

因此 A 类采用 `[§3.3]` 明文保留的例外——**状态自治式**:每个状态的 `physics_update` 末尾自查转换条件,返回目标状态名。代价(转换图散落各文件)用一条纪律补偿:**每个状态文件头部用注释块声明全部出边**,并保持 Mermaid 图与之同步——图从"机器保证"降级为"纪律保证",这是为手感灵活性付出的明码标价。

### 2.3 转换优先级

同一帧可能多个条件同时成立(落地瞬间按着攻击键)。纪律:每个状态的转换检查是**显式排序的 if 链**,顺序即优先级,注释标明。全局默认序:死亡 > 受击 > 强制位移(击退/弹反) > 玩家主动指令(跳/闪/攻) > 环境导出(落地/离地) > 移动/静止。

## 3. 实现选型

**推荐:第 2 档状态对象(纯 RefCounted)+ 状态自治式,由角色节点持有并驱动。**

```gdscript
class_name ActionState extends RefCounted
func enter(ctx) -> void: pass
func exit(ctx) -> void: pass
func physics_update(ctx, delta: float) -> StringName:
    return &""   # 返回目标状态名;空 = 不转换
```

ctx 是角色的运行时状态(速度、朝向、各计时器、对 def 帧数据的引用)+ 一个受限的身体接口。这里要诚实面对 A 类的特殊性:**A 的"逻辑"本身就是物理帧行为,与 CharacterBody 的隔离做不到 C/F 那种纯度**。务实的边界画法:

- ctx 持有的是**身体接口**(读:velocity、is_on_floor;写:velocity、请求 move)而非节点本身——测试时可注入假身体(脚本设定 is_on_floor 序列),不需要跑物理引擎也能测状态逻辑与各宽限窗口
- 状态对象依旧禁止 `get_node` / 播动画 / 出音效——这些由角色节点订阅 `state_changed` 后驱动(H 类接管)
- 第 3 档节点式在 A 类是可接受的次选(编辑器可视、教程生态成熟),但按索引 §3.1 的纪律,选它就要接受"状态逻辑焊死在场景树、假身体注入变难"的代价。人形角色 ≤ 一两个的项目差别不大;想复用状态机给多种单位,第 2 档明显占优

**驱动顺序(角色节点的 `_physics_process` 固定管线)**:

```
1. 采集输入 → 写入意图缓冲(§4.2)
2. current_state.physics_update(ctx, delta) → 内部含移动计算与转换自查
3. 身体接口应用 → move_and_slide()
4. 若返回了目标状态:exit → 换状态 → enter → 发 state_changed
```

转换在 move_and_slide **之后**结算,保证 enter 里读到的 is_on_floor 是本帧最终事实——落地判定类 bug 的一大来源就是这个顺序颠倒。

## 4. 特色问题精讲

### 4.1 输入意图层:按键不直接进状态机

原始输入(`Input.is_action_just_pressed`)只在管线第 1 步被采集一次,翻译成**带时间戳的意图**写入 ctx:

```gdscript
# ctx 内
var jump_requested_at := -INF   # 最近一次按跳的时刻
var move_axis := 0.0

func request_jump() -> void: jump_requested_at = now
func consume_jump(buffer: float) -> bool:
    if now - jump_requested_at <= buffer:
        jump_requested_at = -INF   # 消费掉,防止一次按键触发两次跳
        return true
    return false
```

三个理由:① `just_pressed` 每物理帧只对一帧为真,状态若恰好这帧没查就永久错过——意图层把"瞬时事件"改写为"有保质期的事实";② **输入缓冲**免费获得:落地前 0.1s 按跳,落地后 idle 的转换检查 `consume_jump(0.1)` 仍为真,预输入生效——这是"按键被吃了"这一最伤手感问题的标准解;③ AI 单位复用同一台 A 机时,B 机写同一个意图接口即可,输入系统与决策系统对 A 完全同形(§1.3 的架构兑现)。

### 4.2 宽限窗口家族:手感 = 对玩家撒的善意谎言

A 类的精髓是一组"物理上不该成立、体验上必须成立"的窗口,全部实现为 ctx 里的计时器 + 转换条件里的比较:

| 窗口 | 谎言内容 | 典型值 | 实现 |
|------|----------|--------|------|
| 输入缓冲 | "你按早了也算按准了" | 0.08–0.15s | §4.1 的意图保质期 |
| 土狼时间 | "你离开平台后的一小会儿仍算站着" | 0.05–0.1s | ctx 记录 `left_ground_at`,jump 的条件为 `is_on_floor ∨ now - left_ground_at ≤ coyote` |
| 可变跳高 | "松键的时机决定跳的高度" | — | jump 态内每帧检查:键已松 ∧ 仍在上升 → 竖直速度乘衰减系数 |
| 受击无敌 | "刚挨打的一小段打不到你" | 0.5–1s | hurt 的 exit 启动无敌计时,伤害入口 guard 查它 |

统一纪律:**窗口时长全部进定义层**(角色参数 Resource),不许硬编码在状态里——手感调参是高频迭代,改数据不改代码;且所有计时器用"记录时刻 + 当前比较"而非"每帧递减的倒计时变量",前者天然幂等、无漏减重减 bug。

### 4.3 抖动与迟滞

站在斜坡上 idle/run 每帧互跳、速度在阈值附近 walk/run 高频横跳——**状态抖动**(索引术语表预告过,A 类是第一现场)。三件套解法,按序尝试:

1. **迟滞(hysteresis)**:进出阈值分开。`run→idle` 要求速度 < 5,`idle→run` 要求速度 > 20,中间是死区
2. **最小驻留时间**:enter 后 N 帧内禁止转出(ctx 通用字段 `frames_in_state`,转换条件统一附加)
3. **条件消抖**:对"是否着地"这类物理引擎输出本身抖动的量,取连续 K 帧一致才认——本质是给布尔量加了低通滤波

抖动的危害不只视觉(动画闪烁),更在规则:每次转换都跑 enter/exit,音效重复播放、粒子重复生成、缓冲被误消费。

### 4.4 攻击状态:相位、帧数据与取消窗口

攻击不是一个匀质状态,内部三相:**前摇(startup)→ 判定(active)→ 后摇(recovery)**。三相的时长与判定框(hitbox)时机统称**帧数据**,归属有明确答案:**定义层**(每个招式一份 Resource:三相帧数、取消窗口、伤害、位移),attack 状态是读帧数据的通用解释器——新招式 = 新数据,OCP 在动作系统里的形态。

相位的实现**不要**拆成三个 FSM 状态(startup_attack/active_attack/recovery_attack 会把状态数 ×3 且相位推进是严格线性的)——attack 状态内部用一个相位游标即可,正是 `[§3.6]`"线性序列用游标"在状态内部的应用。

**取消窗口**是动作深度的来源:后摇的后半段允许被特定输入打断(普攻后摇可取消进闪避,但不可取消进另一次普攻)。实现:帧数据里声明 `cancel_windows: [{from_frame, to_frame, into: [dash, jump]}]`,attack 的转换检查在窗口内额外放行对应意图。**取消规则是数据不是代码**——格斗/动作游戏的连招体系本质是这张表,平衡性调整全在数据层发生。

### 4.5 与 H(动画)的接线:双向但不对称

- **正向(A→H,主通道)**:state_changed 信号驱动动画切换,H 侧自行处理过渡混合——A 不关心
- **反向(H→A,仅数据)**:帧数据在动画制作侧标定(动画第几帧出判定)后导出为定义层 Resource,A 读的是这份数据,**不在运行时查询 AnimationPlayer 的播放进度**。逻辑相位游标与动画播放是两条平行时间线,以同一份帧数据为共同真相——动画丢帧、变速播放都不会影响判定时序
- **例外通道**:少数事件(脚步声时刻、特效锚点)可用动画轨道的 Call Method/信号回传,但**只允许触发表现,不允许触发规则**——动画轨道调用 `deal_damage()` 是 A/H 边界破裂的经典形态:改个动画,伤害时序就变了,且 headless 下(无动画播放)规则直接失效

## 5. 升级路径

- **HSM `[§3.5]` 的入场信号**:空中态家族膨胀(jump/fall/double_jump/wall_slide/glide 都要处理"受击→hurt""落地→着地态"),把公共出边上提到 airborne 父状态。地面/空中两个父状态是平台动作游戏 HSM 的标准形态,再深的层级通常不需要
- **并行机**:移动状态 × 武器状态(持枪奔跑边跑边换弹)是正交维度,状态名出现 `run_reloading` 即拆——上下半身分离本质就是两台并行 A 机
- **叛逃信号**:招式表膨胀到需要"当前连段位置"参与转换判定时,考虑把连招推进交给 J 类(输入识别产出连段指令,A 只执行);A 机自身状态数长期 > 15 且互相转换关系网状化,说明动作系统的复杂度需要专门的动作图工具(动作游戏中间件的领域),FSM 手写已到上限

## 6. 坏味道清单(A 类特有)

1. ❌ **状态里直接调 `Input.*`** —— 绕过意图层,缓冲/AI 复用/回放测试全部失效
2. ❌ **`is_action_just_pressed` 出现在转换条件里** —— "按键被吃"bug 的签名写法(§4.1 理由①)
3. ❌ **窗口时长硬编码在状态代码里**(§4.2)
4. ❌ **倒计时变量每帧手动 -= delta** —— 用"记录时刻+比较"替代
5. ❌ **animation_finished 信号驱动规则转换** —— 规则时序被表现绑架;相位推进只认帧数据游标
6. ❌ **动画轨道回调里改血量/状态**(§4.5 例外通道的滥用)
7. ❌ **决策与动作合体**:状态名出现 `chase_attack`、`patrol_walk` —— B 类逻辑渗入,拆机(§1.3)
8. ❌ **转换检查无显式排序**,依赖代码书写顺序的巧合(§2.3)

## 7. 测试要点

A 类可测性最差(手感最终靠人试),但下列部分**必须**自动化——它们恰是回归 bug 的高发区:

1. **假身体注入**:ctx 的身体接口换成脚本控制的 stub,is_on_floor/velocity 按帧脚本给定——所有窗口测试的前提
2. **窗口边界测试**:土狼时间——离地后第 `window-ε` 秒按跳断言起跳、第 `window+ε` 秒断言不起跳;输入缓冲、无敌帧同法。**每个宽限窗口两条边界用例,是 A 类测试的主体**
3. **意图消费测试**:一次跳跃请求断言恰好触发一次跳(消费语义,§4.1 ②)
4. **抖动测试**:构造阈值附近的速度序列,断言 N 帧内转换次数 ≤ 上限
5. **帧数据一致性校验**(数据测试而非逻辑测试):遍历所有招式 Resource,断言三相帧数 > 0、取消窗口落在总时长内、into 引用的状态存在——内容侧填错表是动作系统最日常的故障源
6. **输入脚本回放**:录制"按帧输入序列 → 期望状态序列"的用例集,重构状态机后重放——A 类的回归安全网

## 8. AI 代理协作约定(供 CLAUDE.md 摘录)

1. 实现角色状态机前,先产出:状态清单、每状态出边声明(含优先级排序)、宽限窗口参数表(归定义层),经确认后动代码。
2. 状态对象为纯 RefCounted;身体访问一律经 ctx 的接口;状态内出现 `Input.`、`get_node`、`$`、AnimationPlayer 查询即视为实现错误。
3. 所有瞬时输入走意图层(时间戳 + 消费语义);所有计时用"记录时刻 + 比较"。
4. 攻击类状态一律实现为帧数据解释器,相位用状态内游标;取消规则只允许存在于帧数据中。
5. 交付必须包含假身体 stub、全部窗口的边界用例、帧数据一致性校验;新增招式只允许新增 Resource,不允许改状态代码。
