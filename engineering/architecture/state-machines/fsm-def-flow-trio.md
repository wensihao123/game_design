# FSM 细则 D+E+F:流程三件套(游戏流程 · UI 屏幕 · 回合阶段)

> **索引归属**:`fsm-index.md` §2 中的 D、E、F 三类,合并成篇。合并理由:三者最高价值的内容是**相互边界**(必须并排讲)、共享**状态栈**与**阶段序列**两个结构工具、且运行时是物理协作的三台机。共享概念引用索引 `[§3.x]`,不重复。
>
> **挂载点声明**:F 类的"响应窗口与打断栈"(深度回合制:优先权、结算栈)本篇只讲到管理/模拟游戏够用的深度;需要时增补 `fsm-f-advanced-interrupts.md` 挂载于 §5.3。

---

## 1. 定位:三台机,不是一台

| | D 游戏流程 | E UI/屏幕 | F 回合/阶段 |
|---|---|---|---|
| 回答的问题 | 游戏处于什么**大阶段** | 玩家眼前是**哪层界面** | 规则推进到**哪一步** |
| 节奏 | 事件驱动 | 事件驱动 | tick/事件驱动 |
| 层 | 逻辑层(应用级) | **表现层** | 逻辑层(规则级) |
| 状态是否进存档 | 部分(在哪个模式) | **从不** | **必须** |
| headless 下是否存在 | 骨架存在 | **不存在** | 完整存在 |
| 变更频率 | 极低 | 高(随 UI 迭代) | 中(随规则迭代) |
| 典型状态数 | 5–8 | 屏幕数 × 层 | 3–8 个阶段 |

三者坐标各异,却总被揉进一台 `GameManager` 巨型状态机——这是流程类代码腐烂的头号原因(索引 §2 已预警)。本篇第一要务就是给出可操作的拆分标准。

---

## 2. 拆分标准:三个石蕊测试

判断一个"状态"归 D、E 还是 F,依次问:

**测试 1 —— 存档测试:读档后它必须恢复吗?**
必须逐字恢复 → F(回合数、当前阶段、优先权归属)。只需恢复到"某个大阶段" → D(读档后回到局内,但不必回到读档瞬间的那帧动画)。完全不需要恢复、可从逻辑状态重建 → E(读档后不需要恢复"设置面板开着且滚动到第三页")。

**测试 2 —— headless 测试:没有屏幕时它还有意义吗?**
自动化测试跑一整局游戏,F 的每个阶段都会被走到;D 剩下一个骨架(至少"局内/局外"的区分还在);E 彻底蒸发。**凡是 headless 下蒸发的状态,一律归 E,一律不许写进规则代码。**

**测试 3 —— 变更来源测试:谁的需求会改动它?**
数值/规则策划改 F,交互/UI 设计师改 E,只有"游戏有哪些大模式"这种立项级决策改 D。按模块边界篇的"变更同频"原则,三种变更来源就该对应三个模块。

### 两个经典案例过一遍测试

**"暂停"归谁?** 三台机里的形态完全不同:D 中是**栈上叠加的一层**(不是替换——局内状态还在,只是被压住,见 §3.2);E 中是一层屏幕(暂停菜单被 push);F 中**根本不存在**——规则层没有暂停概念,tick 不来它就静止,它甚至不知道自己被暂停了。把"暂停"写进规则层(如全局 `is_paused` 标志被结算代码检查)是三机混淆的典型症状。

**"结算界面"归谁?** 判断:关掉这个界面后,游戏规则是否处于不同阶段?是——"结算"是 D 的一个状态(局已结束,不可能回到局内),界面只是该状态的 E 侧投影;否(比如只是回顾面板,关掉继续玩)——纯 E,D/F 无感知。

## 3. D:游戏流程状态机

### 3.1 典型形态

```
boot → main_menu → loading → in_game → results → main_menu
                      ↑__________________________|  (再来一局)
```

转换表中心式 `[§3.3]`,状态极少、图极稳定。两个 D 类独有的机制:

### 3.2 暂停是 push,不是 transition

暂停时局内状态原封不动地留在栈底,暂停层压上去;恢复即 pop,零状态重建。这要求 D 用**状态栈** `[§3.5]` 而非平面 FSM,且状态接口比标准四件套多两个钩子:

```gdscript
class_name StackState extends RefCounted
func enter(ctx) -> void: pass    # 首次入栈
func exit(ctx) -> void: pass     # 出栈销毁
func pause(ctx) -> void: pass    # 被新状态压住(局内状态在此停 tick、停输入)
func resume(ctx) -> void: pass   # 压住它的状态弹出,重新成为栈顶
```

纪律:**暂停的实现是"栈顶状态停止驱动 F 的 tick",绝不是 `Engine.time_scale = 0` 之类的全局黑客**——后者会连 UI 动画、菜单过渡一起冻死,而且让"暂停时允许查看信息面板"这类需求无法实现。

### 3.3 异步加载是一等状态,不是转换瞬间

`main_menu → in_game` 之间必然隔着资源加载,而加载耗时不可假装为零。错误做法:在转换 action 里同步加载(卡死一帧到数秒)。正确做法:`loading` 是一个真实状态——enter 时发起 `ResourceLoader.load_threaded_request`,每帧轮询进度,完成事件触发 `loading → in_game`。这个状态天然承载进度条(E 侧投影)、失败重试(loading → error 转换)、最短展示时长(避免闪屏)。**任何"转换需要时间"的场合,把过程升格为状态**——这条原则 E 类(§4.3)还会用到一次。

### 3.4 状态即场景的映射纪律

D 的 enter/exit 通常对应场景树大结构的换装(`change_scene_to_packed` 或手动挂卸子树)。纪律:场景操作**只出现在 D 状态的 enter/exit 里**,项目中任何其他地方出现 `change_scene` 都是味道——流程控制权被旁路了。D 机本体仍是纯对象,由 main 场景(组合根)持有并把场景操作以 Callable 注入,保持 D 的可测性:headless 测试注入空操作即可走完全部流程转换。

## 4. E:UI/屏幕状态机

### 4.1 本体是"屏幕栈 + 模态层",不是平面 FSM

UI 导航的原生结构是栈:主界面 push 设置,设置 push 按键绑定,返回键 = pop。平面 FSM 表达"从任意屏幕返回上一屏"需要 N² 条转换,栈只需要一个 pop——E 类是状态栈 `[§3.5]` 的第一号用户,识别信号(`previous_state` 字段)在 UI 代码里出现率最高。

模态弹窗(确认框、奖励结算)是栈上的又一层,或独立的第二个栈(屏幕栈 + 模态栈),取决于"弹窗是否允许跨屏幕存活"。多数游戏一个栈够用:弹窗就是一层不满屏的 push。

### 4.2 返回键语义集中化

返回键(Android back / ESC / 手柄 B)的处理**只写一处**:E 机的 `handle_back()` —— 栈顶先获得拦截机会(编辑中的表单弹"放弃修改?"),不拦截则 pop,栈见底则转交 D(弹"退出游戏?")。每个屏幕各自监听输入事件处理返回,是 E 类头号味道:行为不一致、拦截逻辑重复、栈状态与实际显示漂移。

### 4.3 过渡状态防双输入

屏幕切换有动画时长,动画期间玩家的连点会触发"对着正在消失的屏幕操作"或"重复 push"。解法就是 §3.3 那条原则的复用:**opening/closing 是屏幕的真实子状态**,过渡期间 E 机拒收导航命令(或入队延迟处理)。简单实现:栈操作期间置 `transitioning` 标志,配合 tween 的 finished 信号清除——但要意识到这个标志就是一台微型二态 FSM,连点 bug 多的项目值得把它显式化。

### 4.4 E 永远是观察者

E 的屏幕从 D/F 的信号读取"该显示什么",通过命令接口(§6)表达"玩家想做什么"。E **持有零份规则真相**:金钱数字是订阅来的,按钮灰不灰是查询 `can_xxx()` 得来的。检验:删光整个 E 层,游戏应能在 headless 下用脚本指令完整跑一局——这同时就是测试 2 的工程兑现。

## 5. F:回合/阶段状态机

### 5.1 先做退化检查:多数 F 是序列,不是图

`收入结算 → 效果应用 → 事件判定 → 通知发布` 这类阶段流,若严格线性、无分支、无回边,按 `[§3.6]` 用**阶段游标 + 阶段表**表达,别上全套 FSM:

```gdscript
const PHASES: Array[StringName] = [&"income", &"effects", &"events", &"notify"]
var cursor := 0

func advance() -> void:
    _run_phase(PHASES[cursor])
    cursor = (cursor + 1) % PHASES.size()
    if cursor == 0: tick_completed.emit()
```

阶段表是数据 → 阶段顺序可配置、可插拔(mod 注入新阶段)、可单测每个阶段。**只有出现条件分支**(战斗阶段仅在有接敌时进入)或**回边**(响应循环)时,才升级为真正的转换表 FSM。

### 5.2 推进驱动者:自动 vs 确认,必须显式建模

阶段推进由谁触发是 F 的核心设计参数:纯自动(模拟游戏的 tick 内阶段,一口气跑完)、玩家确认(回合制的"结束回合"按钮)、混合(自动推进但某些阶段等待玩家输入)。混合模式的正确建模:**"等待玩家"本身是一个阶段状态**,它的出边事件是玩家命令——而不是在自动流程里散布 `await` 或轮询标志。这样存档落在等待阶段时,读档后 F 机自然停在原地等待,无需特殊恢复逻辑。

### 5.3 响应窗口与打断(浅讲 + 挂载点)

管理/模拟游戏偶尔需要"阶段间插入结算"(事件判定阶段触发的灾害,需要在通知阶段前插入一次损失结算)。够用的方案:**阶段间隙的插入队列**——任何系统可向 F 机投递"插入阶段请求",F 在当前阶段结束后、游标推进前依次执行插入项。这保持了阶段原子性(插入不打断进行中的阶段)且顺序确定(队列 FIFO)。

深度回合制(优先权传递、可被响应的响应、后进先出的结算栈——即 TCG 的 stack)复杂度高一个量级,超出本篇范围。**挂载点**:`fsm-f-advanced-interrupts.md`(未建)。

### 5.4 确定性与存档

F 是三台机中唯一要求**完全确定性**的:同一存档 + 同一命令序列 = 同一结果(回放调试、异步对战校验的前提)。纪律与 C 类 4.3 同源:阶段内遍历顺序确定、随机数走带种子的专用 RNG(种子进存档)、禁止读墙钟时间。存档内容:`{cursor 或当前阶段名, 回合数, RNG 状态, 插入队列}`——比 C 类多出 RNG 与队列两项。

## 6. 三机协作协议

三台机的依赖方向是"call down, signal up"在**机器粒度**的重演:

```
        命令(方法调用)              命令
   E ──────────────────→ D ←──────────────── E
   ↑                      │
   │ 信号                  │ 驱动(启停 tick / 装配)
   │                      ↓
   E ←──────────────────  F
        信号(phase_changed, turn_ended, ...)
```

- **D → F**:D 拥有 F。进入 in_game 时装配 F 机,暂停层压栈时停止向 F 供给 tick,退出局内时销毁 F。F 不知道 D 存在。
- **D → E / F → E**:仅通过信号。D 的 `flow_changed` 驱动 E 的整屏换装;F 的 `phase_changed`、`turn_ended` 驱动 E 的横幅、按钮可用性刷新。**D/F 的代码里没有任何 UI 概念**(职责边界的"连知道都不该知道")。
- **E → D / E → F**:仅通过命令接口(`request_pause()`、`end_turn()`),且命令是**意图**不是状态写入——D/F 有权拒绝(guard 不满足),E 靠随后的信号(或命令返回的 bool)得知结果。UI 按钮的可用性用查询(`can_end_turn()`)预判,但**最终裁决权永远在 D/F**:哪怕按钮忘了置灰,非法命令也会被 guard 挡下并触发告警(与 C 类 6.4 同款的 bug 探测器)。

装配全部发生在组合根:main 场景构造 D,注入场景操作 Callable;D 在进入局内时构造 F;E 层节点在 _ready 时向 D/F 订阅。**三台机相互之间零 preload、零直接引用,依赖图三个节点两条单向边,一页纸测试(边界篇测试 6)满分通过。**

## 7. 坏味道清单(三件套特有)

1. ❌ **一台 GameManager 同时含 main_menu、settings_panel、combat_phase 三种粒度的状态** —— 三机混淆的直接证据,用 §2 三测试拆
2. ❌ **`Engine.time_scale = 0` 或全局 is_paused 标志实现暂停**(§3.2)—— 规则层被表现关切污染,且冻死 UI 自身
3. ❌ **转换 action 里同步加载资源**(§3.3)
4. ❌ **D 机之外出现 change_scene 调用**(§3.4)
5. ❌ **各屏幕各自处理返回键**(§4.2)
6. ❌ **UI 直接写规则状态**(绕过命令接口)或**规则代码引用 UI 节点** —— 六边形依赖里唯二绝对禁止的两条边
7. ❌ **自动阶段流里 await 玩家输入**(§5.2)—— "等待玩家"该是显式阶段
8. ❌ **F 阶段代码读墙钟 / 用全局 randi()**(§5.4)—— 确定性破产

## 8. 测试要点

- **D**:headless 注入空场景操作,走完全部流程转换含 loading 的失败重试分支;暂停压栈/弹栈往返后断言局内状态零变化
- **E**:栈不变量测试(空栈 pop、重复 push、过渡期导航命令被拒);"删光 E 层 headless 跑一局"作为持续集成中的架构守护测试
- **F**:表驱动 `[§3.8]` 之上——回放测试(同存档同命令序列双跑,断言逐 tick 状态哈希一致);等待阶段存档往返测试(§5.2);插入队列顺序测试(§5.3)
- **协作**:命令拒绝测试——E 发出 guard 不满足的命令,断言 D/F 状态不变且告警触发

## 9. AI 代理协作约定(供 CLAUDE.md 摘录)

1. 任何"流程/界面/回合"需求,先用 §2 三测试判定归属 D/E/F,在实现说明中写明判定结果,再动代码。
2. D 与 F 为纯 RefCounted;场景操作经 Callable 注入 D;F 的随机数一律走注入的带种子 RNG。
3. 暂停一律实现为 D 栈 push + 停供 tick;禁止 time_scale 黑客与全局暂停标志。
4. E 层所有导航走屏幕栈统一入口;返回键处理只允许存在于 E 机一处。
5. E→D/F 只许调用命令接口;发现 UI 直接写规则状态或规则代码出现 UI 引用,停止实现并上报。
6. 线性阶段流用游标+阶段表,不上 FSM;出现分支/回边时才升级,并在设计文档记录升级理由。
