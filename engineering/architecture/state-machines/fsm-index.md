## 1. 分类坐标系

所有状态机用两个轴定位。**实现选型、转换归属、测试方式的差异,根源都在这两个轴上。**

### 轴一:评估节奏 —— 这台状态机多久"思考"一次?

| 节奏 | 特征 | 典型 |
|------|------|------|
| 帧驱动 | 每帧/每物理帧评估,状态内含 update 逻辑 | 角色动作、动画、输入识别 |
| 事件驱动 | 平时静止,收到事件才可能转换 | 生命周期、流程、叙事、网络 |
| tick/回合驱动 | 在离散结算时刻批量评估 | 阶段状态机、模拟实体批量转换 |

### 轴二:所在层 —— 它是规则,还是表现/基础设施?

| 层 | 状态的性质 | 序列化 |
|----|-----------|--------|
| 逻辑层 | 状态是"真相" | **进存档** |
| 表现层 | 状态是逻辑的视图,可随时重建 | 不进存档 |
| 基础设施层 | 与游戏内容无关的技术状态 | 不进存档 |

> 文档就绪后将"☐ 规划中"改为"✅ yyyy-mm-dd"。

### 难度与价值标注

- **重点四类**:A(难在手感细节)、B(难在复杂度管理,是通往行为树/效用 AI 的门户)、C(难在规模与序列化)、G(难在数据驱动与版本演化)
- **模式固定三类**:D、E、F —— 掌握"状态栈"与"阶段序列"两个工具后基本无悬念
- **单条纪律一类**:H —— 它是视图不是真相
- **成熟套路两类**:I、J

### 高危误用警示

**D/E/F 揉成一台巨型状态机**("游戏流程"混"UI 导航"混"回合阶段")是流程类代码腐烂的头号原因。拆分标准见各自细则,速记版:D 管"游戏处于什么大阶段",E 管"玩家眼前是哪层界面",F 管"规则推进到哪一步"——三者节奏、层、生命周期全都不同。

## 3. 全部细则共享的基础概念(约定于此,细则不再重复)

### 3.1 实现光谱四档

| 档 | 形态 | 适用 | 失效信号 |
|----|------|------|----------|
| 1 | enum + match | 状态 ≤ 4、每状态逻辑 ≤ 10 行 | match 分支超一屏 / 出现嵌套状态判断 |
| 2 | 状态对象(纯 RefCounted) | 逻辑层 FSM 的默认选择 | — |
| 3 | 节点式(状态即 Node) | **仅限表现层**状态 | 规则写进节点状态 = 边界破裂 |
| 4 | 数据驱动(Resource/JSON + 通用解释器) | 内容性状态机(G 类为主) | — |

**四档并存是正常且正确的。用同一套 FSM 框架统一全项目是架构洁癖,禁止。**

### 3.2 状态对象无状态化纪律

所有可变数据放 ctx(实体运行时状态),状态对象只有行为 → 状态实例可享元共享;存档只需序列化"当前状态名"(StringName)。**任何把易变数据锁进状态对象的写法,在存档系统面前崩溃——存档兼容性是一票否决项。**

### 3.3 转换归属:默认转换表中心式

- **转换表中心式**(默认):合法转换集中一张表,`[from, event] -> to`。转换图可审计、天然可数据化、状态间零依赖、可直接生成 Mermaid 状态图、测试可从表机械推导。
- **状态自治式**(例外):转换逻辑写在状态内部。仅用于帧级动作 FSM(转换条件为连续物理量判断)。

### 3.4 guard / action / enter / exit 职责四分

| 部件 | 职责 | 纪律 |
|------|------|------|
| guard(ctx) → bool | 转换前置条件 | **必须纯查询,零副作用** |
| action(ctx) | 转换时副作用 | **唯一允许改状态的地方** |
| enter(ctx) | 本状态初始化 | 不触发转换 |
| exit(ctx) | 本状态清理 | 不触发转换 |

### 3.5 三个正交升级(遇到识别信号才引入,不预支)

| 升级 | 解决 | 识别信号 |
|------|------|----------|
| 层级状态机 HSM | 转换爆炸("任意状态收到 X 都去 Y") | 转换表出现大量同目标的任意态转换 |
| 状态栈(下推自动机) | 返回/恢复语义 | 状态里出现 `previous_state` 字段 |
| 并行状态机 | 正交维度的状态积爆炸 | 状态名出现下划线连接的双形容词 |

### 3.6 FSM 的放弃条件与低估病

- **该放弃**:① 维度深度耦合、组合不可枚举 → 效用 AI / 行为树;② 状态名变成 `phase_1/2/3` 且严格线性 → 用 int 游标 + 数据表(那是序列不是状态机)
- **该引入**(反向病):三个以上相关 bool 标志位(`is_a`、`is_b`、`is_c` 的组合大半非法)→ 立即重构为 enum 状态,回报最快的重构之一

### 3.7 分层归属(逻辑层 FSM 的标准拆解)

| 部件 | 归属层 |
|------|--------|
| 状态定义、转换表 | 静态定义层(Resource/JSON) |
| 当前状态值(StringName) | 运行时状态层(随存档) |
| FSM 引擎、guard、action | 系统层(纯 RefCounted,headless 可测) |
| 状态变化的表现响应 | 节点层(订阅 `state_changed(id, from, to)`) |

### 3.8 测试策略:表驱动

遍历转换表,对每条边构造满足/不满足 guard 的 ctx,断言转换发生/被拒;补非法事件断言被拒绝。**一台 FSM 的测试可从其转换表机械生成。**

## 4. 细则文档的统一结构模板

每篇 `fsm-*.md` 按以下骨架撰写,保证横向可对比:

1. **定位** —— 在两轴坐标系中的位置、与相邻类别的边界
2. **状态与转换的典型形态** —— 示例状态集 + 转换表 + Mermaid 图
3. **实现选型** —— 在四档光谱中的推荐档位及理由
4. **特色问题精讲** —— 本类独有的工程难题(每类 2–4 个)
5. **升级路径** —— 何时需要 HSM/栈/并行,或何时叛逃到其他范式
6. **坏味道清单** —— 本类特有的味道
7. **测试要点** —— 在表驱动基线上的本类补充
8. **AI 代理协作约定** —— 供 CLAUDE.md 摘录的实现纪律

```

   ↓

   ↓
H / I / J          ← 短篇收尾
```

## 6. 术语表

| 术语 | 定义 |
|------|------|
| ctx(上下文) | 实体的运行时状态对象,FSM 各部件读写数据的唯一载体 |
| 转换表 | `[from_state, event] -> to_state` 的集中映射,可挂 guard/action |
| guard | 转换前置条件,纯查询 |
| action | 转换时副作用,唯一改状态点 |
| 享元 | 状态对象无状态化后全局共享实例的模式 |
| HSM | 层级状态机,子状态不处理的事件冒泡给父状态 |
| 状态栈 | push/pop 语义的下推自动机 |
| 并行机 | 一个实体同时运行多台正交 FSM |
| 状态抖动 | 在两个状态间高频横跳(B 类高发) |
| 状态积 | 多维度硬合并产生的 M×N 组合状态(并行机所消除的对象) |
