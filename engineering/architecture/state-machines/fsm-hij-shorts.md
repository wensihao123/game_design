> **索引归属**:`fsm-index.md` §2 中的 H、I、J 三类。三者各自内容不足以撑起完整八节模板(H 的核心已在 A 篇 §4.5 展开大半,I/J 是模式高度成熟的工程套路),故合并为短篇集,每篇采用精简五节:定位 / 核心纪律 / 特色问题 / 坏味道 / 测试。共享概念引用索引 `[§3.x]`。

# H:动画状态机

## H1. 定位

帧驱动 · **表现层**。载体通常是 Godot 的 `AnimationTree`(内建状态机/混合树),而非手写 FSM——本篇讲的不是怎么再造一台,而是**这台内建机器与逻辑层的关系纪律**。

## H2. 核心纪律(全篇只有这一条,其余皆推论)

**H 是逻辑状态的视图,不是真相。** 完整表述三款:

1. 动画状态可随时从逻辑状态**重建**——存档不存它,读档后由逻辑状态推导初始动画;掉帧、变速、跳过都不影响规则
2. 数据流单向:逻辑 `state_changed` → H 侧换装;H 侧**永不**反向驱动逻辑转换(`animation_finished` 触发规则转换是 A 篇坏味道 5)
3. 例外通道(动画轨道回调)只许触发表现——脚步声、粒子锚点可以,`deal_damage()` 绝对不行(A 篇 §4.5)

## H3. 特色问题

**(a) 映射非一一,且映射逻辑归 H 侧。** 逻辑一个 `active` 态可对应 H 侧多个表现(忙碌的几种随机小动作);逻辑的 jump/fall 两态可能共用 H 侧一棵按竖直速度混合的 BlendSpace。翻译表(逻辑状态 → 动画节点名/混合参数)写在表现层适配器里,这正是模块边界篇"翻译责任在消费方"的动画版。逻辑层代码里出现动画名字符串,即为翻译责任倒灌。

**(b) 过渡与打断的表现自治。** 交叉淡入时长、能否被打断、打断时从哪帧接续——全部是 H 侧参数(AnimationTree 的 transition 配置),逻辑层无感知。诱惑出现在"逻辑想等动画播完再转换"的需求上:正解是逻辑侧用自己的时长参数(定义层,A 篇帧数据同源),**两条时间线平行、共享数据、互不查询**。

**(c) 一次性动作的完成表现。** 开宝箱、受击闪白这类 one-shot:逻辑瞬间完成(状态转换 + 效果结算在同一 tick),表现随后慢慢播。反过来让逻辑等表现(结算延迟到动画结束)会把表现时长变成规则参数——网络同步、快进、headless 全部遭殃。需要"表演期间不可再交互"时,那是 E 类的过渡状态(DEF 篇 §4.3)或输入锁,仍不是逻辑状态。

## H4. 坏味道

1. ❌ 动画状态进存档
2. ❌ `animation_finished` 连到规则代码
3. ❌ 逻辑层出现动画名 / H 适配器之外出现 AnimationTree 引用
4. ❌ 结算等动画(H3c)
5. ❌ 用 AnimationTree 状态机承载游戏逻辑(在编辑器里写 guard 表达式查血量之类)——工具好用不代表层对

## H5. 测试

H 侧基本靠眼睛验收,自动化只守边界:headless 全量跑逻辑测试即为"H 可整体缺席"的持续证明(DEF 篇测试 2 同款);翻译表完整性校验——遍历逻辑状态枚举,断言每个状态在翻译表中有映射(内容侧加状态忘配动画的日常错误)。

# I:网络/会话状态机

## I1. 定位

事件驱动 · 基础设施层。`disconnected → connecting → handshaking → connected → reconnecting` 与游戏内容零耦合,是十类中最接近通用软件工程的一台——也因此最值得一次写对、跨项目复用。

## I2. 核心纪律

**每个状态必须回答两个问题:此刻断线怎么办?此刻超时怎么办?** 网络机的完备性不看快乐路径,看这两列是否填满——转换表 `[§3.3]` 在此的价值就是把这两列变成**可检查的空格**:每行的 `on_disconnect` / `on_timeout` 出边不允许留空(哪怕填"忽略",也要显式填)。

## I3. 特色问题

**(a) 超时即事件。** 每个非稳态(connecting/handshaking/reconnecting)入态时向时间系统注册超时,到期发 `timeout` 事件走正常转换——不在状态内轮询计时。实现与 C 篇 §4.2 的 tick 计数同族:时刻记录 + 比较,退避时长归配置层。

**(b) 重连是有预算的循环。** `reconnecting` 的 ctx 携带 `attempt: int`,指数退避(1s/2s/4s/…封顶),预算耗尽转 `failed` 交给上层(D 机弹"连接丢失"界面)。纪律:**attempt 在进入 connected 时才清零**,在 reconnecting 内部自增——清零时机放错(每次进 reconnecting 清零)会造成无限重连。

**(c) 传输态与会话态分离。** "TCP/WebSocket 连着"与"我已登录且在房间里"是两台机:传输机(本篇)管连接生命周期;会话机(登录/大厅/房间/对局)是它的上层住户,传输断开时会话机整体挂起,重连成功后走**会话恢复**(带 token 重认证 + 状态同步),而不是从头登录。两机关系又是"D 拥有 F"模式(DEF 篇 §6)的重演。合并二机的症状:reconnected 之后不知道该回到哪——因为会话上下文被传输机的转换冲掉了。

**(d) 半连接防御。** handshaking 卡死(对端不应答也不断开)是超时事件存在的理由;connected 态则靠心跳(定期 ping,连续 N 次无 pong 视同 `disconnect`)把"僵尸连接"翻译成事件。心跳参数归配置,心跳逻辑归传输机,**上层永远只见离散事件**。

## I4. 坏味道

1. ❌ 转换表存在未填 on_disconnect/on_timeout 的行(I2)
2. ❌ 重连计数清零时机错误(I3b)
3. ❌ 游戏代码直接查 socket 状态——只许消费传输机的信号与状态
4. ❌ 传输/会话一台机(I3c)
5. ❌ 重连成功后从头走登录流程(会话恢复缺失)

## I5. 测试

I 类是十类中最适合表驱动 `[§3.8]` 穷举的:状态 × {disconnect, timeout, 服务器拒绝, 成功} 全矩阵,每格断言目标状态;假传输层(脚本控制的收发/断开时序)驱动;重点用例:退避序列断言、预算耗尽转 failed、handshaking 中途断线、connected 心跳超时、会话恢复后上层状态完整。

# J:输入识别状态机

## J1. 定位

帧驱动 · 基础设施层。把原始输入流识别为语义指令:双击、长按、拖拽、搓招、连招。十类中最接近教科书自动机的一类——本质是**对带时间戳的输入序列做模式匹配**。产出喂给 A 类的意图层(A 篇 §4.1)或 UI。

## J2. 核心纪律

**识别器消费原始输入、产出语义指令,两侧都不许穿透。** 上层(A/E)只见 `dash_command`、`double_click`,不见"0.2 秒内两次 pressed";识别器不知道指令被谁消费。识别器本身是纯对象:输入 = (动作, 时刻) 流,输出 = 指令事件——天然 headless,天然可用录制序列回放测试。

## J3. 特色问题

**(a) 一切识别都是"模式 + 时间窗"。** 双击 = 两次 press 间隔 ≤ 0.3s;长按 = press 后 ≥ 0.5s 无 release;搓招 = ↓↘→ + 拳,相邻输入间隔 ≤ 0.25s。识别器的通用形态:当前匹配位置游标 + 上次输入时刻,新输入到达时——匹配下一位则推进,超窗或不匹配则回退。**窗口参数全部归配置层**(A 篇 §4.2 同款理由:手感调参高频)。

**(b) 失配回退:重置还是重用?** 简单指令失配直接归零。搓招类要小心"失配输入可能是新模式的开头":↓↓↘→ 中第二个 ↓ 对 ↓↘→ 而言,朴素重置会漏识别——失配时应检查当前输入能否作为模式首元素重新开始(KMP 思想的简化版)。指令集大了之后,逐指令独立识别器并行跑(见 c)比单台合并大自动机好维护得多——合并机就是状态积 `[§3.5]` 的 J 类版本。

**(c) 并行识别与裁决。** 双击与单击、长按与短按天然竞争:单击必须**等双击窗口关闭才能确认**(0.3s 内无第二击 → 发单击),这是"单击有延迟"的原理性代价,窗口参数即手感与响应的交易点。多识别器同时命中时需要裁决器:按优先级(更长模式优先:↓↘→+拳 压过 →+拳)且**指令被消费即吞掉构成它的输入**,防止一次搓招同时触发波动拳和前进。

**(d) 微型 FSM 的正名。** 长按/拖拽这类三五态小机(idle → pressing → holding/dragging)是第 1 档 enum+match `[§3.1]` 的理想住户,也是"三个 bool 该并成一个 enum"(索引 §3.6 反向病)在输入代码里的高发区——`is_pressing`、`maybe_drag`、`long_fired` 三标志齐飞的触摸处理代码,人人见过。

## J4. 坏味道

1. ❌ 上层代码直接做时序判断(A/E 里出现"距上次点击 < 0.3s")——识别逻辑外溢
2. ❌ 窗口参数硬编码
3. ❌ 多指令合并单台大自动机(J3b)
4. ❌ 识别命中不吞输入(J3c)
5. ❌ 布尔标志群模拟微型 FSM(J3d)

## J5. 测试

J 类可测性全场最佳:纯函数式(时间戳序列进、指令序列出)。用例形态即"输入剧本":`[(down,0), (up,0.1), (down,0.25), (up,0.3)] → [double_click]`;必测边界——窗口内 ±ε 双例(A 篇 §7.2 同款)、失配重用序列(↓↓↘→)、竞争裁决(搓招完成断言不发方向指令)、乱序压力(随机序列断言不崩溃不误报)。识别器回归 = 剧本库全量重放。

## 全系列收束

十类至此全部完成。回望全局,三条贯穿性的线:

1. **"进不进存档"是最锋利的一刀**——它切开了逻辑与表现(H)、切开了 D/E/F、切开了对话与任务(G §1.3)。判断任何状态归属时,先问这一刀
2. **"X 拥有 Y"的机器组合模式反复出现**:D 拥有 F、C 拥有 B、传输机拥有会话机——上层机的状态决定下层机的生死,下层机对上层无知。这是模块边界思想在状态机粒度的标准形态
3. **每类都有自己的"时间形态"**:C 的 tick 计数、A 的宽限窗口、B 的决策节奏与承诺、I 的超时与退避、J 的匹配窗——但实现范式只有一个:**时刻记录 + 比较,参数归配置**。全系列没有一处每帧递减的倒计时变量
