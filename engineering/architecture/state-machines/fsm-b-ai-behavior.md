# FSM 细则 B:AI 行为状态机(AI Behavior FSM)

> **索引归属**:`fsm-index.md` §2 中的 B 类。共享概念引用索引 `[§3.x]`,不重复。
>
> **本类地位**:FSM 能力上限的试炼场。B 类是十类中唯一"用着用着就该换范式"的一类——本篇 §5 的叛逃路线图(FSM → HSM → 行为树 → 效用 AI)是全篇重心,也是索引 §3.6 放弃条件的完整展开。

---

## 1. 定位

### 1.1 坐标

- **节奏**:事件驱动为主(感知事件触发转换)+ 低频重评估(决策 tick,见 §4.2)。**不是每帧**——这是与 A 类的节奏差异
- **层**:逻辑层,进存档(至少状态名 + 黑板关键项)

### 1.2 覆盖范围与本质

敌人/NPC 的决策层:巡逻/警戒/追击/攻击/逃跑/返回。B 回答"这个单位**想干什么**",不回答"身体怎么执行"(A 类)也不回答"这个单位处于什么生存阶段"(C 类)。一个完整的 AI 单位典型配置是三机叠加:

```
C 生命周期机(在城/在岗/离职/死亡)      ← 存在层
  └─ B 行为机(巡逻/警戒/追击)          ← 决策层,仅在 C 的特定状态下运行
       └─ A 动作机(idle/run/attack)    ← 执行层,消费 B 写入的意图
```

B→A 的通道已在 A 篇 §4.1 预埋:B 不调用 A 的任何方法,只写 A 的**意图接口**(move_axis、request_attack)——玩家输入与 AI 决策对 A 完全同形。

### 1.3 与相邻类别的边界

| 相邻类 | 区分标准 |
|--------|----------|
| A 动作 | 见上。合并症状:状态名出现决策×动作复合词(`chase_jumping`) |
| C 生命周期 | C 管存在("这个 NPC 还在不在、处于哪个人生阶段"),B 管此刻意图。B 机是 C 机某些状态的**内部住户**:NPC 进入"在岗"态时 B 机装配,离岗时销毁——与 D 拥有 F 的关系(DEF 篇 §6)同构 |
| J 输入识别 | 无关系。但注意对称性:玩家单位的"B 位"被输入系统占据 |

## 2. 状态与转换的典型形态

### 2.1 典型状态集(守卫型敌人)

```
patrol      沿路径巡逻
alert       察觉异常,前往调查(有目标位置,无确认目标)
chase       确认目标,追击
attack      进入射程,攻击循环
search      丢失目标,搜索最后已知位置
flee        血量/士气过低,脱离
return      放弃,返回岗位
```

alert 与 search 常被漏掉,但它们是 AI"显得有脑子"的关键:直接 patrol↔chase 二态的敌人像开关,加入"怀疑"与"不甘心"两个中间态后,同一套感知数据立刻产生生物感。**B 类的状态设计本质是给玩家表演决策过程**——每个状态是一段可被玩家读懂的"心理独白"。

### 2.2 转换事件来自感知层,不来自世界本身

B 的转换事件是 `target_spotted`、`noise_heard`、`damage_taken`、`target_lost`——全部是**感知系统的产出**,不是世界事实。"玩家在 10 米内"是世界事实;"我看见了玩家"才是 B 的事件(中间隔着视锥、遮挡、光照、感知延迟)。这个分离在 §4.1 展开,是 B 类架构的第一块基石。

### 2.3 实现选型

**推荐:第 2 档状态对象 + 转换表中心式 `[§3.3]`(回归默认,不沿用 A 的例外)。** 理由:B 的事件经感知层翻译后已是离散消息,正是转换表的主场;且 B 的转换图是策划要反复审阅调整的对象("受击后该不该无条件转追击?"),可审计的表 + 自动生成的 Mermaid 图价值极高。周期性条件(血量低于阈值)统一由决策 tick(§4.2)翻译成事件(`health_critical`)入队,维持"引擎只有一条 process 路径"(与 C 篇 2.4 同构)。

## 3. 黑板:B 类的 ctx 有专名

B 的 ctx 惯称**黑板(blackboard)**,因为它的读写方是多方的:感知系统写入,B 状态读取并写入决策产物,A 机经意图接口消费。典型字段:

```gdscript
class_name AIBlackboard extends RefCounted
# 感知区(感知系统写,B 读)
var visible_target_id := -1          # 当前视野内目标,-1 无
var last_known_pos: Vector2          # 目标最后已知位置(记忆!)
var last_seen_at := -INF
var heard_noise_pos: Vector2
# 决策区(B 写)
var current_target_id := -1          # B 认定的目标(≠ 视野内目标)
var home_pos: Vector2                # return 的去处
# 意图区(B 写,A 读)——即 A 篇 §4.1 的意图接口
var move_target: Vector2
var attack_requested_at := -INF
```

两条纪律:① **感知区对 B 只读**——状态代码不许直接查世界(`get_overlapping_bodies` 出现在 B 状态里即为破界),一切世界信息经感知系统写入黑板,这是 B 可测性的全部来源(§7 用假感知喂黑板);② `visible_target_id` 与 `current_target_id` 分开——前者是此刻事实,后者是决策承诺,search 状态的存在就依赖这个区分(看不见了,但我还认定他)。

## 4. 特色问题精讲

### 4.1 感知系统:传感器 → 事件的翻译层

感知系统独立于 B 机,职责是把连续的世界状态翻译为离散事件 + 黑板更新:

- **视觉**:距离 + 视锥角 + 遮挡射线三重判定,通过则写 `visible_target_id` 并(在从无到有的边沿)发 `target_spotted`。注意**边沿触发**:事件只在状态翻转瞬间发一次,持续可见不刷屏——事件是导数,黑板是原函数
- **听觉**:声音源主动广播(脚步、枪声带半径),感知系统按距离衰减判定,写 `heard_noise_pos` 发 `noise_heard`
- **记忆**:目标离开视野时,`last_known_pos/last_seen_at` 保留——search 状态消费的就是这份记忆;记忆过期(超时)由决策 tick 发 `memory_expired`
- **感知延迟即难度参数**:从"物理可见"到"发出 spotted"之间插一段确认时间(0.2–1s,可随警戒状态缩短:alert 中的敌人眼尖),归定义层。瞬间反应的 AI 不是强,是假

### 4.2 决策节奏:AI 不需要每帧思考

B 机每帧评估既浪费也有害(见 §4.3 抖动)。标准做法:**决策 tick**——每个 AI 以 5–10Hz 重评估周期性条件,感知事件仍即时入队但可带反应延迟。两个免费收益:① **分帧(stagger)**:各 AI 的决策 tick 相位错开(entity_id 取模),一百个 AI 的思考成本均摊到各帧,消灭尖峰;② **AI LOD**:离玩家远/屏幕外的单位降到 1Hz 甚至挂起——B 机对评估频率的不敏感性(事件驱动核心)使 LOD 几乎零成本。这也是 B 与 A 分机的又一红利:A 必须每物理帧跑,B 不必,合并的机器只能按最高频率跑。

### 4.3 决策抖动:A 类抖动的高阶版

A 的抖动是阈值噪声,B 的抖动是**决策反复**,三种典型及解法:

| 抖动形态 | 症状 | 解法 |
|----------|------|------|
| 目标横跳 | 双目标距离接近时每 tick 换目标 | **换目标需超额优势**:新目标得分需高出现目标 20–30% 才切换(得分迟滞) |
| 追逃振荡 | 血量在 flee 阈值附近抖动导致 chase↔flee 循环 | 阈值迟滞(30% 进 flee,50% 才许回)+ flee 为**不可逆承诺**(必须走完到安全点) |
| 犹豫舞蹈 | alert↔patrol 高频往返 | **最小承诺时间**:enter 后 N 秒禁转出(黑板通用字段,转换 guard 统一附加) |

统一心法:**决策要有粘性**。人类玩家把 AI 的果断读作智能,把反复读作故障——哪怕反复在数学上"每 tick 都最优"。这条在 §5 效用 AI 处还会回来:效用架构天生易抖,迟滞与承诺机制必须随身携带。

### 4.4 目标失效:每个状态的必答题

B 类最高产的崩溃源:目标在 chase/attack 中途死亡/销毁/传送。纪律:**黑板只存 entity_id 不存对象引用**(悬空引用直接消灭),每次解引用经查询接口,查无此人即黑板置 `-1` 并向 B 机发 `target_invalid`——转换表里**每个持有目标的状态都必须有 target_invalid 出边**(通常 → search 或 return)。测试 §7.3 逐状态验证此事件,这条用例集抓住的 bug 数在 B 类测试中常年第一。

### 4.5 群体协调(简述)

多个 AI 共享目标时的经典问题:八个敌人同时扑脸。轻量解法是**令牌桶**:目标身上挂 N 个攻击令牌,B 机的 `attack` 入边 guard 是"申请到令牌",exit 归还——未申请到的自动留在 chase 外围(配合包围点分配即成"轮流上"的电影感)。令牌桶归属独立的协调系统(共享资源,与 C 篇 4.3 的电力网同类),不属于任何单个 B 机。深度群体战术(小队阵型、协同压制)超出 FSM 范畴,不展开。

## 5. 升级与叛逃路线图(本篇重心)

B 类的复杂度增长有清晰的阶梯,每级有明确的入场信号。**原则:只在信号出现时走下一级,且各级不互斥——成熟架构往往是组合(见 5.5)。**

### 5.1 第一级:平面 FSM(本篇主体)

守卫、哨塔、简单野怪的终点站。状态 ≤ 8、转换图一页能画下、行为无内部序列——**不要因为"AI 该用行为树"的教条离开这里**,行为树对这个复杂度是纯开销。

### 5.2 第二级:HSM `[§3.5]`

**信号**:转换表里 `damage_taken`、`target_spotted` 的出边在一半以上状态重复出现。上提为 combat/non_combat 两个父状态,公共出边写一次。B 类几乎必然走到这级,成本极低(C 篇通配 from 是它的穷人版)。**HSM 之后仍在膨胀,才考虑换范式**——很多"FSM 不行了"的项目其实只是没用 HSM。

### 5.3 第三级:行为树(BT)—— 当"状态"里长出了"步骤"

**信号**:某个状态的内部逻辑变成了带回退的序列——"攻击 = 走到射程→若被挡则绕→瞄准→开火→若弹尽则换弹→若目标近身则后跳"。FSM 状态是匀质的("处于攻击中"),序列 + 优先级回退是 BT 的原生结构(Selector/Sequence 每 tick 从根重评估)。

判断口诀:**FSM 回答"我处于什么模式",BT 回答"此模式下我按什么优先级尝试做事"**。两者不是竞争关系——最常见的正确形态是 FSM 的状态内挂 BT(chase 状态的 update 即"追击行为树"的 tick)。整树替代 FSM 的做法(用 BT 顶层 Selector 模拟模式切换)可行但丢失了转换表的可审计性与"模式"的存档友好性,不推荐默认采用。

### 5.4 第四级:效用 AI —— 当转换条件从"是否"变成"哪个更"

**信号**:出现"该追 A 还是 B?该战还是逃?取决于血量、距离、弹药、支援远近的综合权衡"——转换不再是布尔判定而是**多候选打分取最优**。这正是索引 §3.6 "维度深度耦合、组合不可枚举"的场景:FSM 要枚举组合,效用 AI 只需为每个候选行为写一个评分函数(各因素曲线相乘),天然处理连续权衡。

代价三条,入场前须知:① 可解释性坍塌——"为什么它逃了"从查转换表变成翻打分日志,**评分明细必须进调试面板**,否则不可调;② 天生易抖(每 tick 重打分)——§4.3 的迟滞与承诺机制原样搬入(当前行为加分、切换需超额优势);③ 曲线调参是新的复杂度税,策划工具跟不上时慎入。

### 5.5 组合形态:三层夹心(推荐的成熟架构)

```
外层 FSM/HSM:模式(non_combat / combat / flee)     ← 可审计、可存档、粘性强
  中层 效用打分:模式内选行为(选目标、选战术)        ← 处理"哪个更"
    内层 BT 或直接意图:执行所选行为的步骤序列        ← 处理"怎么做"
```

各范式待在各自擅长的层:FSM 管少而稳的模式,效用管连续权衡,BT 管步骤回退。**"整个 AI 用一种范式"与"整个项目用一套 FSM 框架"是同一种洁癖**(索引 §3.1),同样禁止。

(GOAP/规划器一级:行为需要多步前瞻与动态重规划时的选项,复杂度与调试成本再上一个量级,管理/模拟游戏几乎不需要,仅存目。)

## 6. 坏味道清单(B 类特有)

1. ❌ **状态代码直接查世界**(`get_overlapping_bodies`、距离计算)—— 感知层被旁路,黑板形同虚设(§3 纪律①)
2. ❌ **黑板存对象引用而非 entity_id**(§4.4)
3. ❌ **某状态缺 target_invalid 出边**(§4.4)
4. ❌ **B 机每物理帧评估**(§4.2)—— 性能尖峰 + 抖动温床
5. ❌ **AI 反应零延迟**(§4.1)—— 感知延迟是设计参数不是性能缺陷
6. ❌ **持续条件刷屏发事件**(非边沿触发,§4.1)
7. ❌ **flee 可被血量回升即时打断**(§4.3)—— 承诺机制缺失
8. ❌ **状态内部出现"步骤 + 回退"的长 if 链**(§5.3 信号)—— 该挂 BT 了,而不是继续加分支

## 7. 测试要点

1. **假感知驱动**:测试不跑物理与视线,直接按脚本向黑板写感知区 + 注入事件序列——B 的全部测试建立在 §3 纪律①之上
2. **场景剧本测试**:"spotted → 3s 后 lost → 期望 chase → search → (超时) → return"式的事件-时间线用例,断言状态序列与关键黑板值。B 类的表驱动 `[§3.8]` 基线以这种叙事形态呈现最自然
3. **目标失效全覆盖**:对每个持目标状态注入 target_invalid,断言转出且无空解引用(§4.4)
4. **抖动压力测试**:构造阈值附近振荡的血量/距离序列,断言单位时间转换次数 ≤ 上限(§4.3 三机制的验收)
5. **决策确定性**:同种子 RNG + 同事件序列双跑,断言决策序列一致(与 F 篇 §5.4 同款,回放调试的前提)
6. **群体测试**:N 个 AI 共享一目标,断言同时处于 attack 的数量 ≤ 令牌数(§4.5)

## 8. AI 代理协作约定(供 CLAUDE.md 摘录)

1. 实现 AI 前先产出:状态清单(含 alert/search 类中间态的取舍理由)、转换表 + Mermaid、黑板字段表(标注每字段的写方与读方)、感知参数表(视距/视锥/确认延迟,归定义层)。
2. B 状态代码禁止出现任何世界查询与节点访问;世界信息只经感知系统入黑板。黑板中一切实体引用用 id。
3. 每个持有目标的状态必须实现 target_invalid 出边;每个阈值型转换必须成对声明迟滞区间;flee 类状态必须声明承诺条件。
4. B 机评估频率为决策 tick(默认 10Hz、按 entity_id 分帧),不得挂到每帧回调。
5. 状态内部逻辑超过"单一行为 + 简单条件"时停止堆砌 if,按 §5 路线图上报升级建议(HSM / 挂 BT / 效用中层),经确认后实施。
6. 交付必须包含假感知驱动的场景剧本用例、目标失效全覆盖、抖动压力测试。
