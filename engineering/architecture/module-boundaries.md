# Godot 游戏开发中的模块边界(Module Boundary)精讲

> **文档定位**:本文档是《Godot 游戏开发中的 SOLID 实践指导》的配套篇。SOLID 文档回答"代码应当如何组织",本文回答更底层的问题——**边界应当划在哪里、由什么构成、如何验证、何时不该划**。核心命题:模块边界的本质是为"变更的传播"设计阻尼。
>
> **适用范围**:Godot 4.x / GDScript 项目,尤其是数据驱动的三层架构(静态定义层 / 运行时状态层 / 系统层)与 agentic 并行开发工作流。

---

## 0. 定义与总纲

模块边界是"模块内部"与"外部世界"之间的分割线,规定:**谁能看到什么、谁能调用谁、依赖朝哪个方向流、数据以什么形态跨线**。

它由四个切面构成,四者是同一件事的四个视角:

| 切面 | 回答的问题 | 决定的事 |
|------|-----------|----------|
| 职责边界 | 这个模块负责什么、不负责什么 | 变更的**落点** |
| 接口边界 | 外部只能通过什么与它打交道 | 变更是否**外泄** |
| 依赖边界 | 依赖朝哪个方向流动 | 外泄后朝哪个**方向传播** |
| 数据边界 | 数据以什么形态、什么所有权跨线 | 传播的**载体是否失控** |

---

## 1. 职责边界:难点不是"划",而是"划在哪"

### 1.1 粒度判断标准:变更同频

**把总是一起改的东西放进同一个边界,把独立演化的东西分开。** 这比"功能相似"更可靠:

- 战斗结算公式跟数值策划走,伤害飘字样式跟 UI 迭代走——变更节奏不同,必须分属两个模块,即使它们"都和伤害有关"。
- 伤害计算与暴击判定几乎总是一起调——强行拆开只制造跨边界沟通成本。
- tick 调度器一旦稳定几乎不动,tick 内各系统的结算规则随内容持续调整——调度器与被调度系统之间应有边界。

### 1.2 归属仲裁标准:信息拥有者原则

功能不知道放哪时,问:**完成它需要的信息主要归谁?**

例:"建筑受相邻建筑加成"需要建筑定义 + 城市网格拓扑,信息拥有者是规则层的空间系统——所以规则写在空间系统里,而非建筑节点脚本,即使直觉上"这是建筑的行为"。

### 1.3 时间维度:不负责的事,连"知道"都不该知道

模块代码里不应出现任何暗示"边界外某物存在"的痕迹:

- ❌ `emit ui_should_update` —— 泄漏了"外面有个 UI"的知识
- ✅ `emit health_changed` —— 只陈述领域事实

这点命名差别决定了替换整个 UI 层时,规则层是零改动还是到处改。

## 2. 接口边界:契约的三种强度

GDScript 无 `private` 关键字,接口边界靠约定与结构建立,需要明确的分级意识。

### 2.1 强度分级

| 强度 | 手段 | 强制力 | 适用 |
|------|------|--------|------|
| 约定级 | 下划线前缀 `_method`、文档注释 | 无 | 模块内部自我提示 |
| 结构级 | Facade 门面 + 内部类不给 `class_name` | 物理隔离 | 模块对外接口 |
| 数据级 | 参数/返回值只用 typed Resource / DTO,禁裸 Dictionary | 类型系统 | 所有跨边界数据 |

**关键技巧:不给 `class_name` 就是 GDScript 里最有效的访问控制。** 没有全局名字的类,外部想依赖都难。对外只暴露门面(如 `EconomySystem.settle_week()` / `query_balance()`),内部的 `_TaxCalculator`、`_UpkeepResolver` 用文件内部类或非 `class_name` 脚本。

**裸 Dictionary 是接口边界的头号腐蚀剂**:契约退化为"你猜里面有哪些 key",字段结构一改,另一方运行时才炸。

### 2.2 命令-查询-信号三分(CQS 的信号语境变体)

对外接口成员分三类,**互不混淆**:

1. **命令**(改状态):`build(def_id, cell)` —— 不返回业务数据,防止调用方把命令当查询用
2. **查询**(不改状态):`can_build(def_id, cell) -> bool` —— 绝无副作用,headless 测试可随意调用
3. **信号**(通知事实):`building_placed` —— 只陈述已发生的事,**过去时命名**(非 `place_building`)

三分之后,接口每个成员的"可安全使用方式"一目了然。

## 3. 依赖边界:无环是底线,稳定方向才是精髓

### 3.1 依赖稳定原则

"单向流动"的完整版本:**依赖箭头应指向更稳定、更抽象的东西。**

节点层 → 规则层 → 定义层之所以正确,不只因为无环,更因为它指向稳定性递增的方向(UI 天天改 / 规则偶尔改 / schema 最少改)。反向箭头意味着稳定者依赖易变者,易变者的每次抖动都震到稳定者——这是"边界破裂"的力学本质。

### 3.2 Godot 中的三个检查面

**(a) 加载期依赖**
`preload` 即建立硬依赖边,无需实际调用。规则层脚本 `preload` 场景文件的那一刻,边界已破。检查模块真实依赖,看文件头部 preload/import 清单比看调用代码更快。

**(b) Autoload 是隐形依赖之王**
任何对 autoload 的引用都是不出现在构造函数里的暗依赖边。"系统类通过 `_init` 注入依赖"在依赖边界视角下的意义:**把所有依赖边显式化,使依赖图可通过只读组合根一处代码画出**。

> 依赖健康度 ≈ 只看 main 场景装配代码能画出多完整的架构图。

**(c) 环的高发地带在同层之间**
层间环容易发现;同层系统之间的环(经济查人口,人口查经济)才是日常杀手。三档解法:

| 解法 | 适用条件 | 代价 |
|------|----------|------|
| 合并 | 两者变更同频(说明本就该是一个模块) | 无 |
| 提取 | 共享查询可下沉为共同依赖的只读数据层(GameState) | 数据层膨胀 |
| 事件解耦 | A 订阅 B 的事实并自维护缓存 | 最终一致性问题 |

tick 制游戏化解最终一致性的惯用手段:**阶段划分**——tick 内先全部结算、再统一发布事件,同一 tick 内不允许读到半新半旧的状态。

## 4. 数据边界:所有权,而不只是形态

### 4.1 核心问题

数据跨过边界之后,**谁拥有它、谁能改它?**

### 4.2 Godot 特有陷阱:Resource 默认共享引用

把 `BuildingDef` 传出去,对方改了 `base_output`,全游戏所有引用处全变——定义层被运行时污染,是最严重的边界破裂形式之一。防线是**类型级分工**:

- 定义层 Resource **一律视为不可变**(约定 + code review 守住,必要时 `duplicate()`)
- 运行时可变状态**一律不用 Resource**,用 RefCounted 状态类
- 两者物理上分属不同类型,想混都难

### 4.3 跨边界数据形态(按安全性排序)

1. **值语义快照**:查询返回新构造的只读 DTO(如 `WeekReport`),外部随便折腾不伤内部。tick 制一周一次分配,成本可忽略。
2. **信号参数传标量或小型快照**:传 `int` / `StringName` / DTO,不传内部对象引用让订阅者自己抠字段。
3. **传内部对象引用**:最危险,仅在明确性能理由下使用,接收方必须按只读对待。

### 4.4 翻译责任归属

规则层产出 `WeekReport`,UI 需要"带颜色标记的涨跌文本"——**翻译必须发生在 UI 侧(或专门 ViewModel 适配层)**。规则层输出格式化字符串的那一刻,UI 知识渗入规则层,数据边界名存实亡。

通用模式:**边界上的翻译层承担消费方适配,生成逻辑本身保持语义纯净。**(同一原则也适用于为外部消费者——如 VLM——设计导出格式的场景:导出层适配消费方,核心数据保持语义化。)

## 5. 边界质量自检(七项测试)

| # | 测试 | 通过标准 | 失败含义 |
|---|------|----------|----------|
| 1 | Headless 测试 | 能脱离场景树单独实例化并测试 | 边界没立住 |
| 2 | 实现变更测试 | 改内部实现,依赖方零改动 | 边界漏了实现细节 |
| 3 | 使用者盲测 | 模块不知道谁在用它 | 边界方向反了 |
| 4 | 新人测试 | 只用公开接口就能向不了解实现的人说清用法 | 接口边界未立住 |
| 5 | 删除测试 | 整模块删掉重写,外部只需改组合根装配几行 | 需改行数 = 泄漏的定量度量 |
| 6 | 依赖图单页测试 | 模块依赖关系可画在一页纸上,无环无双向箭头 | 画不出来 = 边界是糊的 |
| 7 | 并行开发测试 | 两人(或两个 AI 会话)分别改两个模块,合并冲突概率低 | 边界质量直接决定任务可并行拆分度 |

> 第 7 项对 agentic 工作流有直接工程价值:**任务能否安全拆给多个 Claude Code 会话并行执行,本质上是在考核模块边界的质量。**

## 6. 边界的成本:什么时候不该立边界

每条边界都有真实成本:DTO 定义与维护、快照构造开销、间接调用、"隔着边界看不到内部"的调试摩擦。

**判断标准回到 1.1:两侧是否独立演化。**

变更总是同频的两块代码之间立边界 = 纯支出零收益:每次改动要同时改边界两侧 + 边界契约本身,三处改动换零处隔离。

独立小体量项目的合理策略:

- **层间边界(节点/规则/数据)从第一天立死** —— 三层演化节奏差异是结构性的,永远成立
- **同层系统之间的边界按需生长** —— 等两个系统真的出现独立变更需求时再拆,不预支

---

## 7. 一句话收束

模块边界的本质是**为"变更的传播"设计阻尼**:

- 职责边界决定变更的**落点**
- 接口边界决定变更是否**外泄**
- 依赖边界决定外泄后朝哪个**方向传播**
- 数据边界决定传播的**载体会不会失控**

四者是同一件事的四个切面。

## 8. 与 AI 代理协作的补充约定

供 CLAUDE.md 引用或摘录:

1. 新建模块前,先声明其职责边界(负责什么/不负责什么)与对外门面(命令/查询/信号三分清单),再写实现。
2. 跨边界数据一律使用 typed DTO 或不可变 Resource,禁止裸 Dictionary 跨线,禁止传出内部可变对象引用。
3. 发现同层系统间出现相互引用时,停止实现并上报,按"合并/提取/事件解耦"三档方案给出建议后再继续。
4. 定义层 Resource 视为不可变;任何需要修改 Resource 实例的场合先 `duplicate()`。
5. 完成模块后,按第 5 节七项测试逐项自检并在交付说明中报告结果。
