# Patterns P1:解耦与通信(Communication)

> **索引归属**:`README.md` §4 中的 P1 册。双面模板、判定三问见 README,不重复。
>
> **本册地位**:问题域的唯一问题——**"模块之间如何交谈而不相互绑架?"** 与 P3 互为镜像:P3 防耦合过紧(人人都摸得到全局),P1 防解耦过散(事实满天飞、因果无人知)。本册反面主角"事件意大利面"是解耦手段用力过猛的产物——**解耦不是越多越好,是把耦合放到对的位置**。

---

## 0. 问题域坐标:耦合有两个维度

每种通信手段其实在回答两个独立的问题:

- **空间耦合**:调用方知不知道对方是谁?
- **时间耦合**:发出与执行是否同一时刻?

| 手段 | 发出方知道接收方? | 接收方知道发出方? | 同步? |
|---|---|---|---|
| 直接调用 | 知道 | 知道 | 是 |
| 回调注入(Callable) | 不知道(被注入) | 知道 | 是 |
| 信号(点对点) | 不知道 | 知道(主动连接) | 是 |
| 事件总线 | 不知道 | 不知道 | 是/可入队 |
| 命令(+队列) | 不知道 | 不知道 | **否**(可延迟/回放) |

从上到下解耦递增,**可追踪性递减**——每往下走一行,"这段代码为什么被执行了"的答案就更难找一层。本册所有裁决都是在这条交换曲线上找平衡点,总原则:**用满足需求的最靠上的一行**。

## 卡片 1:信号(观察者)

- **粒度层 / 吸收状态**:对象 / 🟢(引擎原生;手写 listener 数组即"重新发明信号")
- **解决的问题**:上游发生了事实,下游若干方需要各自响应,上游不应知道下游是谁
- **Godot 形态**:signal / connect / emit。"call down, signal up" `[SOLID篇]` 的 up 半边
- **适用信号**:① 通知的是**已发生的事实**(过去时可命名:`health_changed`、`building_placed`);② 发出方对"有没有人听、听了做什么"真正无所谓——删光所有监听者,发出方逻辑依然完整

──────── 翻面 ────────

- **入口诱惑**:"连一下就通了"——信号的接线成本低于厘清职责的成本,于是它被用于一切通信,包括那些**发出方其实很在乎结果**的场合
- **腐化信号**:① 信号名是命令式/祈使式(`do_attack`、`update_ui`)——这是用广播伪装的函数调用,发出方在假装不知道接收方;② 发出方 emit 之后检查某个被监听者改写的状态(隐式的请求-响应,靠副作用接力);③ 多个监听者之间存在**顺序依赖**(A 必须先于 B 处理)——信号语义不保证顺序,依赖顺序即依赖实现细节;④ 信号处理器内 emit 下一个信号(链的开端,见 §6)
- **不可逆点**:顺序依赖固化——多处逻辑默默依赖连接顺序后,任何重构连接的动作都成了赌博
- **解药**:①② 改直接调用或命令(卡片 3)——需要结果就承认耦合;③ 把顺序敏感的多方响应收拢为一个编排者依次调用(卡片 4);④ 见 §6 级联治理
- **关联**:`[边界篇]` "信号就是最小接口";`[FSM:B篇 §4.1]` 边沿触发纪律(事件是导数,状态是原函数)在一切信号设计中通用

## 卡片 2:事件总线

- **粒度层 / 吸收状态**:架构 / 🟡(自建,形态是 P3 卡片 1 的合法住户"纯信号总线")
- **解决的问题**:**跨层、远距离**的事实广播——两端相隔太远,点对点连接需要穿透多层持有引用
- **Godot 形态**:零状态 autoload,只有 signal 声明。进阶形态给每个 emit 附带来源标识,供 §6 的事件日志用
- **适用信号**:① 天然的"全局旁观者"存在(成就系统听一切、教程系统听一切、埋点听一切);② 发收双方**不在同一装配范围**内(组合根装配时两者互相拿不到引用,硬拿需要穿层)

──────── 翻面 ────────

- **入口诱惑**:"挂总线上,以后谁想听都行"——为不存在的未来听众预付解耦成本;第二级诱惑:总线用顺手后,**相邻对象也开始过总线**(父子通信走全局广播)
- **腐化信号**:① 相邻/同装配范围的对象经总线通信(能直连的绕远路);② 事件处理器内 emit 总线事件——级联的温床(§6);③ 事件 payload 携带**可变对象引用**(总线成了数据边界 `[边界篇 §4]` 的走私通道——广播出去的引用谁都能改);④ 回答"这个事件都有谁在听"需要全局搜索且结果超过一屏
- **不可逆点**:级联深度 >2 成为常态(事件引发事件引发事件),因果链只能靠断点考古——**事件意大利面**成形
- **解药**:见 §6 专题
- **关联**:P3 卡片 1(总线是 autoload 的合法形态,但"零状态"红线在此复述:总线上出现任何字段即刻违规)

## 卡片 3:命令

- **粒度层 / 吸收状态**:对象 / 🟡
- **解决的问题**:把"一次调用"物化为对象,从而获得调用不具备的能力:**入队(时间解耦)、撤销、回放、序列化、网络传输**
- **Godot 形态**:RefCounted 参数袋(`BuildCommand {def_id, cell}`)+ 系统侧的执行入口。`[FSM:DEF篇 §6]` 的 E→D/F 命令接口即本卡片实战;`[FSM:F篇]` 回放测试、`[FSM:C篇]` 两阶段结算的"待转换清单"都是命令队列的变体
- **适用信号**:① 需要撤销/重做(编辑器、策略游戏的规划模式);② 需要回放/录像/网络同步——命令序列就是最小的确定性输入;③ UI→规则的意图通道(意图可被 guard 拒绝,即命令的"执行失败"分支)

──────── 翻面 ────────

- **入口诱惑**:模式教材光环——"正规项目该有命令层",于是查询也包命令、内部调用也包命令
- **腐化信号**:① 查询被包成命令(命令是改状态的意图,查询走直接调用——CQS `[边界篇 §2]` 的命令侧专用此卡);② 命令对象自己持有执行逻辑并直接改状态——命令膨胀成迷你系统,执行逻辑应归属系统层,命令只是**参数袋 + 合法性自检**;③ 无撤销/回放/网络需求的单机内部调用被强行命令化——仪式感开销
- **不可逆点**:命令自带执行逻辑且互相调用后,系统层被架空,规则散落在几十个命令类里——God Object 的碎片化变体
- **解药**:命令瘦身为数据,执行归系统;不需要命令三能力的通道退回直接调用
- **关联**:卡片 1 翻面①(命令式信号的正解常是真命令);P2 数据驱动卡片(可序列化命令与数据驱动是近亲)

## 卡片 4:中介者(协调者)

- **粒度层 / 吸收状态**:对象~系统 / 🟡(但其最常见形态被引擎结构暗中吸收:**父节点天然是子节点们的中介者**)
- **解决的问题**:N 个同层对象两两通信形成 N² 引用网;收拢为"各方只认识协调者"的 N 条边
- **Godot 形态**:两种——场景树内,父节点接收子节点信号、调用其他子节点方法("call down, signal up"的完整闭环本身就是中介者模式);系统层内,独立协调对象(`[FSM:B篇 §4.5]` 的攻击令牌桶、小队协调器)
- **适用信号**:① 同层对象开始互相持有引用(边界篇同层环的第三种解法"事件解耦"之外,这是第四种:收拢);② 多方响应存在顺序/仲裁需求(卡片 1 腐化信号③ 的正解落点)

──────── 翻面 ────────

- **入口诱惑**:"这段协调逻辑不知道放哪,先放中介者"——中介者是逻辑的引力阱,什么都往里掉
- **腐化信号**:① 中介者开始包含**领域逻辑**(伤害怎么算、令牌该给谁的评分规则)——它的职责是路由与编排,领域逻辑属于参与者或专门系统;② 参与者退化为空壳,一切行为上收——God Object 换了个学名;③ 中介者之间开始互相通信——协调者需要协调者,说明层级划分错了
- **不可逆点**:参与者空壳化之后,"拆中介者"等于重写全部参与者
- **解药**:编排(先谁后谁)留下,决策(该不该、给多少)下放回参与者/系统
- **关联**:P2 God Object 卡片(中介者是其高发变体之一)

## 卡片 5:回调注入(Callable)

- **粒度层 / 吸收状态**:对象 / 🟡(Callable 是一等公民,注入极其顺手)
- **解决的问题**:低层需要在特定时机执行**高层决定的行为**,但不该知道高层——依赖倒置的最轻量形态;也是策略模式的函数级版本
- **Godot 形态**:Callable 参数/字段。`[FSM:DEF篇 §3.4]` D 机的场景操作注入、`[FSM:C篇]` 转换表的 guard/action 字段,均为本卡片实战
- **适用信号**:① **单一**定制点,开策略类嫌重、开信号嫌语义不符(需要返回值/需要恰好一个执行者);② 测试需要替换该行为(注入空操作/记录桩)

──────── 翻面 ────────

- **入口诱惑**:"传个 lambda 最快"——Callable 太顺手,以至于该开策略对象、该发信号的场合也用它
- **腐化信号**:① 同一对象的注入 Callable ≥3 个——定制点成组出现,该升格为策略对象(P2)或接口;② lambda 捕获了大对象/节点引用——生命周期纠缠,节点释放后调用即 "previously freed" 崩溃;③ 回调里再传回调(嵌套地狱)——控制流不可读,该改为信号序列或显式状态
- **不可逆点**:匿名 lambda 散布各处后,行为无名无籍,无法测试、无法搜索、无法复用
- **解药**:成组 → 策略对象;广播语义 → 信号;长流程 → 显式编排。捕获问题:优先绑定纯对象方法,涉节点用 weakref 或经信号中转
- **关联**:P2 策略卡片(本卡片是它的单点简配版,三个定制点是升格线)

## 6. 反面专题:事件意大利面的诊断与治理

腐化力学(README §5.2)在通信域的具体形态:**每一根意大利面都曾是一次正当的解耦**。治理分三层:

**(a) 预防性纪律**(新代码强制):
- 事实命名铁律:总线/信号事件一律过去时陈述句;出现祈使式命名即打回
- **级联深度预算**:事件处理器内 emit 事件,深度记为 +1;预算为 2,超出必须改为显式编排(处理器把需求交给系统,系统在自己的更新时机统一处理——即 `[FSM:C篇 §4.3]` 两阶段思想的通信版)
- 邻居直连原则:同一装配范围内的对象禁止过总线(§0 总原则"用最靠上的一行"的实施细则)

**(b) 可观测性建设**(总线的标配件,不是可选项):
- 事件日志:总线 emit 时记录 {事件, 来源, 帧号, 当前级联深度},环形缓冲最近 N 条
- 因果标注:处理器内再 emit 时携带触发源事件 id——日志即可重建因果树,"为什么它被执行了"从考古变成查表
- 监听者清单命令:调试指令打印某事件的全部连接方——卡片 2 腐化信号④ 的解毒剂

**(c) 存量治理**(与 P3 §6 手术同思路,增量执行):
按事件日志统计流量与级联,治理 Top 榜:高频邻居通信降级直连;深级联链收拢为中介者/系统编排;肥 payload 改为 id + 查询 `[FSM:B篇 §4.4 同款纪律]`。

## 7. 裁决速查表:谁和谁说话,走哪条道

| 通信关系 | 手段 | 依据 |
|---|---|---|
| 父 → 子 | 直接调用 | call down |
| 子 → 父 / 下游 → 上游 | 信号 | signal up |
| 同层相邻 | 经父节点/协调者中转 | 卡片 4;同层直连即埋环 `[边界篇]` |
| 跨层、远距、一对多事实 | 事件总线 | 卡片 2 适用信号,且仅事实 |
| UI → 规则(意图) | 命令 | 卡片 3;可拒绝、可入队、可回放 |
| 低层的单一定制点 | Callable 注入 | 卡片 5;≥3 个升策略 |
| 需要返回值/恰一个执行者 | 直接调用(经注入的引用) | 承认耦合,别用广播伪装 |

## 8. AI 代理协作约定(供 CLAUDE.md 摘录)

1. 选择通信手段时按 §7 表裁决并在实现说明中标注;偏离表格须给出理由。
2. 一切事件/信号命名为过去时事实;祈使式命名不予通过。
3. 事件处理器内 emit 事件的级联深度上限为 2;新建事件总线必须同时交付事件日志与监听者清单指令(§6b 为标配件)。
4. 总线事件 payload 只允许标量、id 与只读 DTO;出现可变对象引用即为实现错误。
5. 同一对象注入 Callable 达 3 个时停止实现,上报升格策略对象的建议;lambda 捕获节点引用不予通过。
