# Patterns P4:时序与更新(Timing & Update)

> **索引归属**:`README.md` §4 中的 P4 册。双面模板见 README;本册与 FSM 系列重叠最多,已裁定结论一律引用(`[FSM:x篇]`),本册只补 FSM 视角之外的空白。
>
> **本册地位**:问题域的唯一问题——**"这段代码在什么时刻运行,由谁决定?"** 反面主角"Update 黑洞"与"帧序耦合"是同一枚硬币:前者把一切逻辑挂到每帧,后者是挂上去之后互相之间长出的隐式顺序依赖。

---

## 0. 问题域坐标:三层时间

Godot 项目里同时流淌着三种时间,**多数时序 bug 源于把逻辑挂错了时间层**:

| 时间层 | 节拍 | 该住谁 | 归属 |
|---|---|---|---|
| 渲染帧(`_process`) | 可变,随帧率 | 视觉插值、动画、UI 响应 | 表现层 |
| 物理帧(`_physics_process`) | 固定步长 | 物理身体、A 类动作机 `[FSM:A篇 §3]` | 表现~执行层 |
| 逻辑 tick | 自定节拍(周/回合/秒) | 一切规则结算 `[FSM:C/F篇]` | 规则层 |

第三层是引擎不提供、必须自建的——管理/模拟游戏的规则时间(每周 tick)与帧率毫无关系,把它寄生在前两层上就是本册多数反模式的起点。三层各配一张速查:**看得见的挂帧,碰得着的挂物理帧,算得出的挂 tick**。

---

## 卡片 1:游戏循环

- **粒度层 / 吸收状态**:架构 / 🟢(引擎完全持有——你不写游戏循环,你把代码挂进它的节拍)
- **解决的问题**:输入、物理、逻辑、渲染的节拍统一与推进
- **Godot 形态**:引擎主循环 + 两个回调节拍。本卡片的实践内容只有一件事:**按 §0 三层把代码挂对地方**,以及自建第三层——逻辑 tick 驱动器(一个由 D 机供给、按游戏速度倍率折算的累加器:攒够一个 tick 时长就结算一次,快进三倍 = 单帧内可结算多个 tick,暂停 = 停止供给 `[FSM:DEF篇 §3.2]`)
- **适用信号**:无条件——所有项目都活在里面

──────── 翻面 ────────

- **入口诱惑**:"就跑个循环等结果"——在回调里写 `while not done: await` 式的阻塞等待
- **腐化信号**:① 规则结算挂在 `_process`——帧率成了游戏性参数(高刷屏玩家资源涨得快);② 任何形式的被动等待循环(await 死等、轮询 sleep)——单线程回调模型下这是自缢,headless 下是挂起(agentic 工作流的头号杀手,值得写进最高纪律);③ delta 误用:固定步长回调里乘 delta 做规则运算(物理帧的 delta 是常数,乘它只是把魔法数字换了个写法),或渲染帧里不乘 delta 做移动(帧率依赖)
- **不可逆点**:规则与帧率的耦合被数值调参"校准"过之后(在 60fps 下调平衡),迁移到 tick 制等于全部重调
- **解药**:规则迁往逻辑 tick;等待改事件回调或显式等待状态 `[FSM:DEF篇 §5.2]`
- **关联**:卡片 4(逻辑 tick 的内部结构)

## 卡片 2:更新方法(Update Method)

- **粒度层 / 吸收状态**:对象 / 🔴(`_process` 是引擎把入口诱惑做成默认模板的典型——新建脚本就带着它)
- **解决的问题**:大量对象各自有每帧行为,每对象自带 update 由循环统一驱动
- **Godot 形态**:两轨——**分散式**(各节点自己的 `_process`/`_physics_process`,适合表现层:各自为政、顺序无关);**集中式**(系统持有对象列表,自己的更新入口里显式遍历,规则层强制此轨——遍历顺序是代码写死的,`[FSM:C篇 §4.3a]` 顺序确定性免费获得,而节点树的回调顺序取决于树结构,重排场景就变结果)
- **适用信号**:对象确实**每帧**都有事做(移动中、播放中、插值中)——这个前提正是翻面的检查点

──────── 翻面 ────────

- **入口诱惑**:"就一行判断,写 `_process` 里最快"——README 公理三的原型案例;`_process` 是所有不知道放哪的逻辑的默认容器
- **腐化信号**:① 某 `_process` 超过 50 行或长出状态分支梯——里面住着一台未显式化的 FSM,按 `[FSM索引 §3.1]` 收编;② **每帧轮询本该事件驱动的东西**(`if quest_done and not rewarded:`)——把"发生"翻译成"反复查看",Update 黑洞的主食;③ 大量空转:成百对象的 `_process` 每帧进来发现无事可做就返回——idle 对象该 `set_process(false)`(状态 enter/exit 顺手开关,A 类机的天然搭配),海量实体则整体迁往集中式 + 分桶 `[FSM:C篇 §4.5]`
- **不可逆点**:多个 `_process` 之间长出隐式顺序依赖后——见 §6 帧序耦合
- **解药**:分支梯 → FSM;轮询 → 信号/事件;空转 → 开关或集中式;规则 → tick
- **关联**:`[FSM:B篇 §4.2]` 决策 tick 与分帧(AI 逃离每帧更新的完整示范)

## 卡片 3:事件队列

- **粒度层 / 吸收状态**:系统 / 🟡
- **解决的问题**:P1 表格最后一行的"时间解耦"正式立卡——产生与消费分离,由此获得:**峰值削平**(本帧产生,分帧消费)、**合并去重**(同帧十次爆炸音效请求合并为一次)、**同 tick 一致性**(阶段间通信,`[FSM:C篇 §4.3b]` 两阶段的"待转换清单"、`[FSM:DEF篇 §5.3]` 插入队列均为本卡片实例)
- **Godot 形态**:手写环形缓冲/Array 足矣;队列归属其消费系统,不做全局
- **适用信号**:① 生产峰值与消费能力不匹配(一帧刷出 200 个伤害数字);② 消费必须发生在特定阶段/时机而产生随时发生(跨阶段通信);③ 需要合并/去重/优先级重排后再消费

──────── 翻面 ────────

- **入口诱惑**:"先入队总没错"——异步化被当作默认美德,能同步直办的也入队
- **腐化信号**:① 消费无预算——每帧清空整个队列,削峰功能名存实亡(该配每帧最大消费数,溢出留给下帧);② **过期事件**照常消费——队列里躺了三帧的"对 X 造成伤害",X 已经死了;消费时必须重校验(`[FSM:B篇 §4.4]` target_invalid 纪律的队列版:队列存 id 不存引用,消费时解引用);③ 队列化成为主要通信方式——一切因果都隔着队列,调试时"谁在什么时候塞的这条"比同步调用栈难查一个量级,P1 意大利面的时间维度版;P1 §0 总原则同样适用:**能同步就同步**
- **不可逆点**:核心流程全面异步化后,执行顺序成为涌现性质,任何时序 bug 都无法本地复现
- **解药**:预算 + 重校验为标配;通信降级回 §7 表更靠上的行;队列配日志(P1 §6b 同款可观测性)
- **关联**:P1 卡片 3(命令入队即两卡合用)

## 卡片 4:tick / 阶段

- **粒度层 / 吸收状态**:架构 / 🟡
- **解决的问题**:规则时间与墙钟/帧率的完全解耦,结算获得原子性与确定性
- **Godot 形态**:FSM 系列已全面建成,本卡片主要是索引:阶段序列 `[FSM:DEF篇 §5.1]`、批量结算三决策 `[FSM:C篇 §4.3]`、tick 计数时间 `[FSM:C篇 §4.2]`、确定性纪律 `[FSM:DEF篇 §5.4]`。补一条 FSM 视角外的:**tick 原子性优先于帧预算**——单 tick 结算过重导致掉帧时,正解是优化结算或降低 tick 频率,**不是**把一个 tick 的结算撕开摊到多帧(撕裂的 tick 意味着存在"半个星期"的可观测状态,一切一致性保证作废;确需摊帧,以整 tick 为粒度限流:本帧最多结算 N 个 tick,快进档位由 N 封顶)
- **适用信号**:存在任何"结算"概念的规则层,无条件

──────── 翻面 ────────

- **入口诱惑**:"要个当前时间而已"——`Time.get_ticks_msec()` 顺手可得
- **腐化信号**:① 墙钟渗入规则(规则层出现 Time.*/OS 时间)——变速、暂停、回放、headless 四杀,`[FSM全系列]` 时间范式(时刻记录+比较,时刻=tick 计数)的对立面;② tick 与物理帧 1:1 绑死(每物理帧必然一 tick)——加速档位做不了,本质是卡片 1 腐化①的隐蔽变体;③ tick 内读写交错未做快照/排序声明 `[FSM:C篇 §4.3]`
- **不可逆点**:与卡片 1 相同——数值在错误的时间基底上被调平后,换基底 = 重调平衡
- **解药**:墙钟只允许存在于表现层与基础设施层;规则层时间一律 tick 计数
- **关联**:F 机(tick 的宿主)、卡片 3(阶段间通信)

## 卡片 5:延迟调用(call_deferred / await)

- **粒度层 / 吸收状态**:对象 / 🔴(一行救报错,引擎亲手递上的创可贴)
- **解决的问题**:把操作推迟到**引擎的安全时机**——物理回调中不能改物理状态/释放节点、信号处理中不能改动发射者正在遍历的集合、树遍历中不能增删节点。这些是引擎的硬约束,deferred 是为它们准备的,**仅为它们**
- **Godot 形态**:`call_deferred` / `queue_free`(本身即 deferred 语义)/ `set_deferred`
- **适用信号**:引擎文档/报错明确指出当前时机不安全——这是唯一适用信号。合法使用几乎总是伴随一条能说出口的引擎约束

──────── 翻面 ────────

- **入口诱惑**:"报错了?deferred 一下就不报了"——它确实不报了;时序创可贴从此上瘾。第二形态:"这个流程要等好几步"——于是规则层长出 await 链
- **腐化信号**:① deferred 被用于修**逻辑**顺序 bug("A 比 B 先跑就错,defer 一下 A")——真实病灶是帧序耦合(§6),创可贴把它按进了下一帧,bug 变成偶发;② **规则层出现 await**——await 链是一台隐式状态机:无状态名、不可存档、headless 下断链即挂起,`[FSM:DEF篇 §5.2]` 已裁定等待必须是显式状态,`[FSM:DEF篇 坏味道7]` 同案;③ deferred 叠 deferred(推迟到推迟之后)——已经在赌引擎帧内排程的实现细节
- **不可逆点**:偶发时序 bug 被多层 deferred"治好"之后,真实执行顺序无人能画——每个 deferred 都是一笔无票据的时序负债
- **解药**:逻辑顺序 → §6;流程等待 → 显式等待状态/事件回调;每处 deferred 註释其对应的引擎约束,注不出来的删掉重想
- **关联**:`[FSM:DEF篇 §5.2]`;CLAUDE.md 被动等待禁令(同一条纪律的 agentic 版)

## 6. 反面专题:帧序耦合

**定义**:正确性依赖于"A 恰好在 B 之前执行",而这个顺序没有任何代码显式保证——它悬挂在节点树顺序、`_process` 回调顺序、信号连接顺序、deferred 排程这些**实现细节**上。

**诊断法**(三个都廉价):重排场景树中兄弟节点顺序,行为变了即确诊;打乱系统初始化顺序重跑测试;`[FSM:C篇 测试2]` 批量确定性测试(注册顺序洗牌)本就是它的探测器。

**治理三板斧**,按适用面排序:

1. **集中更新器显式排序**(卡片 2 集中轨):顺序从树结构的巧合变成一行代码——`for system in [economy, population, events]:` 顺序即文档
2. **阶段化**(卡片 4):顺序需求上升为阶段边界——"伤害必须先于死亡判定"翻译成两个阶段,而不是两个回调的先后
3. **快照读**(`[FSM:C篇 §4.3b]`):消灭顺序敏感本身——大家都读 tick 开始时的世界,谁先谁后无所谓

信号侧的补充:P1 卡片 1 腐化③(监听者顺序依赖)是帧序耦合的信号变体,解药同为收拢显式编排。**总纲一句:顺序要么显式声明,要么显式无关,不存在第三种健康状态。**

## 7. 裁决速查表:什么代码挂什么节拍

| 代码性质 | 归宿 | 依据 |
|---|---|---|
| 视觉插值、UI、动画驱动 | `_process` | §0 表现层 |
| 物理身体、A 类动作机 | `_physics_process` | `[FSM:A篇]` |
| 一切规则结算 | 逻辑 tick(集中更新器) | 卡片 1/4 |
| AI 决策 | 决策 tick(低频分帧) | `[FSM:B篇 §4.2]` |
| 峰值/合并/跨阶段传递 | 事件队列(预算+重校验) | 卡片 3 |
| 引擎安全时机约束 | call_deferred(注明约束) | 卡片 5,仅此 |
| "等待某事完成" | 显式等待状态或事件回调 | 卡片 5 翻面②,规则层禁 await |

## 8. AI 代理协作约定(供 CLAUDEmd 摘录)

1. 写任何每帧回调前,按 §7 表判定节拍归属;规则层代码出现 `_process`/`_physics_process`/`Time.*`/`await` 即为实现错误。
2. `_process` 超过 50 行或出现状态分支梯时停止堆砌,上报 FSM 化/事件化建议。
3. 每处 call_deferred 必须附注所规避的引擎约束;用于逻辑顺序调整的 deferred 不予通过,改走 §6 三板斧。
4. 事件队列交付必须含每帧消费预算与消费时重校验;队列元素只存 id 与值,不存对象引用。
5. 任何被动等待循环(轮询、await 死等)一律禁止;等待翻译为显式状态或事件回调,并配超时出边。
