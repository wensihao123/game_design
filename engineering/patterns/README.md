## 1. 概念框架:三条公理

**公理一:模式是问题-解法的配对,不是好代码的写法。** 模式(pattern)= 在特定问题语境下被反复验证的解法;反模式(anti-pattern)= 被反复验证的"看似解法、实际制造更大问题"的配对。脱离语境套用模式,模式即刻退化为反模式——因此本知识库**不维护"好清单"与"坏清单"两张表**,只维护带适用边界的卡片。

**公理二:模式与反模式常是同一事物的两面。** 单例是 GoF 成员也是头号反模式;继承是 OOP 支柱也是被 SOLID 篇限制到一层的东西;事件总线既解耦模块也制造"事件意大利面"。每张卡片必须同时描述**模式面**与**腐化面**。

**公理三:反模式源于局部合理决策的累积,不是愚蠢。** 每个反模式都有"诱人的第一步"(God Object:"先放这儿以后再拆";Update 黑洞:"就一行,写 _process 最快")。识别反模式的关键不是记住终态长相,而是标注每一个的**入口诱惑**与**不可逆点**——与 FSM 系列"识别信号"方法论同源。

## 2. 分类坐标系

### 轴一:粒度层

| 层 | 范围 | 代表 |
|----|------|------|
| 对象粒度 | GoF 经典,类与对象的组织 | 策略、命令、观察者、享元 |
| 系统粒度 | 游戏编程模式(Nystrom 一系) | 对象池、脏标记、事件队列、空间分区 |
| 架构粒度 | 项目级结构 | 分层、数据驱动、ECS、事件总线 |

### 轴二:Godot 吸收状态(每张卡片必标)

| 状态 | 含义 | 例 |
|------|------|-----|
| 🟢 已吸收 | 引擎原生机制即此模式,正确用引擎 = 正确用模式,勿手写第二套 | 观察者→信号;原型→场景实例化;组合→场景树;享元→Resource 共享 |
| 🟡 需手动 | 引擎不提供,须自己实践 | 策略、命令、对象池、转换表 |
| 🔴 高危 | 引擎提供了,但引擎形态即其腐化高发形态 | 单例→Autoload;更新方法→`_process` |

🔴 类卡片是本知识库的重点照顾对象:引擎把入口诱惑做成了一键操作。

## 3. 双面卡片模板(所有细则统一采用)

```
## 卡片:<模式名>
- 粒度层 / 吸收状态:<对象|系统|架构> / <🟢|🟡|🔴>
- 解决的问题:一句话,必须含语境
- Godot 形态:引擎原生对应物或推荐实现骨架
- 适用信号:出现什么征兆时该引入(≥2 条,可操作)
- ─────── 翻面 ───────
- 入口诱惑:它的腐化形态的"合理的第一步"是什么
- 腐化信号:开始变成反模式的征兆(≥2 条,可操作)
- 不可逆点:过了哪条线重构成本陡增
- 解药:腐化后的重构方向(常是引用另一张卡片)
- 关联卡片 / 关联专题结论:交叉引用,禁止重写已有结论
```

模板刚性执行的理由:适用信号与腐化信号成对出现,才能防止卡片退化为"模式介绍";交叉引用强制执行,才能防止六册之间和与既有专题之间的结论漂移。

分册逻辑:按**问题域**而非粒度层分(同一问题域内,对象/系统/架构级的方案要放在一起比较——"全局访问"一册里单例、服务定位器、依赖注入是同一问题的三档答案,拆开就没法比了)。

**建议研究顺序:P3 → P1 → P2 → P4 → P5 → P6。** 理由:P3(全局访问)是既有专题批判最多、欠账最集中的一册,先还账;P1/P2 是日常使用频率最高的两册;P4 与 FSM 系列的 tick/阶段结论大量重叠,整理成本低;P5 的反面主角是"过早优化",放最后本身就是一种表态;P6(创建与装配)是 P1–P5 之后的增补,与 P3 互为姊妹册(P3 讲对象如何获得协作者,P6 讲对象如何被制造),紧随 P3/P5 结论之后收尾。

## 5. 共享方法论(细则不再重复)

### 5.1 判定次序

评估"要不要用某模式"的固定三问,按序短路:

1. **引擎已吸收?**(🟢)→ 用引擎原生形态,手写第二套即为反模式("重新发明信号"是 Godot 新手三大礼之一)
2. **问题真的出现了?** → 模式解决的问题若尚未出现,引入即过早抽象。判据沿用边界篇:两侧是否已出现独立演化的实际需求
3. **适用信号 ≥2 条命中?** → 命中才引入,并同时记下该卡片的腐化信号作为观察哨

### 5.2 腐化的通用力学

各反模式的具体形态不同,腐化路径高度一致:**便利入口 → 增量累积 → 越过不可逆点 → 结构性绑架**。干预窗口在第二阶段——所以卡片的"腐化信号"全部瞄准增量累积期的可观测征兆(某 autoload 被 80% 脚本引用、某 `_process` 超过 50 行、某事件的订阅者互相触发)。

### 5.3 三条既有结论直接沿用(引用格式:`[边界篇]` `[SOLID篇]` `[FSM:x篇]`)

- 全局访问类卡片的裁决依据:`[边界篇]` 依赖显式化原则与"autoload 是隐形依赖之王"
- 组织类卡片的裁决依据:`[SOLID篇]` 继承一层上限、组合优先、封闭分类法
- 时序类卡片的裁决依据:`[FSM:C篇]` 批量结算三决策、`[FSM:DEF篇]` 阶段序列、全系列"时刻记录+比较"时间范式

## 6. 术语表

| 术语 | 定义 |
|------|------|
| 双面卡片 | 同时描述模式面(适用信号)与腐化面(腐化信号)的知识单元 |
| 入口诱惑 | 反模式形成路径上"局部合理的第一步" |
| 不可逆点 | 腐化路径上重构成本发生陡增的临界线 |
| 吸收状态 | 模式被 Godot 引擎原生机制承载的程度(🟢/🟡/🔴) |
| 事件意大利面 | 事件/信号解耦过度后,因果链不可追踪的状态 |
| 货物崇拜 | 照抄模式形式而不理解其意图的套用方式 |
| 散弹式修改 | 一个变更需要同时修改多处不相邻代码的味道(边界失效的症状) |

## 7. AI 代理协作约定(供 CLAUDE.md 摘录)

1. 引入任何设计模式前,按 §5.1 三问判定并在实现说明中记录判定过程;🟢 类模式禁止手写替代实现。
2. 实现中触发任何卡片的腐化信号时,标注技术债并上报,不顺手扩大。
3. 新增卡片或修订卡片时严格遵循 §3 模板,适用信号与腐化信号必须成对、必须可操作(可被 code review 检查)。
4. 卡片结论与既有专题冲突时,以专题结论为准并上报冲突,禁止两处各说各话。
