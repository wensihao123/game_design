# Patterns P3:全局访问(Global Access)

> **索引归属**:`README.md` §4 中的 P3 册。卡片严格遵循 README §3 双面模板;判定三问、腐化力学见 README §5,不重复。
>
> **本册地位**:整个问题域只有一个问题——**"代码如何获得它需要的协作者?"** 五张卡片是这个问题从"最隐式"到"最显式"的答案光谱。既有专题欠账最集中的一册:`[边界篇]` 对 autoload 的批判、`[SOLID篇]` 的构造注入,在此升格为完整卡片并补上"何时它们反而是对的"。

---

## 0. 问题域光谱

```
隐式 ◄──────────────────────────────────────────► 显式
单例/Autoload → 服务定位器 → 上下文对象 → 构造注入(+组合根)
  谁都拿得到      问注册表要     随包裹领取      出生时发到手上
```

越靠左:取用越方便、依赖越不可见、测试越难。越靠右:装配越费事、依赖图越清晰、替换越容易。**本册的全部内容就是回答:什么东西配待在光谱的哪一段。**

---

## 卡片 1:单例 / Autoload

- **粒度层 / 吸收状态**:对象~架构 / 🔴(引擎把入口诱惑做成了项目设置里的一个加号)
- **解决的问题**:真正跨越整个应用生命周期、且不承载游戏规则状态的服务,需要一个无仪式的访问点
- **Godot 形态**:Autoload。合法住户三种:**纯信号总线**(零状态,只有 signal 声明)、**平台服务门面**(音频总线、存档 IO、成就上报——薄封装,内部无游戏知识)、**引擎级配置**(输入映射管理)
- **适用信号**:① 该对象在 headless 测试中要么无害存在、要么可被整体无视;② 它不持有任何"读档后必须恢复"的数据(存档测试 `[FSM:DEF篇 §2]` 反向应用:进存档的东西没资格当 autoload)

──────── 翻面 ────────

- **入口诱惑**:"这个东西全局都要用"——**这句话经常是真的**,陷阱不在判断而在后续:全局要用 ≠ 全局可写 ≠ 该持有状态。诱惑的第二级更隐蔽:第一个 autoload 立住后,"再加一个也无妨"的边际成本感知为零
- **腐化信号**:① autoload 开始持有游戏规则状态(金钱、进度、实体表);② 被项目过半脚本引用 `[边界篇测试]`;③ autoload 之间互相引用(全局对象的依赖网 = 最难拆的网);④ 写测试时想 mock 某个 autoload——想 mock 即证明它该被注入
- **不可逆点**:状态写散——多个 autoload 各持一块规则状态且互相读写之后,拆分需要同时重构所有触点,成本从线性变平方
- **解药**:卡片 4 构造注入 + §6 手术步骤(存量项目的增量撤离方案)
- **关联**:`[边界篇]`"autoload 是隐形依赖之王";卡片 2(服务定位器是它的半步改良)

---

## 卡片 2:服务定位器

- **粒度层 / 吸收状态**:系统 / 🟡
- **解决的问题**:跨切面技术服务(音频、日志、埋点)需要全局可达,**同时**需要按环境整体替换(headless 换 NullAudio、测试换 MemoryLogger)
- **Godot 形态**:一个注册表 autoload:`Services.register(&"audio", impl)` / `Services.get(&"audio")`。启动时由组合根注册,headless 测试注册空实现
- **适用信号**:① 某服务已有 ≥2 种实现需要按环境切换;② 该服务是纯技术切面——删掉它的全部调用,游戏规则结果不变(音效没响 ≠ 结算错了)

──────── 翻面 ────────

- **入口诱惑**:"比构造注入省事,比裸单例灵活"——两头讨好的中间态,于是什么都往里注册
- **腐化信号**:① 规则系统开始从定位器取依赖(EconomySystem 里出现 `Services.get(&"state")`)——依赖重新隐式化,DI 的全部收益归零;② 注册表条目无人盘点地增长;③ 出现"先 get 判空再降级"的防御代码——说明注册时序已不可靠
- **不可逆点**:`Services.get` 散落进规则层各处后,它就是换了名字的单例地狱
- **解药**:铁律一条——**定位器只准技术切面入住,规则依赖一律走卡片 4**。已腐化的按 §6 手术,把规则依赖逐个改注入
- **关联**:卡片 1(它是单例的可替换版)、卡片 4(它是 DI 的偷懒版——两个邻居都在提醒它的临时性)

## 卡片 3:上下文对象(ctx)

- **粒度层 / 吸收状态**:对象 / 🟡
- **解决的问题**:一组**总是结伴出行**的依赖/数据,逐个传参使签名膨胀
- **Godot 形态**:纯 RefCounted 数据类。FSM 系列的 ctx 与黑板 `[FSM:B篇 §3]` 即本卡片的两个实例——黑板还额外示范了最佳实践:**字段按读写方分区并注释**(感知区/决策区/意图区)
- **适用信号**:① ≥3 个参数在多个方法签名中始终同进同出;② 这组东西有清晰的共同生命周期(同生同灭于一个实体/一次结算/一局游戏)

──────── 翻面 ────────

- **入口诱惑**:"再塞一个字段就不用改签名了"——ctx 是 God Object 最舒适的新马甲:它连名字都自带正当性
- **腐化信号**:① 出现无明确读写方的字段(问不出"谁写它、谁读它");② 不同消费者只使用 ctx 的**互不相交子集**——说明这其实是几个 ctx 被硬装进一个袋子(接口隔离 `[SOLID篇]` 的 ctx 版);③ ctx 开始长方法(它该是数据袋,长出行为说明职责在偷渡)
- **不可逆点**:ctx 被跨层传递(节点层拿到了规则 ctx)之后,它成了绕过一切边界的走私通道
- **解药**:按读写方拆分为多个窄 ctx;跨层需求改为只读快照 DTO `[边界篇 §4]`
- **关联**:卡片 4(ctx 是注入参数膨胀的正解之一,见其翻面)

---

## 卡片 4:构造注入(朴素依赖注入)

- **粒度层 / 吸收状态**:对象~架构 / 🟡
- **解决的问题**:让每个对象的依赖**显式、可测、可替换**——依赖清单就是 `_init` 签名,无处可藏
- **Godot 形态**:`_init(state: GameState, defs: DefDatabase)`,由组合根构造。**不需要任何 DI 框架/容器**——GDScript 项目的对象图深度撑不起容器的复杂度,`[SOLID篇]` 的朴素注入就是完全体
- **适用信号**:无条件——**一切系统层对象的默认形态**。README §5.1 三问在此短路:规则层对象不问"要不要注入",只问"注入什么"

──────── 翻面 ────────

- **入口诱惑**(反向诱惑——诱惑在于**不用**它):"从 autoload 拿只要一行,注入要改构造函数、改组合根、层层传递三处"——单点成本确实更高,收益全在总量与远期
- **腐化信号**(注入本身的教条化):① `_init` 参数 >5 个——不是注入错了,是这个对象职责过多 `[SOLID篇]` 或该收拢为 ctx(卡片 3);② 为传递而传递:中间对象接收依赖只为转交给下游——说明装配路径设计错了,下游应直接由组合根构造;③ 引入反射式 DI 容器/自动装配——GDScript 语境下的货物崇拜
- **不可逆点**:几乎没有——注入式代码退回全局访问总是容易的,反向才难。这个不对称性本身就是选它的理由
- **解药**:参数膨胀 → 拆职责或 ctx;传递链 → 重画装配路径
- **关联**:卡片 5(注入的另一半——没有组合根的注入只是把 new 散得到处都是)

---

## 卡片 5:组合根

- **粒度层 / 吸收状态**:架构 / 🟡
- **解决的问题**:对象图的**装配**集中于一处,使依赖图可以"读一个文件就画出来"
- **Godot 形态**:main 场景脚本(应用级)+ D 机的状态 enter(局级——进入 in_game 时装配 F 机与各系统,`[FSM:DEF篇 §6]` 的装配段即本卡片实战)。装配代码的形态就是一串 new 与注入:`var economy := EconomySystem.new(state, defs)`
- **适用信号**:与卡片 4 成对,无条件采用。**装配根的数量 = 生命周期层级数**(应用一个、一局一个),不多不少

──────── 翻面 ────────

- **入口诱惑**:"就地 new 一个吧,拿到组合根去装太绕了"——次级装配点就此萌芽
- **腐化信号**:① 组合根之外出现"new 系统对象 + 注入依赖"的代码(装配权扩散);② 组合根本身膨胀为数百行的巨型装配函数——God Object 的装配版
- **不可逆点**:装配点扩散到无法回答"这个系统是谁、在哪构造的"时,依赖图重新不可读——等效于回到单例地狱,只是 new 代替了 get
- **解药**:膨胀 → 按模块拆装配函数(`_assemble_economy()`、`_assemble_narrative()`),组合根变成装配函数的目录;扩散 → 收权,例外只留实体工厂(海量同构实体的构造是工厂模式的地盘,不算装配点扩散)
- **关联**:`[边界篇]`"依赖健康度 ≈ 只读组合根能画出多完整的架构图"——本卡片就是那句话的实施细则

## 6. 手术步骤:存量单例地狱的增量撤离

新项目按上述卡片走即可;已腐化项目的价值在这份**可分批发布**的重构方案(每步之后项目均可正常运行):

1. **冻结**:CLAUDE.md/评审规则立即禁止新增 autoload 引用与新 autoload——先止血再手术
2. **盘点**:grep 生成每个 autoload 的引用清单(文件 × 用途),这份清单就是手术的进度表
3. **分类三堆**:每个 autoload 的成员拆解为——**状态**(可变游戏数据)/ **服务**(行为,无规则状态)/ **信号**(纯广播)
4. **状态出逃**(最优先,最痛,收益最大):规则状态迁入注入式 GameState;原字段改为转发属性(getter 代理到新家)保持兼容,引用点随后逐批改为注入,清空后删转发
5. **服务分流**:技术切面服务留下或入驻服务定位器(卡片 2);规则服务改为注入式系统对象
6. **降级或安乐死**:掏空的 autoload 若还剩信号,降级为纯信号总线(卡片 1 合法住户);一无所有则删除

顺序不可颠倒:先状态后服务,因为状态是不可逆点的制造者;转发属性技巧是整个方案能"增量"的关键——它把"一次改几百处"分期成"每周改一批"。

## 7. 裁决速查表

| 要访问的东西 | 光谱位置 | 依据 |
|---|---|---|
| 游戏规则状态与系统 | **构造注入 + 组合根,无例外** | 进存档的东西没资格全局化(卡片 1 适用信号②) |
| 跨切面技术服务(音频/日志/埋点) | 服务定位器;实现单一且稳定时裸 autoload 亦可 | 卡片 2 适用信号 |
| 纯事件广播 | 信号总线 autoload | 零状态即零风险 |
| 一组结伴的依赖/数据 | ctx,按读写方分区 | 卡片 3 |
| 引擎自身全局(Input/AudioServer) | 直接用,但**只准出现在对应适配层** | 引擎全局出现在规则层同样是隐式依赖 |

最后一行值得强调:`Input.is_action_pressed` 写进规则系统,与自建单例地狱是同一种病——`[FSM:A篇]` 的意图层、`[FSM:B篇]` 的感知层,本质上都是给引擎全局修的隔离带。

## 8. AI 代理协作约定(供 CLAUDE.md 摘录)

1. 新增 autoload 前必须论证其属于卡片 1 三种合法住户之一,并通过两条适用信号;论证写入实现说明。
2. 规则层代码中出现 autoload 引用、`Services.get`、或引擎全局(Input/Time 墙钟等)即为实现错误;一律改为构造注入或经适配层。
3. `_init` 参数超过 5 个时停止实现,上报"拆职责 vs 收拢 ctx"的建议。
4. 装配代码只允许出现在已声明的组合根(及实体工厂)中;发现次级装配点,标注技术债。
5. 涉及存量重构时按 §6 六步执行,每步独立提交、独立可运行;禁止跳步直改。
