# Patterns P6:创建与装配(Creation & Assembly)

> **索引归属**:`README.md` §4 中的 P6 册(P1–P5 之后增补)。双面模板见 README。
>
> **本册地位**:P3 的姊妹册——P3 回答"对象如何**获得**协作者",本册回答"对象如何**被制造**出来"。两册合起来是对象的完整供应链。GoF 创建型五模式中,原型与单例已分别在 P2/P3 立卡;工厂方法与抽象工厂在本册卡片 1 中以"意图存活、机制转世"的方式收编;建造者以窄生存位立卡。本册真正的主角是 GoF 完全没有、游戏开发天天流血的题目:**出生/死亡协议**(卡片 2)。

## 0. 问题域坐标:实体供应链

一个实体从无到有再到无,经过五站,恰好对应五张卡片:

```
原料      →   制造      →   入世        →   (批量调度)    →   离世
资源加载      工厂          出生协议         生成器           死亡协议
(卡片5)      (卡片1)       (卡片2前半)      (卡片3)         (卡片2后半)

特殊通道:程序化施工(卡片4)——原料不是现成资产而是算法产物时的旁路
```

本册的反面主角全部是**链条断裂**的形态:构造散落(制造站不唯一)、出生竞态(入世顺序错乱)、加载卡顿(原料站堵在 gameplay 中间)。§6 的供应链审计就是对整条链的完整性检查。

## 卡片 1:数据驱动工厂

- **粒度层 / 吸收状态**:系统 / 🟡
- **谱系**(本卡片特有栏,回答"GoF 工厂哪去了"):GoF 工厂方法的**意图**——调用方不知道具体类型,创建决策外置——完整存活;其**机制**(继承 + override 创建钩子)被双杀:鸭子类型消解了面向抽象类型的仪式,`[SOLID篇]` 继承一层上限封死了子类化路线。转世形态:类型分支不写成子类,写成**数据查找**。俗称的"简单工厂"(带 switch 的静态 create)是中间过渡态,switch 即待消除的类型知识
- **解决的问题**:`spawn(def_id) -> entity`——输入数据标识,输出装配完成的实例;调用方(spawner、任务效果、玩家指令)零类型知识
- **Godot 形态**:def 中声明创建配方——`scene_path`(用哪个原型,P2 卡片 5)、`components: Array[StringName]`(挂哪些部件,P2 卡片 2)、ctx 初始字段。工厂 = 读配方的通用解释器:载入场景 → 注入 def → 按声明装配组件 → 移交出生协议(卡片 2)。P3 卡片 5 预留的"实体工厂例外"在此正式立卡:**工厂是组合根之外唯一合法的批量装配点**
- **适用信号**:① 海量同构实体(def 体系成立即成立);② 创建请求来自数据侧(任务效果 `spawn_entity`、波次表)——数据只能引用 id,天然要求工厂

**收编小节:抽象工厂的三个去处**——它专治"产品族整体切换、保证不混搭",游戏中的三个对应场景各有更便宜的解:UI 主题 → 引擎 Theme 资源(🟢 引擎已吸收:换 Theme 整套控件切换);阵营配套 → **打包 def**(`FactionDef { unit_defs, building_defs, palette }`,一致性由"同取一包"天然保证——数据形态的抽象工厂);平台适配 → P3 卡片 2 按环境注册。**产品族需求一律走打包 def,不开工厂类层级。**

──────── 翻面 ────────

- **入口诱惑**:"这个类型特殊,工厂里 if 一下"——类型知识开始回渗
- **腐化信号**:① 工厂内长出 per-type 分支梯(`if def.id == &"boss_a": 特殊装配`)——特殊性该声明进 def(新组件、新配方字段),工厂保持解释器纯度(P2 卡片 3 翻面①同病);② 工厂之外出现第二创建路径(某系统自己 instantiate + 手工装配)——构造散落,§6 审计对象;③ 工厂绕过出生协议直接把实体塞进世界——快是快了,注册漏一半
- **不可逆点**:if 梯 + 旁路创建点并存后,"实体是怎么被造出来的"没有单一答案,任何装配改动都要找齐所有产地
- **解药**:特殊性进配方;创建点收归唯一;出厂必经协议
- **关联**:P2 卡片 3(def)、卡片 5(原型)——工厂是两者的运行时装配工

## 卡片 2:出生/死亡协议

- **粒度层 / 吸收状态**:系统 / 🟡(GoF 空白区,游戏开发的真实高地)
- **解决的问题**:实体的"出生"远不止 instantiate,"死亡"远不止 queue_free——两者各是一串**顺序敏感**的步骤,散写必漏
- **Godot 形态**:两个唯一入口函数,步骤顺序即协议:

**出生**:`instantiate → def 注入 → ctx 初始化 → 组件装配 → 入树 → 系统注册(空间分区 / tick 列表 / 触发索引 `[FSM:G篇 §4.2]`)→ 广播 spawned`
两条顺序铁律:① **def 注入先于入树**——否则 `_ready` 读 def 读到空(更稳妥的立场:`_ready` 只做表现初始化,规则初始化全部在协议函数内显式完成,不依赖引擎回调时序);② **spawned 事件永远最后**——事件的隐含契约是"听者此刻查询该实体,一切已就绪";注册完成前广播,就是把半成品公示于世

**死亡**:`标记逻辑死(状态转终态)→ 广播 despawned → 系统注销(与注册严格对称)→ 断开信号连接 → 归还池/令牌 `[P5卡片1]``[FSM:B篇 §4.5]` → 延迟物理释放`
关键区分:**逻辑死与物理死分离**——逻辑死是 tick 内的规则事实(即刻生效,`[FSM:C篇]` 终态),物理释放(free)可以晚几秒(尸体表现期、死亡动画);两者之间实体处于"规则上不存在、场景里还可见"的合法状态,B 篇 target_invalid 消费的正是逻辑死信号
- **适用信号**:无条件——有实体生灭的项目都需要,区别只是协议显式还是散落

──────── 翻面 ────────

- **入口诱惑**:"先 instantiate 加进去再说,注册的事回头补"——出生竞态的第一步;死亡侧则是"queue_free 完事"——注销的事永远没有回头
- **腐化信号**:① **幽灵实体**:死了还在被 tick/被索引查到(注销遗漏)——本册反面主角之一的实体形态;② 注册与注销不对称(N 处注册,N-1 处注销——泄漏的注册表版);③ spawned 监听者报空(事件顺序违律);④ `_ready` 里出现规则逻辑(对引擎回调时序的依赖,P4 §6 帧序耦合的创建版)
- **不可逆点**:创建/销毁步骤散落多处后,每加一个新系统注册需求,就要找齐全部产地和坟场各补一行——散弹式修改的定义级案例
- **解药**:两个协议函数为唯一出入口 + §6 对称审计
- **关联**:`[FSM:B篇 §4.4]`(协议的下游消费)、P5 卡片 1(池的借还即协议的轻量循环版)

## 卡片 3:生成器(Spawner)

- **粒度层 / 吸收状态**:系统 / 🟡
- **解决的问题**:在工厂之上加**调度层**——回答 when(什么时机)/ where(什么位置)/ what(生成什么),即游戏的节奏控制器:波次刷怪、资源再生、移民抵达、事件降临
- **Godot 形态**:三问全部数据化——when:波次表/冷却/触发条件(复用 `[FSM:G篇 §2.2]` 条件树);where:生成点集合/区域采样 + **有效性验证**(不卡墙、不贴脸、导航可达);what:权重表 + **生成预算**(每波总点数,单位各有造价——预算制让"波次难度"成为一个可调数字)。spawner 只做调度,**产出统一走工厂 + 出生协议**——它是卡片 1 的客户,不是同行
- **适用信号**:① 生成有节奏设计需求(不是一次性摆放);② 生成规则需要内容侧调整(波次表就是内容)

──────── 翻面 ────────

- **入口诱惑**:"刚生成的怪,顺手在这配点 AI/掉落吧"——spawner 是继中介者、Manager 之后的又一个逻辑引力阱
- **腐化信号**:① spawner 吸入生成物的行为逻辑(AI 参数、掉落表)——那些属于 def,spawner 只该知道 def_id 和数量;② 位置无验证直接落点(卡墙、叠人、刷玩家背后贴脸);③ 无上限/无预算生成(性能与玩法双崩——存活数上限是 spawner 的必备字段,`[P5卡片1]` 池上限的孪生);④ spawner 自己 instantiate(绕开工厂,构造散落)
- **不可逆点**:引力阱型 God Object 的标准路径,P2 §6 已建档
- **解药**:行为归 def、调度归 spawner、制造归工厂——三权分立
- **关联**:卡片 1(下游)、`[FSM:G篇]`(条件树复用)、P2 §6(引力阱)

## 卡片 4:建造者(Builder)

- **粒度层 / 吸收状态**:对象~系统 / 🟡(GoF 建造者的窄生存位)
- **解决的问题**:构造过程**本身**有阶段、有中间状态、需要校验回滚——分步施工,最后验收。GoF 原始动机(构造参数多而复杂)已被 def 取代(参数多?进 def 填表);存活地盘是**程序化生成**:地图生成、随机任务组装、程序化关卡
- **Godot 形态**:阶段流水线 + 中间表示。以地图生成为例:`布房间 → 连走廊 → 放出入口 → 撒装饰 → 校验(连通性/可玩性)→ 不过则回滚重跑`。两条纪律:① **产出蓝图(blueprint),不直接产出实体**——builder 输出纯数据的中间表示,落地交给工厂 + 出生协议批量执行;蓝图可序列化、可单测、可预览,施工与入世解耦;② **种子确定性**——同 seed 同蓝图(`[FSM:F篇 §5.4]` 确定性纪律的生成版),注入带种子 RNG,调试可复现
- **适用信号**:① 构造有阶段且阶段间有依赖;② 产物需要整体校验,可能推翻重来——填表(def)给不了"重来"这个动作,这就是 def 与 builder 的分界:**静态配方用 def,动态施工用 builder**

──────── 翻面 ────────

- **入口诱惑**:"这个构造函数参数有点多,上 builder 吧"——GoF 教材反射;参数多的正解是 def,builder 的仪式(分步、校验、回滚)对静态配置是纯开销
- **腐化信号**:① 阶段间隐式依赖(装饰阶段偷读房间阶段的临时变量而非蓝图字段)——蓝图是阶段间唯一合法通道,旁路即 P4 帧序耦合的生成版;② 无校验出厂(不连通的地图、无解的任务直接上线)——校验阶段是 builder 存在理由的一半,砍掉它 builder 就退化成一串顺序函数;③ builder 直接 instantiate 节点边生成边入世(蓝图纪律违反,生成中途失败留下半个世界)
- **不可逆点**:隐式阶段依赖织密后,调整阶段顺序/插入新阶段等于重写
- **解药**:蓝图作为阶段间契约;校验必设;落地走工厂
- **关联**:卡片 1(蓝图的落地执行者)、`[FSM:DEF篇 §5.1]`(阶段序列骨架通用)

## 卡片 5:资源加载策略

- **粒度层 / 吸收状态**:系统 / 🔴(`preload` 是默认写法、一键顺手,而它正是启动膨胀与依赖雪崩的入口)
- **解决的问题**:每次 instantiate 背后都站着一次加载;加载的**时机与批次**决定启动时长、gameplay 卡顿与内存曲线
- **Godot 形态**:三档机制 × 三类集合。机制:`preload`(编译期绑定,启动付费)/ `load`(运行时同步,调用处付费)/ `load_threaded_*`(异步,复杂度税)。集合策略:**常驻集**(全程要用:核心 UI、通用音效——preload 或启动期加载)/ **关卡集**(本局要用:按清单在 D 机 loading 状态预热 `[FSM:DEF篇 §3.3]`)/ **按需集**(低频偶发:异步加载 + 就位前的占位表现)
- **本卡片的核心陷阱——def 拉链效应**:def 数据库启动时全量加载(应该,它是轻数据),但如果 def 里用 `@export var scene: PackedScene` 直接引用场景,**加载 def 就递归加载了全部场景及其贴图、模型、音效**——整张资源图被一条链拉起,启动时长与内存和内容量同步膨胀。解药:**def 存 path/UID 字符串,场景由工厂按需 load**——def(轻)与资产(重)的加载解耦,这是 def 体系在资源侧的配套纪律,`[FSM:C篇 §4.1]` 三层切分的第四层
- **适用信号**:无条件——不做策略本身就是一种(最差的)策略

──────── 翻面 ────────

- **入口诱惑**:"preload 一下就能用了"——每个 preload 都是一票"启动时就要"的投票,没人统计票数
- **腐化信号**:① gameplay 中途同步 `load` 大资产(掉帧尖峰与加载时刻对齐——profiler 一眼可见,`[P5 §0]` 三问适用);② preload 链雪崩(A preload B,B preload C……脚本加载即资源图加载);③ 关卡集无清单(loading 状态干等第一个 load 请求,预热形同虚设);④ def 拉链(见上)
- **不可逆点**:preload 网织满后,"启动到底加载了什么"只能靠工具考古;def 拉链固化后,解链要改全部 def 字段类型
- **解药**:def 存 path;preload 仅限常驻集且集合有清单;关卡清单交 loading 状态;中途加载一律异步 + 占位
- **关联**:`[FSM:DEF篇 §3.3]`(loading 状态是本卡片的流程宿主)、卡片 1(工厂是按需 load 的执行点)

## 6. 反面专题:供应链完整性审计

本册三个反面主角(构造散落、出生竞态、加载卡顿)都是链条断裂,审计因此是**全链走查**,四项全部可脚本化(适合作为 CI 门禁交给代理维护):

1. **创建点唯一性**:grep `instantiate(` ——合法出现处仅:工厂、builder 落地段、E 层纯表现元素(飘字、特效);其余每处都是构造散落的现行犯
2. **注册/注销对称**:每类系统注册调用(分区、tick 列表、索引、信号连接)与对应注销调用配对计数;不对称即幽灵实体/泄漏候选
3. **事件顺序断言**:测试中给 spawned/despawned 挂监听,监听内立即查询该实体的全部注册面(在不在分区、在不在 tick 表)——半成品公示当场抓获
4. **加载台账**:启动期与 loading 期各打一份"实际加载资源清单",与常驻集/关卡集清单 diff——preload 雪崩和 def 拉链都会在 diff 里现形

## 8. AI 代理协作约定(供 CLAUDE.md 摘录)

1. 实体创建一律经工厂 + 出生协议;代码中出现工厂/builder/E 层表现之外的 `instantiate(` 即为实现错误。
2. 出生协议内 def 注入必须先于入树,spawned 广播必须位于全部注册之后;`_ready` 中禁止规则初始化。销毁一律走死亡协议,禁止裸 `queue_free`。
3. spawner 只声明 when/where/what 数据,不携带生成物行为参数,不自行 instantiate;必须声明存活上限与生成预算。
4. builder 阶段间只允许通过蓝图传递数据;必须包含校验阶段与种子注入;蓝图落地经工厂。
5. def 中引用场景/重资产一律使用 path/UID 字段;`@export` 直引 PackedScene 的 def 字段不予通过。
6. §6 四项审计脚本随本册功能一同交付,并挂入 CI。
